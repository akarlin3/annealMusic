# Implementation Plan — v1.7 AI-Assisted Patch Generation

Use an LLM to generate Anneal Ambiance patches from natural-language descriptions, perform mood transfers, generate evocative patch descriptions, and run semantic similarity searches using Postgres `pgvector` embeddings.

---

## 1. Provider & Model Selection

To balance cost, latency, and generation quality, we will employ a dual-model/dual-provider strategy accessed via abstract client interfaces.

### Model Matrix

| Task              | Primary Model                   | Alternative / Fallback     | Rationale & Token Cost                                                                                                                                                                                                                                                               |
| :---------------- | :------------------------------ | :------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Text-to-Patch** | `claude-3-haiku-20240307`       | `claude-3-5-sonnet-latest` | **Haiku is extremely cost-effective** (~$0.0005/input, ~$0.00125/output). A typical generation is ~500 input tokens and ~300 output tokens. Haiku cost: **~$0.0006 per call**. Sonnet has superior creative vibes but is 10x more expensive (~$0.006 per call). We default to Haiku. |
| **Mood Transfer** | `claude-3-haiku-20240307`       | `claude-3-5-sonnet-latest` | Same rationale as text-to-patch; high structured-output compliance at sub-millisecond speeds.                                                                                                                                                                                        |
| **Description**   | `claude-3-haiku-20240307`       | N/A                        | Extremely cheap and fast; handles poetic descriptions effortlessly.                                                                                                                                                                                                                  |
| **Embeddings**    | OpenAI `text-embedding-3-small` | Voyage AI `voyage-3`       | OpenAI's `text-embedding-3-small` is standard, robust, cheap ($0.02 / 1M tokens), and produces exactly **1536 dimensions**, perfectly matching the proposed pgvector column. Voyage is also supported as an abstraction.                                                             |

## 2. API & Data Model Additions

We will define SQLAlchemy models and Alembic migrations, supporting `pgvector` on Postgres and falling back gracefully to SQLite in test suites.

### Database Migration (`api/alembic/versions/0006_ai_patches.py`)

To ensure SQLite tests do not crash, the Alembic migration and SQLAlchemy model will dynamically detect the dialect:

```python
import alembic.op as op
import sqlalchemy as sa
from sqlalchemy.dialects import postgresql

def upgrade():
    bind = op.get_bind()
    is_postgres = bind.dialect.name == "postgresql"

    if is_postgres:
        op.execute("CREATE EXTENSION IF NOT EXISTS vector")

    # Add columns to patches
    op.add_column("patches", sa.Column("ai_description", sa.Text(), nullable=True))
    if is_postgres:
        from pgvector.sqlalchemy import Vector
        op.add_column("patches", sa.Column("ai_description_embedding", Vector(1536), nullable=True))
    else:
        op.add_column("patches", sa.Column("ai_description_embedding", sa.Text(), nullable=True)) # fallback
    op.add_column("patches", sa.Column("ai_description_source", sa.String(), nullable=True))

    if is_postgres:
        op.execute(
            "CREATE INDEX idx_patches_embedding ON patches USING ivfflat (ai_description_embedding vector_cosine_ops) WHERE visibility = 'public'"
        )

    # Create ai_generations table
    op.create_table(
        "ai_generations",
        sa.Column("id", sa.UUID(), primary_key=True),
        sa.Column("user_id", sa.UUID(), sa.ForeignKey("users.id", ondelete="CASCADE"), nullable=False),
        sa.Column("kind", sa.String(), nullable=False), # 'text-to-patch' | 'mood-transfer' | 'description'
        sa.Column("prompt", sa.Text(), nullable=False),
        sa.Column("input_patch_id", sa.UUID(), sa.ForeignKey("patches.id", ondelete="SET NULL"), nullable=True),
        sa.Column("output_state", sa.JSON(), nullable=True),
        sa.Column("model", sa.String(), nullable=False),
        sa.Column("prompt_tokens", sa.Integer(), nullable=True),
        sa.Column("output_tokens", sa.Integer(), nullable=True),
        sa.Column("cost_estimate_usd", sa.Numeric(8, 6), nullable=True),
        sa.Column("created_at", sa.DateTime(timezone=True), server_default=sa.func.now(), nullable=False),
        sa.Column("cached", sa.Boolean(), nullable=False, default=False),
    )
```

## 3. Schema-in-Prompt Strategy & System Prompt Design

Instead of hardcoding schema limits, the system prompt generator will load the source-of-truth `manifest.v7.json` at startup/request time and serialize it to an inline specification.

### Dynamic Schema Generator (`api/app/services/schema_prompt.py`)

A service function will read the manifest and export a compressed, developer-friendly instruction block containing:

- Shared Parameter Names, Min/Max limits, and Decimal places.
- Engine Namespaces (`sine`, `fm.*`, `gr.*`, `ph.*`).
- Loop slot codes (`LA.*`, `LB.*`, `LC.*`).
- Supported modes and arcs.

### System Prompt for Text-to-Patch

````
You are the Anneal Ambiance AI Patch Generator. Your role is to translate natural language mood descriptions into synthesiser patches conforming to the Anneal Ambiance engine architecture.

### Schema Manifest
[SCHEMA_JSON_INJECTED_HERE]

### Output Format
You MUST respond with a single, raw JSON object. Do not include markdown formatting, backticks, fences (e.g. no ```json), or preambles.
The JSON object must represent the synthesiser state as a flat dictionary of parameter keys and values, which our server will serialize into a URL query-like format.

## 4. Validation, Retry & Clamp Logic

The LLM is requested to produce flat JSON. The server acts as a robust validation and post-processing harness:

1. **Defensive Parsing**: Strip any surrounding markdown fences (` ```json ` or ` ``` `) and leading/trailing whitespace.
2. **Key Conversion**: Parse the flat JSON object and convert it to a URL query-like ampersand-separated string (e.g. `rootFreq=440&spread=1&e=sine`).
3. **Strict Validation**: Call `validate_payload(payload, 7)`.
4. **Retry Loop**:
   - If JSON is malformed, or if `validate_payload` returns errors, trigger a **Retry Call** to the LLM.
   - Provide the error list: `"Your previous response failed validation with these errors: [ERRORS]. Please correct and try again."`
   - Maximum 2 retries (3 calls total).
5. **Soft Clamping**:
   - For simple out-of-range values, clamp the value to the min/max bounds and warn, rather than failing or retrying.
   - For invalid/unknown engines, names, or structural keys, treat it as a hard failure and retry.

## 5. Rate Limiting, Quotas, and Caching

To prevent API abuse and cost spikes, we implement a tiered quota system and prompt-level caching.

### Rate Limits

- **Anonymous Users (per IP/Cookie)**:
  - 20 generations per hour.
  - 100 generations per day.
- **Authenticated Users (linked accounts)**:
  - 50 generations per hour.
  - 300 generations per day.
- Current usage will be tracked using the sliding-window in-memory `RateLimiter` or DB queries on `ai_generations`. Since standard rate limits in `RateLimiter` are hourly, we can implement day-limits via a DB query to `ai_generations` where `created_at` is within the last 24 hours.

### Caching

- An identical generation request (same normalized prompt + same system prompt version + same model) is cached in-memory with a TTL of 1 hour.
- Cache key: `sha256(model + system_prompt_version + normalized_prompt)`.
- Cache hits:
  - Return instantly.
  - Set `cached = True` in the API output (for diagnostics).
  - Do NOT decrement the user's quota or rate limit.

## 6. Mood Transfer & "What Changed" Diffs

### Mood Transfer Prompts

The mood transfer prompt includes the current patch state as flat JSON and asks the model to adjust it.

- **System Prompt Guidelines**: "Given this current patch JSON, modify it according to the user's direction (e.g., 'Make it darker'). Preserve the overall character; change only what the direction implies. Return the full modified JSON patch state."

### Structural Diff

The server computes a structural diff between the input and output parameters:

```python
def compute_structural_diff(before: dict, after: dict) -> list[dict]:
    changes = []
    # Identify keys changed, added, or deleted
    all_keys = set(before.keys()) | set(after.keys())
    for k in all_keys:
        if k in ("v", "payload"):
            continue
        val_before = before.get(k)
        val_after = after.get(k)
        if val_before != val_after:
            changes.append({
                "key": k,
                "oldValue": val_before,
                "newValue": val_after,
                "label": get_param_label(k), # rootFreq -> Root Frequency, ph.damping -> Damping
                "direction": "increased" if (isinstance(val_after, (int, float)) and isinstance(val_before, (int, float)) and val_after > val_before) else "decreased" if (isinstance(val_after, (int, float)) and isinstance(val_before, (int, float)) and val_after < val_before) else "changed"
            })
    return changes
```

The server returns both the new state and the array of `changes` so the UI can toast a friendly summary: "Made the sound darker (lowered filter brightness, increased reverb space)."

## 7. Similarity Search & Embedding Pipeline

Semantic similarity allows the gallery to surface "Similar Patches" row when viewing any patch.

### Pipeline

1. When a patch is made public:
   - Call the `describe-patch` LLM to generate a poetic 6-12 word description if the user didn't write one.
   - Request the embedding vector from the embedding client for that description.
   - Save the description and embedding vector synchronously or as an async task in PostgreSQL.
2. Nearest Neighbors:
   - Perform a Cosine-Similarity nearest-neighbor query using the `<=>` operator (which computes cosine distance).
   - Filter by `visibility = 'public'` and order by distance.
   - Apply a distance threshold filter (e.g., `< 0.4` distance) to prevent showing completely unrelated patches.

## 8. API Endpoints

We will mount five new endpoints under `/api/v1/ai/` and `/api/v1/patches/`:

```
POST   /api/v1/ai/generate-patch       -- Body: { prompt: str }; Returns: { state: str, generation_id: UUID }
