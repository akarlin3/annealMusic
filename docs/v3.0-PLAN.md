# AnnealMusic v3.0 · Pieces & Timeline Foundation Plan

## Goal

Introduce the **Piece** as a new first-class top-level artifact parallel to Patches. A Piece is a timeline-based sequence of **Segments** that governs the sound over time, rather than a single sculptable state.

---

## 1. Final Data Model

We will add two tables: `pieces` and `piece_segments`, matching the proposed requirements exactly.

```sql
-- Migration 0009_v3_0_pieces.py
CREATE TABLE pieces (
  id                 UUID PRIMARY KEY,
  user_id            UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  schema_ver         INTEGER NOT NULL,
  defaults_state     JSONB NOT NULL,         -- Piece-level default sculpt/engine state
  title              TEXT,
  description        TEXT,
  visibility         TEXT NOT NULL DEFAULT 'unlisted',
  ai_description     TEXT,
  total_duration_ms  INTEGER,                -- Sum of segment durations (null for open pieces)
  has_open_segment   BOOLEAN NOT NULL DEFAULT FALSE,
  created_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at         TIMESTAMPTZ NOT NULL DEFAULT now(),
  short_slug         TEXT UNIQUE
);

CREATE TABLE piece_segments (
  id              UUID PRIMARY KEY,
  piece_id        UUID NOT NULL REFERENCES pieces(id) ON DELETE CASCADE,
  position        INTEGER NOT NULL,          -- 0-indexed order
  type            TEXT NOT NULL,             -- 'fixed' | 'arc' | 'open' | 'transition'
  duration_ms     INTEGER,                   -- Nullable for 'open'
  config          JSONB NOT NULL,            -- Type-specific configurations
  UNIQUE(piece_id, position)
);
CREATE INDEX idx_piece_segments_piece ON piece_segments(piece_id, position);
```

### SQLite Triggers / Fallbacks

Since tests run on SQLite, we will write our migration dynamically checking if dialect is PostgreSQL or SQLite. We don't need additional complex triggers for `pieces` because likes, follows, and gallery items for pieces are deferred to v3.7.

## 2. Final API Surface

We will implement the following endpoints in `api/app/routers/pieces.py`:

- `POST   /api/v1/pieces` — Create a new piece + its segments.
- `GET    /api/v1/pieces/:id_or_slug` — Retrieve a single piece, including its ordered segments.
- `PATCH  /api/v1/pieces/:id` — Update pieces metadata, defaults_state, or segments list (re-creates/re-positions segments).
- `DELETE /api/v1/pieces/:id` — Delete a piece (cascades to segments).
- `GET    /api/v1/pieces/me` — List the current user's owned pieces (with cursor pagination).

## 3. URL Schema v8 Spec

### Discriminator at the Root

The schema version bumps to `8`.

- If the schema version is `< 8`, it is parsed as a standard Patch with `kind: 'patch'`.
- If the schema version is `8`, the payload starts with `kind=patch` or `kind=piece`.

### Patch Encoding in v8

`#s=8:kind=patch&e=sine&rootFreq=147&...` (Same as v7 but prepended with `kind=patch&`).

### Piece Encoding in v8

`#s=8:kind=piece&title=<title>&desc=<desc>&def.e=sine&def.rootFreq=147&seg0.type=fixed&seg0.dur=5000&seg0.rootFreq=150&seg1.type=transition&seg1.dur=3000&seg1.easing=linear&seg2.type=open`

1. **Metadata**: `title=<escaped_string>`, `desc=<escaped_string>` (optional).
2. **Piece-level Defaults**: Any parameters starting with `def.` (e.g. `def.rootFreq=147`).
3. **Segments**: Ordered parameters prefixed by `seg<index>.` (e.g. `seg0.type=fixed`, `seg0.dur=5000`, `seg0.rootFreq=160`).

## 5. Piece Player Design

The `PiecePlayer` will be implemented inside the orchestrator layer as a sibling/extension of the existing player state machine.

### State Machine Extension

The player's state machine will be extended to support pieces:
`SessionState = 'idle' | 'starting' | 'playing-patch' | 'playing-piece' | 'stopping'`

- **`playing-patch`**: Replaces the older `running-open` and `running-arc` conceptually. Fits the existing patch playback.
- **`playing-piece`**: Represents the new piece playback.

### Segment-Boundary Handling

We will use `setInterval` / `requestAnimationFrame` to drive the piece timeline.
On segment boundaries:

1. Advance `activeSegmentIndex`.
2. Retrieve the next segment.
3. Configure the engine for the next segment.
   - If the next segment is a **`transition`** segment, we will activate the transition interpolation pipeline.
   - If there is an engine change (e.g. physical -> granular), we will call the existing `crossfadeTo` method inside the orchestrator.
4. Update parameter targets dynamically.

### Open Segment "Next" Affordance UX

When an `open` segment is encountered, the player will sustain indefinitely. The playback progress holds at the segment end. The UI will show a **"Next Segment"** button, clicking which advances the player to the next segment immediately.

---

## 6. Transition Segment Interpolation

Transition segments interpolate from the parameters at the end of the previous segment to the parameters at the start of the next segment.

### Easing Functions

We support three easing functions:

- `linear`: `t`
- `easeInOut`: `t * t * (3 - 2 * t)`
- `exponential`: `t === 0 ? 0 : Math.pow(2, 10 * (t - 1))`

### Interpolation function

`interpolateParams(prevState, nextState, t, easing)` will merge all numeric values proportionally. Non-numeric values (like `engineId`) are switched at `t >= 0.5`.

## 7. Timeline Editor Minimal

The UI will be a horizontal timeline editor built inside `src/piece/TimelineEditor.tsx`.

- **Layout**: Warm dark theme. Pixels-per-second is fixed (e.g. 3px/s) so a 10-minute piece is ~1800px. Standard scrollbar.
- **Color by Type**:
  - `fixed`: Sleek Teal
  - `arc`: Indigo/Violet
  - `open`: Crimson
  - `transition`: Golden/Orange
- **Operations**:
  - Add segment at end.
  - Delete segment.
  - Reorder via drag & drop.
  - Resize segment by dragging the right edge (duration increments/decrements; disabled for `open`).
- **Properties Panel**: Appears below the timeline when a segment is selected, displaying customizable parameters (e.g. arc selection for `arc` segment, easing for `transition`, override sliders for `fixed`).

---

## 8. Defaults + Segment Merging

When playing a Piece, the baseline is `defaults_state` (fully specified patch state).

- Every segment has a custom config representing **overrides**.
- When entering a segment, we merge the overrides on top of the `defaults_state`.
- For `transition` segments, the start state is the merged state of segment `N-1`, and the target state is the merged state of segment `N+1`.

## 9. Recording Integration

v1.5 stem export will become piece-aware:

- For Pieces, the offline renderer walks the segment timeline sequentially, applying parameter updates and compiling the master mix-down.
- Mix-down only in v3.0 (no multi-stem zip for pieces; full stems are deferred to v3.7).

## 12. Risks & Tradeoffs

- **Offline Rendering Performance**: Running multiple sequential segments in the offline renderer must be click-free and memory-efficient. We will pre-allocate resources and run dry parameter sweeps.
- **Engine Swaps**: Swapping engines at segment boundaries might cause small CPU spikes; using the orchestrator's existing crossfade engine mitigates this.
