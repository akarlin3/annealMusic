## 0. Current-state diagnosis (what v0.6 gives us, what the backend must respect)

| Primitive (file)                                                                                                                                                                                   | What it is today                                                                   | v0.7 consequence                                                                                                                                        |
| -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- |
| URL schema, **v4** (`src/share/schema.ts`, `src/share/encode.ts`)                                                                                                                                  | A flat, human-readable `key=value&…` payload behind `#s=4:`. **Not JSON.**         | The patch `state` we persist **is this string**. The server must validate it; the client already encodes/decodes it.                                    |
| Schema is **derived**, not declared (`CONTROL_DEFS` in `src/state/params.ts`, engine defs in `src/audio/engines/*`, `PRESET_ARCS` in `src/session/arcs.ts`, `GRAIN_BOUNDS` in `src/loop/types.ts`) | Bounds/keys/decimals live in exactly one place per concern, all in **TypeScript**. | The server (Python) cannot import these. We need a **cross-language bridge** for the single-source-of-truth rule (§6).                                  |
| `decodeState()` is **lenient** — clamps out-of-range, drops unknowns, never throws, returns `warnings[]`.                                                                                          | Tolerant by design (old share links must keep loading).                            | Server-side save validation must be **strict** (reject, not clamp) — saving is a write, not a tolerant read. Two different postures over one schema.    |
| Loop buffers are **runtime-only**; the URL carries flags + grain params but never audio (`encodeLoops`).                                                                                           | A frozen slot's _config_ survives a link; its _audio_ does not.                    | Captures (§ object storage) are the missing audio. A patch references captures by id; on load the client re-hydrates buffers into slots.                |
| `MAX_CAPTURE_SEC = 60`, stereo @ 48 kHz (`src/loop/types.ts`)                                                                                                                                      | ~11 MB as WAV, ~0.7 MB as Opus 96 k per 60 s.                                      | Drives capture size caps and the encode choice (§10).                                                                                                   |
| Web app deployed to **Firebase Hosting** (`firebase.json`, `.github/workflows/`), SPA with `**→/index.html` rewrite. CI = lint/typecheck/vitest/build.                                             | Static SPA on a CDN. No server today.                                              | **The spec's "Railway" applies to the API only.** Web stays on Firebase (§1, §15). Path routing for `/p/<slug>` already works via the existing rewrite. |
| State store is `zustand` (`useParamStore`); URL sync via `subscribeStoreToHash` (debounced `replaceState`).                                                                                        | Client owns all state; URL is the only persistence.                                | The API client is a **sibling** of the URL layer, not a replacement. Inline URLs keep working with the backend down (§17).                              |

**The one real obstacle: the schema is TypeScript, the validator must also be Python.**
The heuristic-drift rule is explicit — _"URL schema lives once and is the single
source of truth for both client and server validation."_ The schema is currently
~5 TS modules, intertwined with UI and audio-engine code, and the spec forbids
touching the engine code. So we cannot lift it into a shared package. The
resolution (§6) is a **generated manifest**: one small script reads the existing
TS definitions in place and emits a machine-readable JSON descriptor that the
Python server validates against, with a CI contract test that fails on drift.

## 1. Stack confirmation

**Confirmed:** FastAPI + SQLAlchemy 2.x (async) + Pydantic v2 + Alembic, on
**Railway-managed PostgreSQL**, with **Cloudflare R2** for object storage.

**Deviation flagged (PrismTask parity):** PrismTask co-locates everything on
Railway. AnnealMusic's **web stays on Firebase Hosting** (already wired, CDN-backed,
zero migration cost) — only the **API** goes to Railway. Migrating the SPA to
Railway would churn `firebase.json`, the deploy workflow, and DNS for no v0.7
benefit. Web↔API is a plain CORS boundary (§12).

**Object storage — Decision: Cloudflare R2, not a Railway volume.**

|               | Railway volume                   | **Cloudflare R2 (chosen)**                                     |
| ------------- | -------------------------------- | -------------------------------------------------------------- |
| API           | POSIX filesystem                 | S3-compatible (boto3)                                          |
| Egress        | counts against Railway bandwidth | **$0 egress**                                                  |
| Durability    | tied to one container/region     | multi-region object durability                                 |
| Serving audio | must proxy bytes through the API | **presigned URLs**, bytes never touch the API                  |
| Cost          | bundled, opaque                  | storage ~**$0.015/GB-mo**, **no egress fees**, free tier 10 GB |

Audio is egress-heavy (captures ~5 MB, recordings ~50 MB+); R2's zero-egress
pricing and S3 API (so `GET /captures/:id` → 302 to a presigned URL) are decisive.
A volume couples blob durability to the app container and forces every byte
through FastAPI. **Cost flag:** R2 is near-free at v0.7 scale (well under the 10 GB
free tier); the real cost lever is the 1 GB/anonId quota (§8), not the bucket.

## 2. Repo structure — Decision: monorepo, _minimal_ restructure

Add a Python service and a generated-schema directory **without relocating the
existing web app**:

```
/                      # web app stays at repo root (unchanged paths, vite, firebase, husky)
  src/ ...             # untouched except new src/api/ client module (§16)
  api/                 # NEW — FastAPI service
    app/               # FastAPI app, routers, models, storage adapter
    alembic/           # migrations (§11)
    tests/
    pyproject.toml
    Dockerfile
    docker-compose.yml # postgres + minio + api + web (§14)
  schema/              # NEW — generated cross-language schema bridge (§6)
    manifest.v4.json   # generated artifact (committed)
    gen-manifest.ts    # generator: reads src/ TS defs, emits the JSON
  docs/
```

**Rejected: the spec's `apps/web` + `apps/api` split.** It's the textbook
monorepo shape, but moving the SPA into `apps/web` rewrites every `@/` tsconfig
path, the vite root, `firebase.json`'s `public` dir, husky/lint-staged globs, and
the deploy workflow — a large, purely-mechanical diff with real regression risk,
for zero functional gain in v0.7. The spec's stated reason for monorepo ("schema
is shared between client encode/decode and server validation") is satisfied by the
**manifest** (§6), which lives at `schema/` and is consumed by both sides — _not_
by co-locating the apps. If a later slice wants the symmetric layout, that's a
clean, isolated move then. **Override available at "go" if you want the full split
now.**

## 3. Final data model (refined)

Changes from the proposal are called out; everything else is as proposed.

```sql
CREATE TABLE users (
  id            UUID PRIMARY KEY,                    -- the anonId (client-minted UUIDv4)
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_seen_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  bytes_used    BIGINT NOT NULL DEFAULT 0,           -- maintained on capture/recording write+delete
  patch_count   INTEGER NOT NULL DEFAULT 0,          -- ADDED: denormalized for cheap quota checks
  capture_count INTEGER NOT NULL DEFAULT 0,          -- ADDED
  recording_count INTEGER NOT NULL DEFAULT 0         -- ADDED
);

CREATE TABLE patches (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  schema_ver    INTEGER NOT NULL,                    -- the URL schema version of `state`
  state         JSONB NOT NULL,                      -- { "v": <ver>, "payload": "<encoded string>" } (§6)
  title         TEXT,
  description   TEXT,
  visibility    TEXT NOT NULL DEFAULT 'unlisted'
                  CHECK (visibility IN ('unlisted','public')),  -- ADDED: DB-level enum guard
  capture_refs  UUID[] NOT NULL DEFAULT '{}',
  short_slug    TEXT UNIQUE NOT NULL,                -- CHANGED: minted at creation, never null (§4 routing)
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_patches_user    ON patches(user_id, created_at DESC);
CREATE INDEX idx_patches_public  ON patches(visibility, created_at DESC) WHERE visibility = 'public';
CREATE UNIQUE INDEX idx_patches_slug ON patches(short_slug);

CREATE TABLE captures (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  storage_key   TEXT NOT NULL,
  duration_ms   INTEGER NOT NULL CHECK (duration_ms > 0 AND duration_ms <= 60000),
  sample_rate   INTEGER NOT NULL,
  channels      INTEGER NOT NULL,
  bytes         INTEGER NOT NULL,
  format        TEXT NOT NULL CHECK (format IN ('opus','wav')),
  ref_count     INTEGER NOT NULL DEFAULT 0,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_captures_user  ON captures(user_id);
CREATE INDEX idx_captures_orphan ON captures(ref_count, created_at) WHERE ref_count = 0;  -- ADDED: sweep (§ CP3)

CREATE TABLE recordings (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  storage_key   TEXT NOT NULL,
  duration_ms   INTEGER NOT NULL,
  bytes         INTEGER NOT NULL,
  format        TEXT NOT NULL CHECK (format IN ('opus','wav')),
  patch_id      UUID REFERENCES patches(id) ON DELETE SET NULL,
  title         TEXT,
  visibility    TEXT NOT NULL DEFAULT 'unlisted' CHECK (visibility IN ('unlisted','public')),
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_recordings_user ON recordings(user_id);
```

Refinements: **denormalized count columns** on `users` (quota checks without
`COUNT(*)`), **`short_slug` NOT NULL** minted at creation (every patch is
`/p`-addressable immediately — see §4), **CHECK constraints** on enums/duration
(defense in depth behind app validation), and an **orphan partial index** to make
the sweep cheap. `state` is **JSONB `{v, payload}`** not a bare string — keeps the
authoritative encoded payload intact while leaving room for v0.8 to add a
generated column for gallery filtering without a data migration.

## 4. Final API surface (refined)

```
GET    /healthz                              -- liveness (no DB)
GET    /readyz                               -- readiness: DB + storage reachable, migration head applied

POST   /api/v1/users                         -- idempotent upsert on x-anon-id; returns user + quota
GET    /api/v1/users/me                      -- quota, counts, bytes_used

POST   /api/v1/patches                       -- { state, title?, description?, visibility?, capture_refs? }
                                             --   validates state (§6), mints short_slug, returns full patch
GET    /api/v1/patches/:idOrSlug             -- MERGED: resolves UUID or slug; visibility-gated
GET    /api/v1/patches/me?cursor=&limit=     -- cursor-paginated list for current anon
PATCH  /api/v1/patches/:id                   -- title/description/visibility only; state immutable
DELETE /api/v1/patches/:id                   -- hard delete; decrements capture ref_counts

POST   /api/v1/captures                      -- multipart WAV upload; validates; transcodes; returns id + meta
GET    /api/v1/captures/:id                  -- 302 to presigned R2 URL
DELETE /api/v1/captures/:id

POST   /api/v1/recordings                    -- schema only in v0.7 (CP4); export pipeline is v1.0
GET    /api/v1/recordings/:id
GET    /api/v1/recordings/me
DELETE /api/v1/recordings/:id
```

**Refinements vs proposal:**

- **`POST /shortlinks` removed (merged).** Every patch gets a `short_slug` at
  creation, so a separate mint step is dead weight. `GET /patches/:idOrSlug`
  resolves either form, so the client never needs a slug→id round trip.
- **`GET /patches/:id` → `:idOrSlug`** for the same reason.
- **Pagination is cursor-based** (`created_at, id` keyset) not offset — stable
  under inserts, which matters once the gallery (v0.8) lists public patches.

## 5. Anon-ID lifecycle

- **First load:** no network call. The client mints nothing until the first save.
- **First save:** if `localStorage['am_anon_id']` is absent, the client generates a
  UUIDv4, stores it, and sends it as the `x-anon-id` header. The server upserts the
  `users` row (idempotent — `POST /users` is safe to call repeatedly).
- **Persistence:** `localStorage['am_anon_id']`. Read once into the API client at
  startup; written on first mint.
- **Recovery when storage cleared:** **lost, accepted** (per spec). A future auth
  slice lets a user claim anon content by email. As a _best-effort soft recovery_,
  the server also mirrors the id into a non-`httpOnly`, `SameSite=Lax` cookie
  (`am_anon`); if the user clears `localStorage` but not cookies, the client
  rehydrates from the cookie. This is convenience, not a guarantee.
- **Header vs cookie — Decision: header-primary.** `x-anon-id` is the source of
  truth for every authenticated request. It's explicit, not ambient, so it carries
  **no CSRF surface** (§12). The cookie is a recovery mirror only and is **never
  sufficient on its own for a write** — a request with a cookie but no header is
  treated as anonymous. Missing header on a write → server mints a fresh id,
  returns it in the body + `Set-Cookie`, and the client adopts it.

## 6. State validation — single source of truth (the crux)

**Decision: a generated manifest, with a CI contract test.**

1. `schema/gen-manifest.ts` imports the **existing** TS definitions in place
   (`CONTROL_DEFS`, `KEY_BOUNDS`, `SHARED_KEYS`, `ENGINE_ORDER` + `engineParamDefs`,
   `PRESET_ARCS` ids, `ARC_DURATION`, `GRAIN_BOUNDS`, `SLOT_IDS`, `SCHEMA_VERSION`)
   and emits `schema/manifest.v4.json`: a flat descriptor of every legal key, its
   bounds and decimals, the engine→param map, valid arc ids, grain field codes, and
   slot ids. **No engine code changes** — the generator only reads.
2. The TypeScript side keeps validating exactly as it does today (its validators
   _are_ the source the manifest is generated from — zero drift by construction).
3. The Python server loads `manifest.v4.json` at startup and validates an incoming
   patch `payload` **strictly**: every key known, every value numeric and **in
   range (reject, don't clamp)**, engine/arc/slot ids valid, no duplicate keys,
   payload under a length cap. This is the deliberate posture split — the client's
   `decodeState` is lenient for old share links; the server's save path is strict.
4. **CI contract test:** regenerate the manifest and `git diff --exit-code` it; a
   stale committed manifest fails the build. A second test asserts the Python
   validator accepts a corpus of client-encoded payloads and rejects mutated ones.

**Source of truth = the TS definitions.** Manifest = a compiled view of them.
Server = a consumer of the manifest. Nothing is hand-ported, so nothing drifts.

> Rejected alternative: hand-write Pydantic models mirroring the bounds. That
> re-declares the schema in a second language — exactly the drift the rule forbids.

## 7. Rate limits — confirmed, with per-IP fallback

Per **anonId** (sliding window):

| Action             | Limit      |
| ------------------ | ---------- |
| `POST /patches`    | 60 / hour  |
| `POST /captures`   | 20 / hour  |
| `POST /recordings` | 5 / hour   |
| any `GET`          | 600 / hour |

**Missing/forgeable anonId → fall back to per-IP**, stricter: 10 writes/hour,
200 GET/hour per IP (anonIds are trivially forgeable, so the IP ceiling is the real
floor). Over limit → **HTTP 429** with a `Retry-After` header and `{ "error":
"rate_limited" }`.

**Implementation — Decision: `slowapi`, in-memory, single instance for v0.7.**
Railway runs one API container; in-memory windows are correct there and add no
Redis cost. **Flag:** horizontal scaling (v0.8+) needs a shared store — leave a
`RATE_LIMIT_BACKEND` seam so swapping to Redis is config, not a rewrite.

## 8. Quotas — confirmed, with explicit status codes

Per anonId: **100 patches, 50 captures, 10 recordings, 1 GB total bytes.**

| Condition                                                        | Status                    | Body                                                                 |
| ---------------------------------------------------------------- | ------------------------- | -------------------------------------------------------------------- |
| count quota reached (e.g. 101st patch)                           | **409 Conflict**          | `{ "error": "quota_exceeded", "resource": "patches", "limit": 100 }` |
| single upload exceeds per-file cap (capture > 60 s / > size cap) | **413 Payload Too Large** | `{ "error": "file_too_large" }`                                      |
| upload would exceed 1 GB total bytes                             | **409 Conflict**          | `{ "error": "quota_exceeded", "resource": "bytes" }`                 |
| rate limit (§7)                                                  | **429**                   | `{ "error": "rate_limited" }`                                        |

Rationale: **429 is rate (time-based), 409 is quota (state-based), 413 is a single
oversized request.** Conflating them (the spec's "413? 429?") would make the client
unable to tell "slow down" from "you're full." Counts come from the denormalized
columns (§3); enforced inside the same transaction as the insert to avoid races.

## 9. Object storage layout & signed URLs

- **Bucket layout:** `captures/<userId>/<captureId>.opus`,
  `recordings/<userId>/<recordingId>.<opus|wav>`. User-prefixed so a `DELETE`
  cascade / sweep can list-by-prefix.
- **Encoding:** captures → **Opus 96 kbps** (default; tiny, transparent for ambient
  texture). Recordings → configurable (`opus` default, `wav` lossless opt-in for
  v1.0 export).
- **Read path:** `GET /captures/:id` → **302 to a presigned R2 GET URL** (TTL ~1 h).
  Audio bytes never transit FastAPI. The API only checks existence + ownership/
  visibility before redirecting.
- **Write path — Decision: server-mediated upload (not presigned PUT) for v0.7.**
  The client POSTs the WAV to the API; the API validates format/duration/bytes,
  transcodes (§10), and puts the Opus to R2. Captures are ~11 MB WAV — acceptable
  through the API. Presigned PUT would skip validation (client could upload
  anything). **Flag:** if upload latency bites, move to presigned PUT + a
  post-upload validation/transcode worker — isolated change.
- **`StorageClient` interface** abstracts this: an S3/R2 impl and an in-memory impl
  for tests (CP1).

## 10. Audio encoding pipeline — Decision: client uploads WAV, server transcodes to Opus

The captured loop is an **in-memory `AudioBuffer`** (processed, already in the loop
slot), not a live `MediaStream`. So:

- **Client:** serialize the `AudioBuffer` → **WAV (PCM)** deterministically (a tiny
  encoder; no codec dependency, works identically on every browser). Upload WAV.
- **Server:** validate, then **transcode to Opus 96 k via `ffmpeg`** (bundled in the
  Docker image). Store Opus.

**Why not client-side Opus via `MediaRecorder`?** (a) `MediaRecorder` records a
_stream_, not a buffer — we'd have to play the buffer through a
`MediaStreamDestination` in real time just to capture it, which is slow and lossy.
(b) **Safari's `MediaRecorder` Opus support is unreliable/absent** (logged in
`docs/COMPAT.md` territory) — WAV-from-buffer sidesteps the whole compatibility
matrix. (c) Server transcode gives one consistent codec/bitrate regardless of
browser. **Tradeoff:** larger upload (~11 MB vs ~0.7 MB) and server CPU per
transcode; both are fine at v0.7 scale and capped by §7/§8. Documented in
`docs/COMPAT.md`.

## 11. Migration strategy — Alembic from day one

- Lives at `api/alembic/`, versions in `api/alembic/versions/`.
- Naming: `alembic revision --autogenerate -m "<slug>"` → `<rev>_<slug>.py`.
  CP1 lands the initial full-schema migration; **no schema change ever lands
  without a migration** (enforced by review + a CI check that models match head).
- **Run on deploy:** Railway **release command** `alembic upgrade head` runs in a
  release phase _before_ the new container serves traffic. If the migration fails,
  the deploy fails and the old container keeps serving.
- **CI:** integration tests run `alembic upgrade head` against an ephemeral Postgres
  (service container) so migrations are exercised every build.

## 12. CORS, CSRF, security headers

- **CORS:** allow-list = production web origin (the Firebase domain / annealmusic
  root) + `http://localhost:5173` (vite). Methods `GET, POST, PATCH, DELETE`;
  allowed headers `x-anon-id, content-type, x-request-id`; `allow_credentials` only
  if the cookie mirror is enabled, and then origins must be explicit (no `*`).
  Origins come from a `CORS_ORIGINS` env list.
- **CSRF:** writes require the `x-anon-id` **header**, which a cross-site form
  cannot set and a browser won't attach automatically → **no CSRF vector**. The
  `am_anon` cookie is `SameSite=Lax` and **never accepted alone for a write**.
- **Security headers (middleware):** `Strict-Transport-Security`,
  `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin`,
  `X-Frame-Options: DENY` on API responses, and `Cache-Control: no-store` on
  authenticated JSON. CSP is primarily the web tier's job (Firebase) and is out of
  API scope.

## 13. Logging / observability

- **Structured JSON logs** (stdlib logging + `python-json-logger`): one line per
  request with method, path, status, latency-ms, and a **hashed** anonId (never log
  the raw id).
- **Request IDs:** middleware accepts an inbound `x-request-id` or generates a UUID,
  binds it to the log context, and echoes it on the response.
- **Sentry — stub only for v0.7.** Init guarded by `SENTRY_DSN`; **no-op when
  unset** (per spec), so v0.8 wires it by setting one env var. No PII, no payloads
  in breadcrumbs.

## 16. Client integration

- **`src/api/` (new):** `types.ts` (request/response types mirroring §4),
  `client.ts` (typed `fetch` wrapper: base URL from `VITE_API_BASE`, injects
  `x-anon-id`, timeouts, typed errors for 409/413/429), `anon.ts` (localStorage
  anonId get/mint, cookie soft-recovery).
- **Save Patch button** beside the existing `CopyLinkButton`. **Save dialog:**
  title, description, visibility (`unlisted` default / `public`), and an
  **"include captures"** toggle (**off by default** — params-only, per the
  cost/privacy-aware spec).
- **My Patches drawer:** lists the anon user's patches (title, date, capture-count
  badge), quick-load on click, delete with confirm.
- **Inline vs short-link — Decision (matches proposal):** **Copy Link → inline
  `#s=4:` URL** (self-contained, works offline forever). **Save → server short
  link `/p/<slug>`** (requires backend, supports captures + My Patches).
- **`/p/<slug>` route:** the SPA reads `location.pathname`, and on a `/p/` prefix
  calls `GET /patches/:idOrSlug`, loads `state` into the store (reusing the existing
  decode path), then fetches any `capture_refs` and re-hydrates loop slots —
  including restoring **frozen** slots with their remembered grain params (v0.6
  behavior). Firebase's existing `**→/index.html` rewrite already serves the route.

## 17. Backend-down behavior (tested, not assumed)

- All v0.1–v0.6 features are **pure client** and keep working with the API offline.
- The API client has a short timeout and typed failure. **Save** failure → toast
  ("Couldn't reach the server — your link still works"), no state corruption.
  **My Patches** while offline → empty/offline state, not a crash.
- **Inline `#s=` URLs never call the backend** — sharing and loading by inline URL
  is fully offline.
- A `/p/<slug>` load with the backend down shows a graceful error and offers the
  user their current inline link.
- Tested via mocked `fetch` rejection in the client suite (CP2/CP3).

## 18. Risks & tradeoffs

| Risk / tradeoff                                    | Severity                         | Mitigation                                                                                             |
| -------------------------------------------------- | -------------------------------- | ------------------------------------------------------------------------------------------------------ |
| **Manifest drift** (TS schema vs Python validator) | High if unmanaged                | Generated, not hand-ported; CI contract test diffs the manifest and round-trips a payload corpus (§6). |
| **Monorepo-lite vs full `apps/*` split**           | Reversible, later cost           | Documented (§2); the symmetric move is clean and isolated if a future slice wants it.                  |
| **Rate limiter is in-memory** (single instance)    | Low now, blocks horizontal scale | `RATE_LIMIT_BACKEND` seam → Redis when scaling (§7).                                                   |
| **Server transcode CPU** (ffmpeg per upload)       | Low at v0.7 scale                | Capped by §7/§8; move to presigned-PUT + worker if it bites (§9).                                      |
| **R2 vendor coupling**                             | Low                              | Pure S3 API via `StorageClient`; portable to any S3 provider.                                          |
| **WAV upload size** (~11 MB/60 s)                  | Low                              | Within caps; Opus stored, not WAV; flagged in COMPAT.                                                  |
| **Anon-id forgeable**                              | Accepted                         | Per-IP rate-limit floor (§7); quotas are best-effort abuse control, not security.                      |
| **Cost**                                           | Low                              | R2 under free tier at this scale; the lever is the 1 GB/anonId quota, not the bucket (§1, §8).         |
| **`state` immutability** (edit = new patch)        | Design choice                    | Matches "new state = new patch" in the spec; PATCH only touches metadata (§4).                         |
