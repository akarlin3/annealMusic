**The one real obstacle:** the engine is real-time + timer-driven, and the spec
forbids both (a) diverging the renderer from the client engine and (b) touching
the engine code. Faithful previews therefore require running the _actual_ engine
in a runtime that (1) drives the same wall-clock timers and (2) reproduces
Chrome's DSP bit-for-near-bit. That runtime is **headless Chromium** (§3).

## 1. Final data model (refined)

ALTERs to `patches` and one new `reports` table. Changes from the proposal are
called out.

```sql
ALTER TABLE patches
  ADD COLUMN preview_storage_key  TEXT,
  ADD COLUMN preview_duration_ms  INTEGER,
  ADD COLUMN preview_status       TEXT NOT NULL DEFAULT 'none'   -- ADDED
        CHECK (preview_status IN ('none','rendering','ready','failed')),
  ADD COLUMN load_count           INTEGER NOT NULL DEFAULT 0,
  ADD COLUMN published_at         TIMESTAMPTZ;

-- visibility gains 'flagged'. The v0.7 CHECK must be replaced, not extended.
ALTER TABLE patches DROP CONSTRAINT ck_patches_visibility;
ALTER TABLE patches ADD  CONSTRAINT ck_patches_visibility
      CHECK (visibility IN ('unlisted','public','flagged'));

CREATE INDEX idx_patches_gallery ON patches(published_at DESC, id DESC)
      WHERE visibility = 'public';
CREATE INDEX idx_patches_loads   ON patches(load_count DESC, published_at DESC, id DESC)
      WHERE visibility = 'public';
CREATE INDEX idx_patches_search  ON patches
      USING GIN (to_tsvector('english',
                 coalesce(title,'') || ' ' || coalesce(description,'')))
      WHERE visibility = 'public';                                -- Postgres-only

CREATE TABLE reports (
  id           UUID PRIMARY KEY,
  patch_id     UUID NOT NULL REFERENCES patches(id) ON DELETE CASCADE,
  reporter_id  UUID,                                  -- anon id; null for fully-anon
  reason       TEXT NOT NULL CHECK (reason IN ('spam','inappropriate','other')),
  detail       TEXT,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  status       TEXT NOT NULL DEFAULT 'open'
        CHECK (status IN ('open','dismissed','upheld'))
);
CREATE INDEX idx_reports_open ON reports(status, created_at DESC) WHERE status = 'open';
```

**Refinements vs proposal:**

- **`preview_status` added** (the proposal implied but didn't declare it). CP1/CP2
  both need the four-state machine `none → rendering → ready | failed`. Without it
  the `GET /preview` endpoint can't choose between 202/302/503.
- **`published_at` is set once** — on the _first_ transition to `public`, never
  overwritten (a public→unlisted→public round-trip keeps the original date).
  Matches "most recent" sort intent.
- **Composite index tiebreak `id`** on both gallery indexes so the keyset cursor
  (§10) has a total order even when `published_at`/`load_count` tie.
- **`reason`/`status` CHECKs** mirror the v0.7 defense-in-depth posture
  (app-level validation + a DB guard).
- **GIN search index is Postgres-only**, created in the migration with no
  `metadata` counterpart (SQLite tests never see it; see §0, §9).

SQLAlchemy model mirrors: new columns on `Patch`, relaxed `CheckConstraint`, new
`Report` model. The GIN index is **not** declared on the model (migration-only) so
`create_all` on SQLite doesn't choke.

## 2. Final API surface (refined)

```
GET    /api/v1/gallery                      -- public, no auth. ?sort=&filter=&q=&cursor=&limit=
GET    /api/v1/patches/:id/preview          -- 302 ready | 202 rendering | 404 not-public | 503 failed
POST   /api/v1/patches/:id/load             -- increments load_count (rate-limited per IP+patch); 200

POST   /api/v1/reports                       -- { patch_id, reason, detail? } ; anon-optional
GET    /api/v1/admin/reports?status=open     -- auth: x-admin-key
PATCH  /api/v1/admin/reports/:id             -- { status: 'dismissed' | 'upheld' }
PATCH  /api/v1/admin/patches/:id/visibility  -- { visibility: 'public' | 'flagged' | 'unlisted' }

# Existing, with new behavior:
POST   /api/v1/patches                       -- if visibility=public: screen + set published_at + enqueue render
PATCH  /api/v1/patches/:id                   -- visibility→public: same screen + publish + enqueue
GET    /api/v1/patches/:idOrSlug             -- 'flagged' now returns 403 {error:'under_review'} (was readable)
```

**Refinements vs proposal:**

- **`GET /preview` uses status codes as the protocol**, not a body flag: `302`→
  presigned R2 URL (mirrors `GET /captures/:id`), `202`→ still rendering, `404`→
  not public (or no preview), `503`→ render failed. Cacheable only on `302` (§11).
- **`POST /load` always returns 200** and increments _only_ if under the per-IP+
  patch limit; over-limit is a **silent no-op** (UX is never blocked — the user
  still navigates to the patch). Prevents the rate limit from breaking "Load."
- **Publish screening lives on both `POST` and `PATCH`** (a patch can be born
  public or flipped public). Single screening function (§7).
- **`GET /patches/:idOrSlug` now special-cases `flagged`** → `403
{error:'under_review'}` so the SPA short-link page shows "under review" rather
  than a generic 404 (the creator's own link included — v0.8 spec §4).
- **Admin endpoints** sit under `/api/v1/admin/*` behind a single `require_admin`
  dependency (constant-time compare of `x-admin-key` to `ADMIN_KEY`). Missing/
  wrong key → `401 {error:'unauthorized'}`.

## 3. Server-side rendering decision — **Option B (headless Chromium), real-time capture**

This is the crux. The default recommendation in the spec is Option B; the
diagnosis in §0 makes that not just convenient but **correct**, for a reason the
spec only gestures at:

> The engine is real-time and wall-clock-timer-driven, and its sound depends on
> Chrome's exact DSP (`ConvolverNode` impulse-response reverb, `BiquadFilter`,
> oscillator detune). "Same engine logic" can only mean "same runtime."

### Why not Option A (node-web-audio-api / pure-JS)

1. **Parity risk = the forbidden divergence.** node-web-audio-api is a _different_
   DSP implementation (Rust). Its convolver and biquad will not match Chrome's
   sample-for-sample. The reverb tail (`makeIR`, 4 s / 2.4 decay) is exactly where
   that shows. The spec: _"if the renderer diverges from the client, that's a bug."_
   Option A _builds in_ the divergence.
2. **Shimming cost.** The orchestrator reads global `window`/`navigator` and the
   `AudioContext` constructor. Option A needs a globals shim just to import it.
3. **Still real-time anyway.** The timer problem (§0) is runtime-agnostic —
   `OfflineAudioContext` rendering would skip the `setInterval` drift in _both_ A
   and B. So A doesn't even buy us offline speed without re-architecting the
   engine (which the spec forbids). Its only real win is a lighter image.

### Option B mechanics

- **Renderer host:** Playwright + Chromium on the Railway API host.
- **Render harness = a dedicated Vite build target** (`src/render/headless.ts` +
  `render.html`) that imports the **real `Orchestrator`** and exposes
  `window.__annealRender(payload, opts) → Promise<{ b64, mime }>`. It:
  1. decodes `payload` with the lenient client decoder (same as `/p/<slug>` load);
  2. for arc patches, starts the session **with the arc engaged** (so the preview
     is the genuine opening, not open-mode);
  3. for `L<id>.cap=1` slots, `fetch`es each capture from the API (presigned 302),
     `decodeAudioData`, `loadLoopBuffer` — **reusing `captureSlotsFromPayload`**;
  4. routes the orchestrator's master through a `MediaStreamAudioDestinationNode`
     and records `opts.durationSec` (20 s) in **real time** with `MediaRecorder`
     (`audio/webm;codecs=opus`);
  5. returns the bytes base64.
- **Why a bundled harness, not the production web URL:** loading
  `https://annealmusic.../render` would couple preview generation to Firebase
  uptime and to web/API version skew. Instead the `render/` bundle is **built into
  the API Docker image** and loaded via `file://` (or a localhost static mount).
  "Same engine logic" stays literally true: the same `src/audio/` is compiled in,
  pinned to the API release.
- **Server post-step:** Chromium hands back WebM/Opus; the server **repackages to
  Ogg/Opus, 96 kbps, mono** via the existing `ffmpeg` (already in the image), then
  `storage.put("previews/<patch_id>.opus", ...)` and writes back
  `preview_storage_key`, `preview_duration_ms`, `preview_status='ready'`.

### Infrastructure changes

- **Dockerfile:** add Playwright + a pinned Chromium (`playwright install --with-deps
chromium`) and the system libs it needs; `COPY` the built `render/` bundle.
  Image grows ~400 MB. ffmpeg already present.
- **Railway:** the API service gains the memory headroom for Chromium. No new
  service for v0.8 (in-process worker, §5). Flagged: if memory bites, split a
  dedicated render service consuming a queue — isolated change.
- **docker-compose (local):** the `api` service uses the same image, so Chromium
  is available locally too; `TRANSCODE_ENABLED` and a new `RENDER_ENABLED` gate it.

### Performance budget

| Stage                         | Budget                             |
| ----------------------------- | ---------------------------------- |
| Page load + harness init      | ~2 s                               |
| Real-time capture (20 s clip) | ~20 s (real time, unavoidable)     |
| Repackage (ffmpeg) + upload   | ~1–2 s                             |
| **Total per preview**         | **~24 s — under the <30 s target** |

Concurrency cap **2** (`asyncio.Semaphore`), one reused browser, page-per-render.
Worst case under gunicorn `-w 2`: 2 processes × 2 = 4 Chromium pages — documented
memory ceiling; reduce render concurrency to 1/process if it bites.

### Failure handling

- Render throws / browser crash / timeout (hard cap ~45 s) → retry up to **2×**
  with backoff; terminal failure → `preview_status='failed'`.
- `GET /preview` then returns **503**.
- The card **always** shows the deterministic static visualizer frame (§14), so a
  failed preview degrades to "no audio thumbnail," not a broken card. No separate
  waveform-heuristic fallback is needed — the static art _is_ the fallback.

## 4. Headless engine path (Option B specifics)

- **Trigger protocol:** `page.evaluate` calls the in-page global
  `window.__annealRender(payload, { durationSec, captureUrls })`. No postMessage
  ceremony; Playwright ⇄ page is a direct evaluate/return.
- **Factored out of `src/audio/`: nothing.** The harness _imports_ the existing
  `Orchestrator` unchanged — that's the whole point of Option B. The only new web
  code is `src/render/headless.ts` (the harness) + `render.html` + a `vite.config`
  render target. The engine stays untouched, honoring "don't touch engine code."
- **Capture hydration in-page** reuses `captureSlotsFromPayload` +
  `orchestrator.loadLoopBuffer`, the same functions `usePatches.loadPatch` uses, so
  a previewed patch-with-captures sounds identical to a loaded one.

## 5. Async render worker — **in-process `asyncio` queue, bounded concurrency 2**

Per the spec's v0.8 recommendation, and consistent with the in-memory rate limiter:

- A bounded `asyncio.Queue` + a small pool of worker tasks started in the FastAPI
  **lifespan**. Publish enqueues `(patch_id)`; a worker picks it up, opens its
  **own** DB session (never the request's), renders (§3), writes back.
- Concurrency gated by `asyncio.Semaphore(2)`.
- **`RENDER_BACKEND` seam** (`inprocess` default) mirrors `RATE_LIMIT_BACKEND`, so a
  future external worker (Arq/RQ) is config, not a rewrite.

**Single-instance caveats (documented, mitigated):**

- The queue is per-process; gunicorn `-w 2` ⇒ two queues. Because the enqueue and
  the render happen in the _same_ process that served the publish, this is correct
  — there's no cross-process handoff to lose.
- A process restart mid-render leaves a row stuck in `'rendering'`. **Mitigation:**
  a lifespan **startup sweep** re-enqueues any `preview_status='rendering'` older
  than a threshold; additionally `GET /preview` re-enqueues a stale `'rendering'`
  row it encounters. Idempotent (write is keyed by `patch_id`).

## 6. Preview rendering parameters

- **Duration:** 20 s (proposal accepted). Captures the full 3 s bloom + 17 s of
  evolved texture.
- **Arc patches:** render the first 20 s **with the arc engaged** — `startSession`
  in arc mode at the patch's own arc id/duration, captured for the first 20 s.
  Gives the genuine opening (settle phase), not an open-mode imitation.
- **Capture patches:** captures hydrated from R2 at render time (§4), so the
  preview includes the looped/frozen audio exactly as a loaded session would.
- **Output:** Ogg/Opus, **96 kbps, mono** — small enough to stream on a card grid;
  the existing ffmpeg path already targets libopus.
- **Immutability:** v0.7 makes `state` immutable (edit ⇒ new patch). Audio depends
  only on `state` + captures, so **a preview is written once and never
  regenerated.** Title/description edits don't touch audio → no re-render.

## 7. Moderation

- **List home (heuristic-drift rule):** a single `api/app/moderation.py` holds the
  static profanity/spam term list, referenced by the publish path (and any future
  submission flow). Env-extensible via `MODERATION_EXTRA_TERMS` (comma-separated).
  No third-party API in v0.8.
- **Screening target:** `title` + `description` text only (never the patch
  payload, which has its own strict validator). Case-insensitive, word-boundary
  match to limit Scunthorpe false positives; spam heuristics = URL-count + repeated-
  char runs.
- **Auto-reject behavior:** publish (`POST` or `PATCH` to `public`) with a match →
  **`422 {error:'content_rejected', field:'title'|'description'}`** and **no state
  change** (visibility stays as it was; nothing is enqueued). The web surfaces a
  clear inline message on the save/publish dialog.
- **Manual takedown / report flow UX:** gallery card `…` menu → **Report** →
  `ReportDialog` (reason radio: spam / inappropriate / other + optional detail) →
  `POST /reports`. Anonymous allowed (reporter_id from `x-anon-id` if present, else
  null). Uphold (§8) sets `visibility='flagged'`, which removes the patch from the
  gallery and from short links (403 under-review) within the gallery cache TTL.

## 8. Admin auth

- **`ADMIN_KEY` env var.** A `require_admin` dependency does a **constant-time**
  compare against the `x-admin-key` header; absent/empty `ADMIN_KEY` ⇒ admin
  endpoints are **disabled** (return 404, not 401 — no oracle that the panel
  exists). Missing/wrong key when enabled ⇒ `401 {error:'unauthorized'}`.
- **Panel** at `/admin`: a key-entry form on first load → stored in
  `sessionStorage` for the tab → sent as `x-admin-key`. Lists open reports (patch
  title + link to preview, reason, detail, reporter shown as "anonymous" or an
  anon-id prefix). Per-report actions: **Dismiss** (`status='dismissed'`) / **Uphold**
  (`status='upheld'` + `PATCH .../visibility {flagged}`). Legible, unstyled beyond
  that. No admin features beyond this (deferred).

## 9. Search

- **Postgres:** `to_tsvector('english', title ‖ ' ' ‖ description)` against
  `plainto_tsquery('english', q)`, served by `idx_patches_search` (GIN). Weighting:
  **title `A`, description `B`** via `setweight`, so title matches rank first.
  Language fixed to `english` (no i18n in v0.8 — documented limit).
- **SQLite (tests/local):** dialect branch → `title ILIKE '%q%' OR description
ILIKE '%q%'`. Correctness of FTS itself is asserted by a **Postgres-marked
  integration test** in the `migrations` CI job; unit tests cover the `LIKE` path +
  the sort/filter/cursor matrix.
- `q` is bounded (≤128 chars) and treated as plain text (`plainto_tsquery` is
  injection-safe; the ILIKE branch parameter-binds + escapes `%`/`_`).

## 10. Pagination — keyset cursors

- **newest:** order by `(published_at DESC, id DESC)`, cursor = last row's
  `(published_at, id)`.
- **oldest:** `(published_at ASC, id ASC)`.
- **most-loaded:** `(load_count DESC, published_at DESC, id DESC)`.
- **Cursor encoding:** `base64url(json.dumps({s:<sort>, k:[...keys]}))`. The sort
  mode is **embedded in the cursor**; a request whose `sort` disagrees with the
  cursor's `s` is rejected (`400 bad_request`) — a cursor is not portable across
  sort modes (the keyset shape differs). Opaque to the client.
- Page size **24** (CP4), `limit` capped 1–48. `next_cursor` null at the end.
  Stable under inserts (keyset, not offset) — matches the v0.7 `/patches/me` style,
  extended to a composite key.

## 11. Caching — **short edge/browser TTL on listings, immutable on previews**

- **Decision: cache, don't just rely on indexes.** Gallery is read-heavy and the
  first surface where DB load matters (spec). A short TTL cuts repeat queries
  drastically while keeping "flip to unlisted disappears within minutes" honest
  (30–60 s ≪ minutes).
- `GET /api/v1/gallery` → `Cache-Control: public, max-age=30, stale-while-revalidate=60`.
- `GET /api/v1/patches/:id/preview` (on the **302 only**) → `Cache-Control: public,
max-age=31536000, immutable` (preview bytes are immutable per patch, §6). The
  202/404/503 responses are `no-store`.
- **Middleware check:** v0.7's middleware sets `no-store` on authenticated JSON.
  Gallery/preview are **unauthenticated** GETs and must be exempted so their cache
  headers survive. CP1/CP4 verify the header on the wire (a test asserts it).
- CDN: if Cloudflare fronts Railway, the `max-age`/`s-maxage` rules apply at edge;
  otherwise browser cache + `stale-while-revalidate` still help. Documented in
  `docs/DEPLOY.md`.

## 13. Routing — introduce `react-router-dom`

No router exists today; `App.tsx` hand-parses `location.pathname` for `/p/`.
Introduce `react-router-dom` (single small dep) in `main.tsx`:

- `/` → `App` (unchanged).
- `/p/:slug` → `App` (reads `:slug` via `useParams`, replacing the regex; same
  load flow). Under-review (403) → a small "this patch is under review" state.
- `/gallery` (+ `?q=&sort=&filter=`) → `GalleryPage` (query state in the URL,
  deep-linkable).
- `/admin` → `AdminPage`.
- A subtle **"Gallery"** link in the main-app header (CP3).

Firebase's existing `** → /index.html` rewrite already serves arbitrary client
routes — no hosting change.

## 14. Card visualizer — deterministic single frame

- Reuse `src/visual/draw.ts: drawFrame` on a small (e.g. 240×135) canvas with
  `spectrum=null` (→ `VISUAL.defaultAmp`), **one tick**, no RAF loop.
- **Determinism:** `count`/`freqs` derived from the patch params (rootFreq, spread,
  density) via a small pure helper; **`phases` seeded from a hash of the payload
  string** so different patches arrange differently but the _same_ patch always
  renders the _same_ image. Loop rings drawn from the `L<id>` flags.
- Client-side only (no server-rendered image / OG card — deferred). This frame is
  always present, doubling as the preview-failure fallback (§3).

## 15. Risks & tradeoffs

| Risk / tradeoff                                    | Severity          | Mitigation                                                                                                                          |
| -------------------------------------------------- | ----------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| **Real-time render is slow** (~24 s/clip)          | Medium            | Within the <30 s budget; async + concurrency-2; previews are write-once (§6).                                                       |
| **Chromium memory under gunicorn -w 2**            | Medium            | Documented ceiling (≤4 pages); concurrency-1/process or a split render service if it bites (flagged).                               |
| **In-process queue lost on restart**               | Low–Med           | Startup sweep + `GET /preview` re-enqueue of stale `rendering` rows (§5).                                                           |
| **Renderer divergence from client**                | High if unmanaged | Option B _is_ the client runtime (Chromium); harness imports the real `Orchestrator`; capture hydration reuses client code (§3/§4). |
| **SQLite vs Postgres FTS divergence**              | Medium            | Dialect branch; FTS asserted in the Postgres CI job, `LIKE` + matrix in unit tests (§9).                                            |
| **`flagged` constraint migration**                 | Low               | Drop+recreate CHECK in the migration; model mirrors it; downgrade restores.                                                         |
| **Cursor reuse across sort modes**                 | Low               | Sort embedded in cursor; mismatch → 400 (§10).                                                                                      |
| **Load-count gaming**                              | Low               | Per-IP+patch rate-limit bucket; over-limit = silent no-op (§2).                                                                     |
