### The one real obstacle: context lifecycle

The orchestrator creates its `AudioContext` **and** post-fx chain inside
`start()` (`orchestrator.ts:172-216`) and **closes** the context in `stop()`
(`orchestrator.ts:302-310`). But the spec is explicit and non-negotiable:

> _"input is independent of session start/stop (input can be live before user
> presses Begin)"_ … _"input survives engine swaps, arc start/stop"_ … and
> _"Both feed into the shared post-fx chain."_

You cannot connect nodes across two `AudioContext`s, so input **must** route
into the **same** context that owns the post-fx chain. Therefore the
context + post-fx chain can no longer be owned by the engine/session lifecycle.

**Decision (needs sign-off — see §13.1): decouple a persistent "audio core"
(context + post-fx chain) from the engine/session lifecycle, with
reference-counted teardown.**

- New private `ensureCore()` builds `ctx` + post-fx (the body currently inside
  `start()` lines 175-216) **once**, lazily, on the first of either Begin **or**
  Connect-input — both are user gestures, so the autoplay rule is satisfied.
- `start()` calls `ensureCore()`, then builds the engine voice and routes its
  bus into `nodes.filter` (unchanged from line 219 onward).
- Session teardown (`stop()`) fades + stops **engine voices** only. It calls
  `teardownCore()` **only if `inputVoice` is not connected**. If input is live,
  the core (ctx + post-fx + input) stays up; engine buses are removed.
- `InputVoice.connect()` calls `ensureCore()`; `disconnect()` calls
  `teardownCore()` **only if no engine is running**.
- One implementation detail this creates: the arc-end fade ramps
  `masterVol → 0` (`orchestrator.ts:514-526`). When input keeps the core alive,
  teardown must **restore `masterVol` to the live `volume`** afterward (and fade
  the **engine buses**, not `masterVol`, on a keep-core stop) so a connected
  instrument isn't left muted. Flagged here, handled in CP1.

The **rejected** alternative — "input requires a running session / gets its own
context" — violates the spec (no input before Begin; cross-context connection is
impossible). The decoupling is the minimum change that satisfies the spec; its
blast radius is contained to `orchestrator.ts` and its tests (§13.7).

## 1. Audio graph integration

Input is a **sibling voice**, parallel to the engine bus, summing into the same
`filter` entry point of the shared post-fx chain. The input's audibility in the
output is gated by `monitorGain` (default **0** = silent, §6); its **analyser
tap is pre-gate** so the meter + visualizer ring react even when monitoring is
muted.

```
                         ┌───────────────── ENGINE (existing) ─────────────────┐
                         │  engine.getOutputNode() → bus(Gain, fade) ───────┐   │
                         └──────────────────────────────────────────────────│───┘
                                                                             ▼
  ┌──────────────────────── INPUT VOICE (new, optional) ───────┐    nodes.filter (lowpass, brightness)
  │ getUserMedia → MediaStreamSource                            │            │
  │   → [channelSum Gain]   (only if stereo summed to mono)     │       ┌────┴───────────────┐
  │   → highpass  (80 Hz)                                       │       │                    │
  │   → compressor (DynamicsCompressor)                         │   dryGain            convolver→wetGain
  │   → shaper (WaveShaper, bypassed by default)                │       │                    │
  │   → voiceGain  ("Input Level")                              │       └────────┬───────────┘
  │   → driftFilter (Biquad peaking, freq ← mean drift)         │              master
  │        ├─────────────► analyser  (meter + ring; ALWAYS on) │                │
  │        └─► monitorGain (0 = muted default) ────────────────┼──► nodes.filter (join here)
  └─────────────────────────────────────────────────────────────┘              │
                                                                              masterVol (volume)
                                                                                │
                                                                             analyser (post-fx, visualizer spectrum)
                                                                                │
                                                                            destination
```

- The input joins **at `nodes.filter`**, exactly where engine buses join, so the
  live voice is colored by the same brightness lowpass, reverb (`space`), and
  master volume as the engines — it sits _inside_ the field, not on top.
- `monitorGain` is the **only** path from input to the speakers. Default 0 →
  input is processed, metered, and visualized, but inaudible (feedback-safe).
- The input's **own** `analyser` (returned by `getAnalyser()`) is separate from
  the post-fx analyser; it always sees the signal regardless of `monitorGain`.

## 2. `getUserMedia` constraints (final)

```ts
const constraints: MediaStreamConstraints = {
  audio: {
    echoCancellation: false, // AEC notch-filters + dynamically ducks; destroys sustained tones
    noiseSuppression: false, // spectral gating eats quiet/ambient instrument tails
    autoGainControl: false, // AGC pumps level against our own compressor; we own gain staging
    channelCount: 1, // request mono; see stereo note below
    deviceId: deviceId ? { exact: deviceId } : undefined, // omit ⇒ default device
  },
  video: false,
};
```

**Why each processing flag is disabled:** the three "voice cleanup" DSP blocks
are tuned for speech intelligibility on a call, not for music. `echoCancellation`
applies an adaptive filter that notches and ducks anything resembling a returning
echo — catastrophic for sustained, harmonically rich instrument tones.
`noiseSuppression` is a spectral gate that chews quiet passages and reverb tails.
`autoGainControl` continuously re-normalizes level, fighting our deliberate
`DynamicsCompressor` and `voiceGain`. We want the **raw** signal so it hits the
engine clean (this is also the README promise in CP3).

**Safari quirks (documented in COMPAT, CP3):**

- Safari requires `getUserMedia` to be called from a **user gesture** — satisfied
  by the Connect button click (same constraint the audio context already obeys).
- Safari has **historically ignored** some constraint flags (notably
  `autoGainControl` and at times `noiseSuppression`); we still pass them (no harm)
  but treat them as best-effort. This is exactly why the **feedback guard** (§7)
  and the default-muted monitor matter more on Safari.
- `enumerateDevices()` returns **empty `label`s until permission is granted** on
  all browsers (privacy). The UI must render placeholder labels pre-grant (§11).

**Stereo (flagged decision — §13.4):** v0.5 ships **mono-summed** by default
(`channelCount: 1`). The spec allows preserving channels "if the device is
explicitly stereo." **Recommendation: defer true stereo to a later slice** and
ship mono-only in v0.5 — a single routed voice is the v0.5 contract, the drift
filter + post-fx are mono-centric, and stereo doubles the chain + meter
complexity for marginal v0.5 value. If a stereo device is detected we still sum
to mono (the `channelSum` Gain node in §1 is only inserted when the source
reports >1 channel). Confirm or override.

## 3. Device enumeration & switching

**Recommendation (confirmed per the prompt's lean): ship a device picker.**
Audio interfaces (USB DI for bass/guitar, mics) are the target use case, and the
default device is rarely the right one. The picker:

- `devices.ts: enumerateInputDevices()` wraps
  `navigator.mediaDevices.enumerateDevices()`, filters `kind === 'audioinput'`,
  and returns `InputDevice[]` (`{ deviceId, label }`). Labels are empty pre-grant.
- A `defaultInputDevice(devices)` helper picks the entry with `deviceId` `''` or
  `'default'`, else the first.
- Selecting a device calls `InputVoice.connect(deviceId)` again: the existing
  stream is stopped and a fresh `getUserMedia({ deviceId: { exact } })` is opened.
  This is a **re-connect on an existing user gesture** (the picker click), so it
  never trips the "don't re-prompt aggressively" rule — permission is already
  granted, so no new prompt appears.
- The list refreshes on `devicechange` (§9) while connected.

## 4. Voice processing chain (finalized)

```
MediaStreamSource
  → channelSum (Gain, only when source.channelCount > 1)   // sum stereo → mono
  → highpass     BiquadFilter  type=highpass  freq=80 Hz  Q=0.7
  → compressor   DynamicsCompressor  threshold=-24 dB  ratio=3  knee=30  attack=0.01 s  release=0.25 s
  → shaper       WaveShaper  (curve=null ⇒ bypassed; default OFF)
  → voiceGain    Gain  value=1.0   // "Input Level" slider, 0..2 (≈ −∞..+6 dB)
  → driftFilter  BiquadFilter  type=peaking  freq=700 Hz(base)  Q=1.2  gain=+4 dB   // freq ← mean drift
  ├─► analyser   AnalyserNode  fftSize=1024  smoothing=0.6     // meter + ring; ALWAYS live
  └─► monitorGain Gain  value=0 (muted default) → nodes.filter
```

**Compressor defaults — justification.** These are deliberately _gentle_,
"glue" settings, not effect-level compression:

- `threshold −24 dB`, `ratio 3:1` — catches only peaks above a moderate level
  with mild 3:1 reduction; a plucked bass transient or vocal peak is tamed, but
  quiet playing passes untouched. Keeps the instrument from spiking the post-fx
  and the feedback guard.
- `knee 30 dB` — a very soft knee, so compression eases in inaudibly rather than
  clamping at a hard point (musical, not "pumping").
- `attack 0.01 s` — fast enough to catch transients without clicking.
- `release 0.25 s` — slow enough to avoid audible breathing, fast enough to
  recover between notes.

**Drift-modulated filter mapping (the texture-binding move).** The drift loop's
**mean detune across partials** (cents, realistically ~±30, hard-clamped ±60 by
`drift.ts:DETUNE_CLAMP`) drives the **center frequency** of the `peaking`
`driftFilter`. So the live voice gets a slow, wandering formant emphasis that
breathes on the _same_ Ornstein–Uhlenbeck/Kuramoto field as the engine partials
— that shared motion is what makes it read as "part of the field," not "an
instrument over a backing track." Mapping (lives in `InputVoice`, one place):

```
n = clamp(meanCents / 60, -1, 1)                  // normalize to [-1, 1]
freq = 700 * 2 ** (n * 0.5)                        // ±0.5 octave sweep: ~495 Hz … ~990 Hz
driftFilter.frequency.setTargetAtTime(freq, t, 0.08)   // smoothed so 20 Hz updates don't zipper
```

Which value drives which axis: **mean drift → driftFilter center frequency**
(peaking band position). Magnitude (Q, gain) is fixed; only the band _position_
moves, so the coloration wanders but the level stays stable. The base frequency,
octave range, and Q are module constants, easy to tune in CP3.

## 5. Latency measurement strategy

Web Audio does **not** expose true mic→node input latency, so v0.5 surfaces a
documented **estimate** (the spec says "surface the measured value … so the user
can compensate" — we surface a best-available estimate and say so).

`latency.ts` is the **single home** for the formula (heuristic-drift rule), used
by both `InputVoice.connect()` (returns it) and the UI readout:

```ts
export function estimateLatencyMs(ctx: AudioContext): number {
  // baseLatency: context's internal block latency (output side).
  // outputLatency: full output pipeline latency where supported (Chrome/FF);
  //   absent on Safari ⇒ fall back to 2× baseLatency as a coarse proxy.
  const base = ctx.baseLatency ?? 0;
  const out = (ctx as { outputLatency?: number }).outputLatency ?? base;
  return Math.round((base + out) * 1000);
}
```

- Surfaced as a mono-caps readout, e.g. `~30 MS INPUT LATENCY` (§11).
- This is an **output-pipeline estimate**; the true round trip includes the
  device's input buffer, which the API hides. CP3 documents the formula and (if a
  loopback cable is available) compares it to a measured round trip in COMPAT.
- It is always a **positive finite number** (asserted in tests, CP1).

## 6. Monitoring vs. silent input — gain stages spelled out

| Stage          | Node          | Default         | Role                                                               |
| -------------- | ------------- | --------------- | ------------------------------------------------------------------ |
| Input Level    | `voiceGain`   | `1.0`           | User-controlled trim (`setLevel`), 0..2.                           |
| Drift color    | `driftFilter` | freq←drift      | Texture binding; not a level stage.                                |
| **Meter/ring** | `analyser`    | always live     | Taps **pre-`monitorGain`** so visuals react even when muted.       |
| **Monitoring** | `monitorGain` | **`0` (muted)** | The only path to the speakers. `setMonitoring(true)` ramps → `1`.  |
| Field color    | post-fx       | shared          | brightness/space/volume apply equally to engine + monitored input. |

**Default muted monitoring** is the feedback-safe state: the user hears
themselves acoustically or through their amp/instrument headphones, and hears the
Anneal Ambiance field (ideally through headphones). The visualizer + VU meter still
react because the analyser is upstream of the gate. Enabling monitoring ramps
`monitorGain` 0→1 over ~80 ms (no click) and shows a feedback warning tooltip
(§7, §11).

## 7. Echo / feedback guard

Two layers:

1. **Affordance (CP2):** monitoring is **off by default**; the toggle carries a
   tooltip — _"Monitoring plays your input through the speakers. Use headphones
   to avoid feedback."_ Pre-prompt copy also recommends headphones.
2. **Automatic gate (CP3):** while monitoring is **on**, sample the input
   analyser RMS at meter cadence. If RMS stays above a threshold
   (`≈ 0.9` normalized) **continuously for > 2 s**, treat it as runaway feedback:
   ramp `monitorGain` down (dim, not hard-cut) and raise a toast — _"Possible
   feedback — reduce input or use headphones."_ The detector only runs when
   monitoring is enabled (no point otherwise) and uses a sustained-time test so a
   loud strum doesn't trip it. The RMS/threshold/duration constants live beside
   the meter logic.

## 8. Permission UX (state machine)

`InputState = 'idle' | 'prompting' | 'connected' | 'denied' | 'error'`
(plus `'dismissed'` handled as a return to `idle`).

| State       | Trigger                                   | UI behavior                                                                                                                                                  |
| ----------- | ----------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `idle`      | initial / after disconnect / dismissed    | Panel shows **"Connect input"** CTA + one-line pre-prompt copy (why + "audio never leaves your device").                                                     |
| `prompting` | user clicks Connect → `getUserMedia` open | CTA shows a spinner / "Allow microphone access…"; button disabled to prevent double-prompt.                                                                  |
| `connected` | permission granted, chain built           | Device picker, level meter, Input Level, Monitoring toggle, latency readout, Disconnect (§11).                                                               |
| `denied`    | `NotAllowed`/`PermissionDenied` error     | Clear per-browser fix copy (Chrome: site settings; Safari: Settings ▸ Websites ▸ Microphone) + **Retry** (re-prompts on click — one prompt per user action). |
| `error`     | `NotFoundError`/`NotReadableError`/other  | "No input device found" / "Device is in use" message + Retry.                                                                                                |

- **One prompt per user action.** Connect/Retry each map to exactly one
  `getUserMedia` call. We never auto-retry or loop prompts.
- **Typed errors:** `InputVoice.connect()` rejects with a typed
  `InputError { kind: 'denied' | 'notfound' | 'unreadable' | 'unsupported' | 'unknown'; message }`
  (mapped from the `DOMException.name`), never a bare throw (spec: "Permission
  errors are typed").
- Pre-prompt copy is **precise, not overclaiming**: input is processed entirely
  in-browser via Web Audio and is never uploaded — true, and stated as such.

## 9. Device-change / suspend recovery

- **`devicechange` (while connected):** subscribe to
  `navigator.mediaDevices.devicechange`. On fire, re-`enumerateInputDevices()` and
  refresh the picker. If the **active** device has vanished from the list, attempt
  a graceful reconnect to the **default** device; emit a `device-changed` event
  the UI surfaces as a toast (_"Input device changed — reconnected to default"_).
  If reconnect fails, transition to `error` with the disconnected message; the
  **audio graph stays alive** (engines/post-fx untouched).
- **Stream/track ended (unplug):** each `MediaStreamTrack` gets an `onended`
  handler → same graceful-reconnect-to-default path → UI shows "device
  disconnected" without crashing the graph.
- **Tab hidden (`visibilitychange`):** **do not** suspend or tear down the input
  stream — we don't fight the browser's own context suspension. Per-browser
  behavior (Chrome keeps the stream; Safari/Firefox may suspend the context with
  audio) is observed and documented in COMPAT (CP3). The graph survives
  background→foreground transitions.
- All listeners are added on connect and removed on disconnect (no leaks).

## 10. Module plan

```
src/input/
  types.ts        // InputDevice, InputState, InputConfig, InputError, InputVoiceEvents
  InputVoice.ts   // class: owns getUserMedia stream + the §4 node chain; single output node; analyser
  latency.ts      // estimateLatencyMs(ctx)  — the lone latency formula (heuristic-drift rule)
  devices.ts      // enumerateInputDevices(), defaultInputDevice(devices)

src/audio/
  orchestrator.ts // EXTEND: ensureCore()/teardownCore() (§0), optional InputVoice ownership,
                  //         route input → nodes.filter, feed mean drift each tick, lifecycle rules
```

`InputVoice` public API (per the spec):

```ts
class InputVoice {
  constructor(ctx: AudioContext);
  connect(
    deviceId?: string,
  ): Promise<{ deviceLabel: string; latencyMs: number }>;
  disconnect(): Promise<void>;
  setLevel(gain: number): void; // voiceGain
  setMonitoring(enabled: boolean): void; // monitorGain 0↔1, smoothed
  setDriftModulation(meanCents: number): void; // §4 mapping → driftFilter.frequency
  getOutputNode(): AudioNode; // monitorGain (what the orchestrator wires to filter)
  getAnalyser(): AnalyserNode; // pre-gate analyser
  isConnected(): boolean;
}
```

Orchestrator additions: `getInputVoice()`, `connectInput(deviceId?)`,
`disconnectInput()`, and inside `startDrift()` the tick also computes
`mean = Σ detune / n` and calls `inputVoice?.setDriftModulation(mean)`. The drift
loop currently only runs while a session is live; since input can predate Begin,
a **lightweight drift-modulation source** is needed when input is connected but
no session runs — **decision: when input is connected and no session is running,
run the same 20 Hz timer but in a "modulation-only" mode** (compute drift state
purely for the mean, no engine fan-out). Simplest correct option; flagged in
§13.3.

## 11. UI plan

**Input panel** — a new `InputPanel.tsx`, placed near the engine selector
(below the Mode/Engine row, above or beside `ControlPanel`). Always-visible
section header `INPUT` in the same mono-caps style as `Mode`/`Engine`.

- **`idle`:** "Connect input" button (amber, matches Begin styling) + one-line
  pre-prompt copy + a small "use headphones" hint.
- **`prompting`:** disabled button + "Allow microphone access…".
- **`denied` / `error`:** copy + **Retry**.
- **`connected`:** a compact stack —
  - **Device picker** — dropdown (`<select>`, accessible) of input devices;
    handles empty labels with `Input 1`, `Input 2` placeholders.
  - **VU/peak meter** — horizontal LED-style strip beneath the picker, **warm
    amber** gradient, **peak indicator** dot at the top of the trail, **clipping
    indicator** (brief red) when RMS > 0.95. Updates at **~30 Hz** off
    `requestAnimationFrame`, throttled, reading `getAnalyser()` time-domain data.
  - **Input Level** slider (0..2, default 1).
  - **Monitoring toggle** — off by default; tooltip warns about feedback.
  - **Latency readout** — mono caps, e.g. `~30 MS INPUT LATENCY`.
  - **Disconnect** affordance.
- **Local state, not the param store.** A `useInput` hook holds `InputState`,
  device list, selected device, level, monitoring, latency, device label — none
  of it is persisted or URL-encoded (§0). The hook talks to
  `orchestrator.connectInput/disconnectInput` and the `InputVoice` setters.
- **Arc/session lock semantics (confirmed — §13.5):** the input panel stays
  **fully user-controllable during arc mode and open mode alike**. The arc never
  touches input (the prompt's "simpler default"). So unlike the engine selector /
  sculpt sliders, **the input panel is never disabled** by `arcLocked`/`isPlaying`.

## 12. Visualizer integration (confirmed, with the wiring)

**Confirmed: input amplitude pulses a faint additional ring around the central
halo** — subtle, must not dominate.

- `DrawState` gains an optional `inputLevel?: number` (0..1 RMS proxy).
- `Visualizer.tsx` reads `orchestrator.getInputVoice()?.getAnalyser()` once per
  frame, computes a smoothed RMS, and passes it as `inputLevel`.
- `drawFrame` draws **one** soft amber stroked ring at `baseR * haloRadiusMultiple`,
  with `opacity` and a slight `radius` swell driven by `inputLevel` (e.g.
  `alpha = 0.05 + inputLevel * 0.18`). When no input is connected,
  `inputLevel` is `undefined` and the ring is skipped — **zero visual change**
  for users who never connect input. New palette/visual constants
  (`inputRing*`) keep magic numbers out of `draw.ts`.
- It reads as a breathing halo edge that swells when you play — part of the field,
  tuned for subtlety in CP3 (opacity/curve iteration).

## 13. Risks & tradeoffs

1. **Context lifecycle decoupling (§0).** The core change. Touches `start()`,
   `stop()`, arc-end fade, and orchestrator tests (which currently assert ctx
   close on stop). Mitigation: reference-counted teardown + a focused test that
   input survives a session stop and an engine swap, and that the ctx is closed
   only when both are inactive. **This is the item I most want sign-off on.**
2. **Latency is an estimate, not a measurement.** Web Audio hides true input
   latency. We surface an output-pipeline estimate and label it as such; COMPAT
   documents the formula + any loopback comparison. Accept, or downgrade the UI to
   "approx" wording — flag.
3. **Drift modulation when no session runs (§10).** Running a modulation-only 20 Hz
   timer while input is connected pre-Begin is the simplest way to keep the live
   voice breathing before a session starts. Alternative: freeze the drift filter
   at center until Begin. **Recommendation: modulation-only timer** (keeps the
   texture-binding promise even for solo practice). Flag if you'd rather freeze.
4. **Mono-only in v0.5 (§2).** Stereo preserved-through-chain is deferred;
   stereo devices are summed to mono. Recommended for scope. Confirm/override.
5. **Input never locks during arc (§11).** Per the spec's simpler default, the arc
   doesn't touch input and the panel stays live. Confirmed default; easy to lock
   later if undesired.
6. **Safari best-effort constraints (§2).** Safari may ignore `autoGainControl`/
   `noiseSuppression`; the feedback guard + default-mute are the safety net.
   Documented, not "fixed" (can't be — it's the UA).
7. **Mock surface (CP1).** `audioMock.ts` must grow `createDynamicsCompressor`,
   `createWaveShaper`, `createMediaStreamSource`, and a `navigator.mediaDevices`
   stub (`getUserMedia`, `enumerateDevices`, `devicechange`). Contained to the
   test harness; no production impact.

### Confirmations requested at the gate

- **(a)** OK to **decouple the audio core lifecycle** from the session, with
  reference-counted teardown (§0, §13.1)? — _the one structural change._
- **(b)** **Mono-only** input for v0.5 (stereo summed), stereo deferred (§2)?
- **(c)** **Modulation-only drift timer** while input is connected pre-Begin (§13.3)?
- **(d)** Input panel **never locks** during arc/session (§11)?
- **(e)** Latency surfaced as a labeled **estimate** (§5, §13.2)?

### Explicitly deferred / flagged (per the prompt)

- **Loop pedal** → v0.6 (deferred).
- **Pitch detection / harmonization** → not on roadmap. _Flag: I do not recommend
  adding it in v0.5; v0.5 is texture-blending only, as specified._
- **MIDI / multi-input / multi-track** → not on roadmap.
- **Sidechain / ducking (input ducks engine or vice versa)** → _flag: not in
  v0.5. Recommend deferring; it would couple input level to engine gain and
  complicate the "sibling voice" model. Easy add later if wanted._
- **Built-in tuner** → not on roadmap (easy later).
- **Saving input device in URL** → never (runtime concern; §0).
