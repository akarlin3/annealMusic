# Implementation Plan - AnnealMusic v6.0 · Lesson Foundation

This document outlines the design and step-by-step implementation plan for Standing up the **Education Surface** (`/learn`) in AnnealMusic v6.0.

---

## 1. Final Data Model

We will implement three new tables in PostgreSQL (production) and SQLite (testing/local development) mapped using SQLAlchemy's modern `Mapped` and `mapped_column` type annotations.

### `tracks` Table

Represents curriculum topics aligned with the pedagogical pillars.

```sql
CREATE TABLE tracks (
  id            UUID PRIMARY KEY,
  slug          TEXT UNIQUE NOT NULL,
  title         TEXT NOT NULL,
  description   TEXT,
  position      INTEGER NOT NULL DEFAULT 0,
  color         TEXT,                       -- Tailwind-compatible or HSL hex/rgba identifier
  archived_at   TIMESTAMPTZ,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### `lessons` Table

Represents a structured sequence of steps teaching a particular topic.

```sql
CREATE TABLE lessons (
  id                UUID PRIMARY KEY,
  track_id          UUID NOT NULL REFERENCES tracks(id) ON DELETE CASCADE,
  slug              TEXT NOT NULL,
  title             TEXT NOT NULL,
  description       TEXT,
  difficulty        TEXT NOT NULL DEFAULT 'intro',  -- 'intro' | 'intermediate' | 'advanced'
  estimated_minutes INTEGER NOT NULL DEFAULT 10,
  position          INTEGER NOT NULL DEFAULT 0,
  prerequisites     UUID[] NOT NULL DEFAULT '{}',   -- Array of lesson IDs
  archived_at       TIMESTAMPTZ,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(track_id, slug)
);
```

### `lesson_steps` Table

Represents one individual screen/card inside a lesson.

```sql
CREATE TABLE lesson_steps (
  id            UUID PRIMARY KEY,
  lesson_id     UUID NOT NULL REFERENCES lessons(id) ON DELETE CASCADE,
  position      INTEGER NOT NULL,
  type          TEXT NOT NULL,                    -- 'text' | 'demo' | 'prompt' | 'reflection'
  config        JSONB NOT NULL,                   -- Step type-specific config payload
  UNIQUE(lesson_id, position)
);
```

## 2. Final API Surface

We will implement public retrieval endpoints and secure administrative CRUD endpoints in the FastAPI backend:

### Public Curriculum Endpoints

- `GET /api/v1/tracks` - List all active/public tracks.
- `GET /api/v1/tracks/{slug}` - Retrieve a single track, including all its associated lessons.
- `GET /api/v1/lessons/{track_slug}/{lesson_slug}` - Fetch a specific lesson with all its nested steps in sorted position order.

### Admin CRUD Endpoints (Authenticated/Admin)

- `POST /api/v1/admin/tracks` - Create a new track.
- `PATCH /api/v1/admin/tracks/{id}` - Partially update a track.
- `POST /api/v1/admin/lessons` - Create a new lesson.
- `PATCH /api/v1/admin/lessons/{id}` - Partially update a lesson.
- `POST /api/v1/admin/lesson-steps` - Add a step to a lesson.
- `PATCH /api/v1/admin/lesson-steps/{id}` - Partially update a step.

## 3. Bridge Extension (postMessage Transport & Lesson APIs)

We will extend v5.0's RPC bridge to communicate bidirectionally across `iframe` boundaries:

### `PostMessageTransport`

- A same-origin transport using the standard `postMessage` API.
- If running inside the iframe (Server): forwards responses/notifications to `window.parent`.
- If running in the outer window (Client): targets the iframe's `contentWindow`.
- Validates the origin to ensure security.

```typescript
export class PostMessageTransport {
  private onMessageCallback: (msg: any) => void = () => {};
  private iframeEl: HTMLIFrameElement | null = null;

  constructor(iframeEl?: HTMLIFrameElement) {
    this.iframeEl = iframeEl || null;
    window.addEventListener('message', this.handleMessage);
  }

  private handleMessage = (event: MessageEvent) => {
    if (event.origin !== window.location.origin) return;
    this.onMessageCallback(event.data);
  };

  send(message: any): void {
    if (this.iframeEl) {
      // Client -> Iframe (Server)
      this.iframeEl.contentWindow?.postMessage(message, window.location.origin);
    } else {
      // Iframe (Server) -> Parent Client
      window.parent.postMessage(message, window.location.origin);
    }
  }

  onMessage(callback: (msg: any) => void): void {
    this.onMessageCallback = callback;
  }

  close(): void {
    window.removeEventListener('message', this.handleMessage);
  }
}
```

### New RPC Methods (Additive)

1. **`anneal.lesson.loadPatch(state)`**: Hydrates the active parameter state with the given patch.
2. **`anneal.lesson.loadPiece(state)`**: Loads a full piece configuration (stubs for v6.0).
3. **`anneal.lesson.highlight(controlKey)`**: Temporary visual highlight on the specified parameter control (renders a subtle pulse border/glow for 3 seconds).
4. **`anneal.lesson.constrain(constraints)`**: Replaces the active allowed parameter control key array in `useParamStore`.
5. **`anneal.lesson.releaseConstraints()`**: Clears the constraint list, restoring full control.

## 4. `/learn` Bundle Architecture

We will create a clean separate entry point for the education UI:

- **`learn.html`** - Serves as the HTML shell (loaded via `/learn` in the FastAPI app or Vite Dev Server).
- **`src/learn/learnEntry.ts`** - Renders the React root for the education surface.

  - Excludes the audio/worklet systems which load only inside the embedded main application iframe.

## 5. Lesson Player Layout

We will build a high-fidelity visual layout using Vanilla CSS:

- **Desktop (Split View):**
  - **Left Panel (40%):** The lesson chrome. Header with track & lesson titles, exit button. Main body displaying the current step content, navigation buttons (Next/Prev), and progress indicators (minimalist dots).
  - **Right Panel (60%):** Embedded iframe showing the live main AnnealMusic app (`/`).
- **Mobile Responsive Layout (Vertical Stack):**
  - Linear stack with the lesson chrome placed on top.
  - Collapsible iframe below it with a toggle button ("Expand App" / "Minimize App") to adjust iframe height.
- **Calm by Design adherence:**
  - Standard progress indicator uses simple dots (`• • ◦ ◦ ◦`). No gamification (no streaks, level gauges, XP progress bars, or celebratory fanfare sound effects).

## 6. Step Type Implementations

We will construct highly polishable React step components inside `src/learn/stepTypes/`:

1. **`TextStep`**: Renders beautifully formatted Markdown content, headers, and optional descriptive layout graphics.
2. **`DemoStep`**: Automatically executes the `loadPatch`/`loadPiece` bridge methods to configure the synthesizer, with an option to trigger auto-play. Renders alongside explanation cards.
3. **`PromptStep`**: Displays a manual challenge prompt ("Sculpt brightness while coupling is high") and requires the user to click an "I tried it" action button to proceed.
4. **`ReflectionStep`**: Provides a beautiful, focused text area allowing the user to reflect on the concept (unblocked, optional next).

## 7. Constraint System

We will implement a bulletproof restriction mechanism in the main app:

- Add a new reactive property `constraints: string[] | null` to the global `useParamStore`.
- Update `setParam` and `setEngineParam` in `useParamStore`: if `constraints` is active, reject any changes to parameters whose keys are not in the constraint list.
- In `ControlPanel.tsx`, disable controls when `constraints` is active and the current control key is not present in the allowed list:
  ```typescript
  const allowed = !constraints || constraints.includes(key);
  const disabled = arcLocked || !allowed;
  ```
- Add a temporary highlight state in the main app to support `anneal.lesson.highlight(controlKey)`. If triggered, it sets a temporary state causing that slider to render with a beautiful, pulsating amber border for 3 seconds.

## 10. Risks & Mitigations

| Risk                                | Mitigation                                                                                                                                                                                                 |
| :---------------------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Iframe postMessage Latency**      | Handshakes and simple messages are same-origin and instantaneous. We will design lightweight non-blocking RPC calls with robust error fallbacks.                                                           |
| **Constraint Tampering by Users**   | We enforce restrictions in the frontend store level inside the iframe. Since there's no state bypass allowed in production, this is fully secure for educational pacing.                                   |
| **Lesson Surface feeling detached** | We embed the actual, fully functional live app in the player. By using `highlight` and `constrain`, the lesson directly interacts with the knobs the user sees, creating a unified interactive experience. |
