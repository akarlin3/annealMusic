# AnnealMusic v5.4 · Pyodide Foundation + Script Sandbox

This document outlines the architecture, data modeling, sandboxing strategy, and integration roadmap for introducing local, sandboxed Python scripting into the AnnealMusic research surface.

---

## 1. Pyodide Version Pin & Upgrade Policy

- **Pinned Version:** Pyodide `0.26.4`
- **Underlying CPython Version:** CPython `3.12`
- **Upgrade Policy:**
  - Minor version updates (e.g., `0.27.x`) will only occur at major milestone milestones (e.g., `v5.5`, `v5.6`).
  - Any version bump requires full validation of the sandbox boundaries, testing that package load operations from JS still resolve properly, and verifying the `anneal` Python module interop.

---

## 2. Sandbox Model & Enforcement

To prevent malicious execution and protect local user environments, the scripting surface runs with strict isolation rules:

- **Web Worker Boundary:** Pyodide runs entirely inside a dedicated background `Web Worker` thread (`pyodide-worker.js`). It has no access to the browser `window`, `document`, main-thread `localStorage`, or other origins.
- **Network Egress Block:**
  - Pyodide provides browser fetch bindings via the `pyodide.http` family.
  - In worker initialization, we override `pyodide.http.pyfetch` and `pyodide_js.http.pyfetch` to raise a `PermissionError` when called.
  - Standard Python network libraries (`urllib`, `socket`, `http.client`, `requests`) are monkeypatched in Python to raise a `PermissionError`.
- **File System Access:** Scripts operate strictly within Pyodide's virtual in-memory file system (`MEMFS`). No local filesystem escape or interaction is possible.
- **Subprocesses:** Pyodide does not compile OS-level bindings; attempts to use `subprocess` or `os.system` raise a runtime exception.
- **Imports:** Only standard CPython library modules and the bundled `numpy` library may be imported. Third-party imports are completely blocked in `v5.4`. (Curated white-listing will be introduced in `v5.5`).

## 3. The `anneal` Python Module API

The worker instantiates a custom Python-side `anneal` module built on top of Pyodide's JS interop bindings:

```python
import sys
import types
import pyodide_js
import numpy as np

# Construct nested namespace structure
anneal = types.ModuleType("anneal")
sys.modules["anneal"] = anneal

anneal.state = types.ModuleType("state")
anneal.engine = types.ModuleType("engine")
anneal.session = types.ModuleType("session")
anneal.datalog = types.ModuleType("datalog")
```

### Full Method Surface:

#### `anneal.state`

- `anneal.state.get() -> dict`: Returns the full cached state (sculpt params, engine ID, engine params, tuning, and mode) synchronously.
- `anneal.state.set(params: dict) -> None`: Submits a state patch to update active parameters asynchronously.
- `anneal.state.set_engine(engine_id: str) -> None`: Asynchronously swaps the active synthesizer engine.
- `anneal.state.subscribe(keys: list[str], callback: callable) -> str`: Subscribes to changes on specific keys. Triggers `callback(diff: dict)` on update. Returns a subscription ID.
- `anneal.state.unsubscribe(sub_id: str) -> None`: Cancels a state subscription.

#### `anneal.engine`

- `anneal.engine.get_spectrum() -> np.ndarray`: Returns the latest magnitude spectrum as a 512-bin floating-point NumPy array.
- `anneal.engine.get_partials() -> list[dict]`: Returns a list of dictionaries `[{"freq": float, "amp": float}]` representing active partial frequencies and linear amplitudes.

#### `anneal.session`

- `anneal.session.start() -> None`: Starts the active listening session.
- `anneal.session.stop() -> None`: Stops the active listening session.

#### `anneal.datalog`

- `anneal.datalog.start(mode: str = "standard") -> None`: Starts datalogging in the specified mode (`"lightweight"`, `"standard"`, `"full"`).
- `anneal.datalog.snapshot() -> np.ndarray`: Returns the latest N ticks from the datalogger ring buffer as a structured NumPy array.
- `anneal.datalog.stop() -> None`: Stops the datalogger.

#### `anneal.version() -> str`

- Returns the current system and schema version string.

## 4. Worker Architecture for Pyodide

To keep `/research` page entry lightweight and highly responsive, Pyodide is **lazy loaded** on first script run:

```
[Main Thread React app]
   |
   |-- (First Script Run)
   V
[Start Pyodide Worker] -> Instantiates Web Worker
   |
   +--> Loads Pyodide v0.26.4 (CDN) & NumPy
   +--> Sets up 'anneal' Python module & overrides network APIs
   +--> Subscribes to parameter updates over BroadcastChannel
```

- **Local Cache Sync:** To resolve `state.get()`, `engine.get_spectrum()`, and `engine.get_partials()` synchronously without causing UI freezes, the worker maintains a local cache of parameter store states, active spectrum arrays, and partial lists.
- **State Updates:** The worker listens to parameter updates using a `BroadcastChannel` connected to `BridgeServer`. This maintains near-instant synchronization without manual polling.
- **FFT Data Flow:** FFT spectrum frames are streamed from the main thread only when a script is actively running or when a session is active, keeping communication overhead minimal.

## 5. Bridge-to-Python Message Protocol

Messages between the main thread and the worker are serialized using a strict command-response protocol over `postMessage`:

### Main to Worker:

- `{"type": "init"}`: Triggers cold startup and registers modules.
- `{"type": "run", "code": "..."}`: Submits Python code to execute in the worker script environment.
- `{"type": "repl", "code": "..."}`: Submits a single-line or multi-line block to the REPL workspace.
- `{"type": "cache-update", "payload": {...}}`: Dispatches updated state parameters to the local cache.
- `{"type": "fft-update", "spectrum": [...], "partials": [...]}`: Updates the active spectrum arrays.

### Worker to Main:

- `{"type": "status", "stage": "loading" | "ready" | "error", "progress"?: number, "error"?: string}`: Reports loading progress and CDN fetch speeds.
- `{"type": "stdout", "text": "..."}`: Streams print outputs and trace outputs.
- `{"type": "stderr", "text": "..."}`: Streams error output and stack tracebacks.
- `{"type": "run-complete", "success": boolean}`: Emitted when a script run finishes.
- `{"type": "repl-complete", "success": boolean, "result"?: string}`: Emitted when a REPL evaluation completes.
- `{"type": "bridge-call", "method": "...", "params": {...}}`: Delegates a state update call back to the main thread's `BridgeClient`.

## 6. Editor UX (CodeMirror 6)

- **Appearance:** Styled as a beautiful glassmorphic container in alignment with the existing Research UI system.
- **Syntax Highlighting:** Fully syntax-highlighted editor utilizing the official `@codemirror/lang-python` package.
- **Basic Auto-Complete:** Simple identifier completions matching Python keywords and local namespaces (e.g. `anneal.state`).
- **Run Flow:** Runs asynchronously, feeding print outputs and error tracebacks directly into the bottom output stream panel. Include a "Stop" button that restarts the Web Worker if a script hangs.
- **Save/Load:** Interacts with the backend `/api/v1/scripts` CRUD endpoints if the researcher is signed in, falling back to local browser storage (`localStorage`) if unsigned.

---

## 7. REPL Design

- **Console Prompt:** An interactive prompt styled with standard Python REPL prefixes (`>>>` and `...` for multi-line inputs).
- **Workspace Persistence:** Runs in the same Python global scope as the script editor, enabling researchers to define helper functions in the editor and immediately interact with them in the console.
- **Navigation & Formatting:** Implements standard navigation keys:
  - **Shift+Enter**: Execute multi-line statement blocks.
  - **Tab**: Inserts four spaces for formatting indentation.
  - **Up/Down Arrows**: Cycles through history logs of executed REPL commands.

## 8. Backend `user_scripts` Data Model

Stored in the database and linked to the user's persistent anonymous profile:

```sql
CREATE TABLE user_scripts (
  id            UUID PRIMARY KEY,
  user_id       UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  name          TEXT NOT NULL,
  source        TEXT NOT NULL,
  language      TEXT NOT NULL DEFAULT 'python',
  visibility    TEXT NOT NULL DEFAULT 'private',  -- 'private' | 'unlisted'
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

### Constraints:

- `visibility`: Constrained to `'private'` and `'unlisted'`.
- Quota limit: Max 50 user scripts per profile.

## 9. Bundle Size Accounting

- **CDN Optimization:** By sourcing the compiled WASM environment and standard Python runtime packages from CDN, the Web browser bundle is kept lightweight.
- **Frontend Overheads:** Scripting UI components (CodeMirror + workers hookups) add approximately **~120KB** (gzipped) to the research bundle. This has zero impact on the primary customer `/listen` player page bundle.

## 11. Risks & Mitigations

1. **Pyodide Fetch/CDN Timeout:** Pyodide requires reliable internet access to download WASM runtime components on first start.
   - _Mitigation:_ We display a clear status loader during execution initialization. If the CDN fetch fails, we show an elegant fallback banner with a manual "Retry" button.
2. **Infinite Loops Hanging Worker:** A script containing `while True:` will completely freeze the Web Worker execution thread.
   - _Mitigation:_ We implement a "Stop Script" button on the UI, which calls `.terminate()` on the worker thread, instantiates a clean worker instance, and restores the cached global workspace.
3. **Overhead on State Sync:** Flooding the worker with per-sample audio stream data could easily block the main thread.
   - _Mitigation:_ Only parameters and essential features are cached actively. FFT spectrum arrays are transferred in-place using standard JS `Float32Array` or `Uint8Array` cloning, throttled to 50Hz, and only active when scripts are running.
