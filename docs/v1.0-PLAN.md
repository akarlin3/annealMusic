# Anneal Ambiance v1.0 — Plan (Checkpoint 0)

The v1.0 ship: physical-modeling engine, recording export, embed route. This
plan is grounded in the actual v0.9 codebase (sine/FM/granular engines, the
`AnnealEngine` interface, the Zustand store, the v5 share schema, the FastAPI
backend with the v0.7 `recordings` schema, and the v0.8 Playwright preview
renderer).

> The user directed: **build the entire v1.0 in one pass, no checkpoint gates.**
> This document therefore doubles as the design record; the implementation lands
> across the same commit series the checkpoints describe.

## Section A — Physical modeling engine

### A.1 Sub-model technical specs

All three are **continuously excited** (filtered-noise injection, not an
impulsive pluck/strike) so they sustain for ambient use. Shared excitation lives
in one utility (`physical-dsp/noise.ts`) — no copies across models.

- **String** (`KarplusStrong`): a delay line of length `sampleRate / f0` with a
  one-pole low-pass in the feedback path. `damping` → feedback gain (Q of the
  string); `brightness` → loop low-pass cutoff. Continuous excitation injects
  filtered noise into the delay each sample (the extension that turns a plucked
  string into a bowed/aeolian sustain). Ref: Karplus & Strong 1983; Jaffe &
  Smith 1983 (extensions).
- **Tube** (`Waveguide`): a cylindrical bidirectional delay (two rails) with a
  reflection at the closed end and a simple memoryless reed model at the mouth:
  `reed(p) = clamp(1 - k·Δp)` driving the pressure difference, continuously fed
  by breath-noise (excitation). `damping` → bore loss filter; reed stiffness `k`
  is a sub-model param. Ref: Julius O. Smith, _Physical Audio Signal Processing_
  (CCRMA), "Digital Waveguide Models — woodwinds/clarinet".
- **Plate** (`ModalBank`): ~20 modal resonators (biquad bandpass) per partial,
  tuned to a (slightly inharmonic) plate eigenfrequency series
  `f0·sqrt(1 + B·n²)`. Continuous noise excitation feeds all modes. `damping` →
  mode Q (decay), `brightness` → per-mode gain rolloff. Ref: modal synthesis /
  Chaigne & Lambourg plate models (eigenfrequency distribution only).

### A.2 AudioWorklet design

- **One processor type per sub-model** (`string-processor`, `tube-processor`,
  `plate-processor`) for clarity, each a thin `AudioWorkletProcessor` wrapper
  around the pure DSP class. The DSP classes are plain TypeScript (no worklet
  globals), so they are the single source of truth and are unit-tested directly
  in jsdom; the worklet wrapper only marshals params and pulls samples.
- **Param transport**: `excitationLevel`, `damping`, `brightness`, and per-partial
  `detuneCents` are `AudioWorkletNode` **AudioParams** (k-rate) for smooth,
  sample-accurate updates from the drift loop. Structural changes (model swap is
  a full engine rebuild; `f0` base, reed stiffness, plate inharmonicity) go via
  `port.postMessage`.
- **Module URLs + registration timing**: Vite has no native AudioWorklet
  bundling (`new URL('./x.ts', import.meta.url)` inlines the raw, untranspiled
  source as a `data:` URL — broken at runtime). So a dedicated build
  (`vite.worklet.config.ts`) bundles the three processors + shared DSP into one
  self-contained classic IIFE at `public/worklets/physical.js` (~2 KB gz). A
  single `ctx.audioWorklet.addModule('/worklets/physical.js')` registers all
  three processors, **lazily on first physical-engine select**, memoized per
  `AudioContext`. Classic script (no imports) ⇒ loads in every browser, and the
  shared noise utility is bundled exactly once.
- **Worklet globals typing**: a single ambient declaration
  (`physical-worklets/worklet-globals.d.ts`) declares `AudioWorkletProcessor`,
  `registerProcessor`, and `sampleRate`. lib.dom does not ship these, so there is
  no conflict; a separate tsconfig proved unnecessary.

### A.3 Per-partial topology

`HARMONICS.slice(0, density)` → one worklet node per partial (≤ 8). Each node's
`f0` AudioParam is `rootFreq · ratio^spread`; the per-partial amplitude uses the
shared `partialShape(i)` baseline (no LFO node — the model's own motion + drift
provide movement). All partial nodes fan into a single engine `GainNode`
(`getOutputNode()`), routed by the orchestrator into the post-fx filter exactly
like the other engines. Node creation goes through an **injectable factory**
(`WorkletNodeFactory`) — the default builds real `AudioWorkletNode`s; tests
inject a mock, so the engine's lifecycle/detune/mapping is unit-tested without a
real worklet (mirrors granular's injectable `SourceLoader`).

### A.4 CPU budget and fallback

Worst case: plate × 20 modes × 8 partials = 160 biquads, plus excitation. The
fallback is **refuse the engine swap with a toast** (the prompt's option, and the
honest one — silently dropping modes changes the timbre). Detection: if
`audioWorklet.addModule` rejects, or `AudioWorkletNode` construction throws, or
the platform reports no `audioWorklet`, `PhysicalEngine.start` throws a typed
`PhysicalUnsupportedError`; the orchestrator catches it during `crossfadeTo`,
keeps the current engine, and surfaces the error so React shows
"Physical engine unavailable on this device." Secondary guard: plate mode count
is a constant (`PLATE_MODES = 20`) that the build can lower for low-`deviceMemory`
hints, logged not silent. AudioWorklet errors are never swallowed.

### A.5 Param mapping

| Param             | Range             | Effect                                       |
| ----------------- | ----------------- | -------------------------------------------- |
| `model`           | string/tube/plate | selects sub-model (engine rebuild on change) |
| `excitationLevel` | 0–1               | noise-injection amplitude (drive)            |
| `damping`         | 0–1               | feedback gain / mode Q (↑ = shorter decay)   |
| `brightness`      | 0–1               | excitation + loop filter cutoff              |
| `reed`            | 0–1               | (tube) reed stiffness `k`                    |
| `inharm`          | 0–1               | (plate) eigenfrequency inharmonicity `B`     |

`model` is encoded as an index (0/1/2) so it rides the existing numeric
engine-param bag and URL codec unchanged. Sub-model params that don't apply are
simply ignored by the active processor.

## Section B — Recording export

### B.1 Client vs server for offline render

**Client-side `OfflineAudioContext`** (the recommendation). No server cost, and
the patch DSP is the same code. **Fallback chain**: (1) client OfflineAudioContext;
(2) if the device lacks AudioWorklet (physical patches) or OfflineAudioContext,
fall back to the v0.8 **server render** path (Playwright/Chromium, which supports
worklets natively) by POSTing a render request; (3) if neither, realtime capture
only, with a toast. v1.0 implements (1) + the realtime path fully; (2) reuses the
existing render queue and is wired as a documented fallback.

### B.2 Realtime capture audio graph

- **Tap point**: `masterVol` output (post-fx, post-volume, pre-`analyser`) — this
  is exactly "what the user heard", and includes input + loops since they all sum
  through the post-fx filter. A dedicated tap `GainNode` (gain 1) hangs off
  `masterVol` so capture never perturbs the speaker path.
- **Opus path**: tap → `MediaStreamAudioDestinationNode` → `MediaRecorder`
  (`audio/webm;codecs=opus`, transcoded server-side to Ogg/Opus like previews).
- **WAV path**: tap → reuse the `loop/capture.ts` AudioWorklet PCM tap to
  accumulate Float32 chunks, finalized with the existing `encodeWav` on stop.
  (Worklet PCM, not `MediaRecorder`, because WAV must be lossless.)

### B.3 Recording flow UX

- **Record button** in the header, between Save Patch and the Begin/Settle
  button. Only meaningful while a session/input is producing sound.
- **Indicator**: a pulsing dot + mono `mm:ss` counter replacing the button label
  while recording.
- **Max length**: 60 min hard cap; warning toast at 50 min; auto-stop at 60.
- **On stop**: `RecordingDialog` asks title + format (Opus default / WAV) +
  visibility, shows a size estimate, then uploads.

### B.4 Recording management

**My Recordings** as a sibling drawer to My Patches (its own button), reusing the
drawer shell. Per recording: inline `<audio>` play, download, delete, copy
`/r/<id>` share link.

### B.5 Quota

Confirmed already enforced server-side (v0.7): `quota_recordings = 10` per anon
user and `quota_bytes = 1 GiB` total (`api/app/config.py`, enforced in
`routers/recordings.py`). v1.0 adds the **upload pipeline** (multipart PUT of the
encoded blob → storage → metadata row) which the v0.7 endpoint stubbed.

## Section C — Embed route

### C.1 Route

`/embed/:slugOrId` (client route) backed by a server shell. The embed only ever
**streams the v0.8 preview audio**; it runs no engine/DSP, so it is independent of
the synth code.

### C.2 Bundle splitting

A second Vite entry (`embed.html` → `src/embed/embedEntry.ts` → `EmbedApp.tsx`).
It imports only React + a tiny preview-fetch client + the player — no orchestrator,
engines, granular, or physical. Target **< 50 KB JS+CSS gzipped**, enforced in CI.

### C.3 Theming

`?theme=dark|light` (default dark). Dark: bg `#0c0a09`, text `#f5f5f4`, accent
`#f59e0b`. Light: bg `#faf9f7`, text `#1c1917`, accent `#b45309`.

### C.4 iframe security headers

- Embed route (`/embed/*` and its API): `Content-Security-Policy: frame-ancestors *`
  and **no** `X-Frame-Options` (the deny header blocks all framing and has no
  allow-list equivalent that modern browsers honor, so it is dropped for this
  route only).
- All other routes: keep `X-Frame-Options: DENY` (existing middleware default).
- Implemented by making the security middleware route-aware (prefix check).

### C.5 Public surface

A "Get embed code" affordance: the gallery card "⋯" menu gains an item, and My
Patches cards (public only) get the same. Opens a dialog with copy-pastable
`<iframe>` HTML and width/height presets.

## Cross-cutting

### Module plan (as built)

```
src/audio/engines/
  physical.ts                     // PhysicalEngine : AnnealEngine
  physical-dsp/
    noise.ts                      // shared continuous-excitation generator
    string.ts  tube.ts  plate.ts  // pure DSP classes (unit-tested)
  physical-worklets/
    worklet-globals.d.ts
    string-processor.ts  tube-processor.ts  plate-processor.ts
src/record/
  RealtimeRecorder.ts             // Opus (MediaRecorder) + WAV (PCM tap)
  OfflineRenderer.ts              // client OfflineAudioContext path
  RecordingDialog.tsx
  MyRecordings.tsx
src/embed/
  EmbedApp.tsx  embedEntry.ts
embed.html                        // second Vite entry
```

### URL schema v6

`SCHEMA_VERSION = 6`; supported `[1..6]`. Adds `e=physical` + the `ph.*`
namespace (`ph.model`, `ph.excitationLevel`, `ph.damping`, `ph.brightness`,
`ph.reed`, `ph.inharm`). Recording short links are a separate concern
(`/r/<id>`), not patch state; an optional `?r=<recording_id>` query is reserved.
The decoder stays lenient + versioned exactly as today.

### Server-side preview rendering for physical

The v0.8 renderer is **Option B (Playwright/Chromium)**, which supports module
AudioWorklets natively — physical patches render server-side with no polyfill.
The headless harness already builds the real `Orchestrator`, so it picks up the
new engine for free; the only requirement is that the worklet module URLs resolve
from the deployed origin (they do — same bundle).

### Sentry

A v0.7 stub exists (`api/app/sentry.py`, no-op unless `SENTRY_DSN`). Left as-is;
wiring real DSN config is deploy-time, not code, so it stays deferred.

### Risks & tradeoffs

- **Physical CPU**: 160 biquads is the ceiling; mitigated by refuse-with-toast
  fallback and a tunable mode count. Biggest unknown on old mobile.
- **Embed bundle budget**: keeping the embed free of the audio core is essential;
  CI gate enforces < 50 KB so regressions fail loudly.
- **Offline render client-vs-server**: client is free but slower and worklet-
  dependent; server is the documented fallback reusing v0.8 infra.
