# Implementation Plan тАФ v4.5 Session History + Curated Library

This plan details the architecture for **v4.5**, two compressed deliverables in one slice:

1. **Per-user Session History** тАФ a private, calm-by-design record of every Listening
   Session play (timestamp, actual duration listened, optional reflection). No streaks,
   no points, no nudges. A record the user can revisit, nothing more.
2. **Curated Listening Library** (`/listen`) тАФ an editorial catalog of Listening
   Sessions, organized by length, intention, and audio character, with a server-rendered
   audio preview per card. The real meditation entry point, kept distinct from the
   creator-side `/gallery` discovery surface.

> **Branch:** `feat/v4.5-history-library` (development on `claude/busy-thompson-GgyFw`)
> **Prereq:** v4.4 shipped (breath pacing visuals live, schema v20).

## Open Questions

None blocking. Three product calls, defaulted sensibly and noted inline:

1. **Library preview source.** Listening Sessions have no preview columns of their own;
   they reference a `piece_id` *or* `patch_id`, both of which already carry the v0.8
   `preview_*` columns and `/preview` endpoints. The library therefore **derives** a
   listing's preview from its source artifact rather than duplicating preview state.
   The CP3 "sweep job" verifies the *source* artifact has a ready preview before a
   listing is surfaced (┬з7, ┬зCP1). Refinable in CP1 once endpoints are wired.
2. **Editorial taxonomy values.** Intentions and character tags are seeded as a fixed
   vocabulary in shared constants (┬з4) so the admin UI, API validation, and `/listen`
   filters agree. Adding a value later is a one-line constant change, no migration.
3. **In-progress play handling.** `POST /me/sessions` on session *start* creates a row
   with `completed_at = NULL`; the same play is finalized via the returned id on
   completion/end (┬з1.1). A play abandoned without a finalize call still has an honest
   `duration_listened_ms` from the start write (defaults to 0; see ┬з1.1).

## 1. Data Model

Two new tables. Final DDL (refined from the prompt sketch тАФ GUID/JSON portable types to
match the existing SQLite-test / Postgres-prod split used elsewhere in `models.py`).

### 1.1 `session_plays`

```sql
CREATE TABLE session_plays (
  id                       UUID PRIMARY KEY,
  user_id                  UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  listening_session_id     UUID NOT NULL REFERENCES listening_sessions(id) ON DELETE CASCADE,
  started_at               TIMESTAMPTZ NOT NULL,
  completed_at             TIMESTAMPTZ,
  duration_listened_ms     INTEGER NOT NULL DEFAULT 0,
  reflection               TEXT,
  created_at               TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_session_plays_user ON session_plays(user_id, started_at DESC);
```

- **Per-user, requires account.** `user_id` is the `users.id` (the per-device anon user).
  History is meaningful only for signed-in accounts, which span devices via the v1.3
  account тЖТ users mapping. Reads aggregate across `identity.owned_anon_ids` so history is
  cross-device for authed users. Anonymous plays are **not** logged; the client shows a
  gentle "sign in to keep your history" nudge post-session (┬з2, no write occurs).
- **Lifecycle.** Start тЖТ insert (`started_at = now`, `completed_at = NULL`,
  `duration_listened_ms = 0`). Completion/end тЖТ `PATCH`-style finalize sets
  `completed_at` and the *actual* `duration_listened_ms` (may be < full duration).
- **Reflection.** Optional, `тЙд 500 chars` (enforced in schema validation, ┬з2). Editable
  and deletable after the fact.
- **Not stored:** sculpt actions, audio recordings, biometrics, device fingerprints,
  IP тАФ nothing beyond the four columns above. Documented as a hard boundary in ┬з8.
- `created_at` is the audit row-creation time; `started_at` is the play's clock time
  (the two differ only by request latency but are kept distinct for honesty).

### 1.2 `library_listings`

```sql
CREATE TABLE library_listings (
  id                       UUID PRIMARY KEY,
  listening_session_id     UUID NOT NULL REFERENCES listening_sessions(id) ON DELETE CASCADE,
  intention                TEXT,
  length_category          TEXT,
  character_tags           TEXT[] NOT NULL DEFAULT '{}',
  editor_pick              BOOLEAN NOT NULL DEFAULT FALSE,
  editor_pick_at           TIMESTAMPTZ,
  curator_note             TEXT,
  added_at                 TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at              TIMESTAMPTZ
);
CREATE INDEX idx_library_active ON library_listings(intention, length_category) WHERE archived_at IS NULL;
CREATE INDEX idx_library_picks  ON library_listings(editor_pick_at DESC) WHERE editor_pick = TRUE AND archived_at IS NULL;
```

- `character_tags` is an array. In Postgres it's `TEXT[]`; in the SQLite test DB it maps to
  the existing portable `JSONType()` list column (same approach as `bell_schedule`).
  Partial indexes are Postgres-only and guarded in the migration (`if dialect == postgresql`).
- `length_category` may be **denormalized** from the source listening session's existing
  `length_category` field at add-time, but is editable per-listing so the curator can
  override (the editorial length bucket need not match the technical duration exactly).
- **Editorial-only.** Rows are created *only* via the admin endpoints (┬зAPI). A listing
  pointing at an archived/removed source session is treated as inactive.
- `archived_at` is a soft-delete; archived listings never appear in `/listen` but are
  retained for the picks-history view.

> Both tables added in one Alembic revision `0015_v4_5_history_library` (head = `0014`).

## 2. API Surface

All under the existing FastAPI app, reusing `get_identity` / `CurrentUser` / `require_admin`.

### History (`/api/v1/me/sessions`) тАФ requires a signed-in account

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/me/sessions` | Log a play on **start**. Body: `{ listening_session_id, started_at }`. Returns `{ id }`. |
| `PATCH` | `/me/sessions/:id` | Finalize and/or edit: `{ completed_at?, duration_listened_ms?, reflection? }`. |
| `GET` | `/me/sessions` | List, newest first, paginated (`?limit&cursor`). Joins source session title/thumbnail. |
| `GET` | `/me/sessions/stats` | Minimal stats (see ┬з3, single computation site). |
| `DELETE` | `/me/sessions/:id` | Forget a session (hard delete). |

- **Auth gate.** These routes require `identity.account_id` (a real account). A request
  without one returns `401`/a gentle structured error the client renders as the sign-in
  nudge тАФ it never mints history for anon users.
- **Ownership.** Every `:id` op verifies the play's `user_id тИИ identity.owned_anon_ids`.
- `reflection` validated `тЙд 500` chars; trimmed; empty тЖТ stored `NULL`.
- The prompt listed `POST` as "log or update"; we split start (`POST`) from finalize
  (`PATCH`) so the in-progress row has a stable id and we never need a guess-the-row
  upsert. This keeps the single-writer story clean.

### Library (public reads)

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/library` | `?intention&length&character&picks=only`. Active listings only, joined to source session + derived preview. |
| `GET` | `/library/picks` | Current editor picks, `editor_pick_at DESC`. |

### Library (admin тАФ `x-admin-key`)

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/admin/library` | Add a listing: `{ listening_session_id, intention, length_category, character_tags, curator_note? }`. Triggers preview sweep (┬з7). |
| `PATCH` | `/admin/library/:id` | Update tags / `editor_pick` / note / reorder. Setting `editor_pick=true` stamps `editor_pick_at`. |
| `DELETE` | `/admin/library/:id` | Archive (soft): sets `archived_at`. |

## 3. Stats тАФ single computation site (heuristic-drift rule)

All stats live in **one** function (`api/app/routers/me_sessions.py::compute_stats`),
consumed only by `GET /me/sessions/stats`. The client never recomputes; it renders the
server numbers verbatim. Returned shape:

```jsonc
{
  "total_sessions": 12,
  "total_listened_ms": 15480000,     // тЖТ "4.3 hours"
  "average_length_ms": 1290000,      // тЖТ "21 min"
  "this_month": { "sessions": 12, "total_listened_ms": 15480000 }
}
```

**Deliberately understated framing.** The UI renders exactly: *"12 sessions, 4.3 hours
total this month."* and *"average 21 min."* No comparison to prior periods, no "best
week," no rank, no streak тАФ by design. The stats object intentionally **omits** any field
that could seed an engagement loop (consecutive-days, longest-streak, daily-goal-progress).
This omission is asserted in a test (┬зCP1) and the calm review (┬з5).

## 4. Editorial Taxonomy

Fixed vocabularies in `src/library/taxonomy.ts` (mirrored as a Python constant for API
validation тАФ single source mirrored, drift-guarded by a small test).

- **Length categories** (reuse existing listening-session `length_category`):
  `short` (тЙд10 min) ┬╖ `medium` (10тАУ25) ┬╖ `long` (25тАУ45) ┬╖ `extended` (45+).
- **Intentions:** `morning` ┬╖ `evening` ┬╖ `sleep` ┬╖ `difficult_day` ┬╖ `focus` ┬╖
  `open_practice` ┬╖ `closing_the_week`.
- **Character tags** (multi):
  `drone` ┬╖ `composed` (Pieces) ┬╖ `spoken_word_free` (all тАФ informational badge) ┬╖
  `with_bells` ┬╖ `with_tunings`.

Each has a calm display label and short description. No tag implies ranking or difficulty.

## 5. Calm-by-Design Checklist (v4.5)

`docs/CALM_BY_DESIGN.md` is **created** in this slice (it does not exist yet) with the
running review framework, and a v4.5 section appended each checkpoint. The v4.5 checklist:

- [ ] No streak counter anywhere (history, library, admin).
- [ ] No points, badges, achievements, levels, XP.
- [ ] No daily/weekly goal, no "X days in a row," no "you missed yesterday."
- [ ] No notifications: no email, no push, no in-app reminder banner. None ship; none
      are stubbed; the codebase contains no scheduler that could send one.
- [ ] Stats framing is descriptive, not motivational (┬з3) тАФ no comparatives, no rank.
- [ ] History is private: no public route, no share button, no "share my stats."
- [ ] Reflections are private and unshareable.
- [ ] Anonymous post-session nudge is a single gentle prompt, dismissible, non-recurring.
- [ ] **CI lexical gate:** a test scans v4.5 UI strings for banned tokens
      (`streak`, `level up`, `achievement`, `daily goal`, `badge`, `don't break`,
      `keep it up`, `missed`) and **fails** review on a match. Lives in
      `src/test/calm-by-design.test.ts`, run in CI.

This lexical gate is the hard CI item called out in the rules of engagement.

## 6. UI Design

### 6.1 History UI (`/me/sessions`)

- New route in `main.tsx`; entry from the account menu in `App.tsx` header.

- New route, separate from `/gallery`.
- `LibraryFilters`: three independent browse axes тАФ **length**, **intention**,

- Filters compose via query params on `GET /library`. No infinite-scroll engagement
  pattern тАФ simple, finite, paginated grid.

## 7. Server-side Preview Rendering

- Listening sessions reuse the v0.8 pipeline **through their source artifact**: each
  listing's preview is the `piece`/`patch` `/preview` (already `preview_status`-tracked,
  rendered, R2-stored). No new preview columns on `listening_sessions`.
- **Sweep on add:** `POST /admin/library` enqueues a check that the source artifact has a
  `ready` preview; if `none`/`failed`, it kicks the existing render job and the listing is
  flagged `preview_pending` in the API response until ready. `/listen` shows a calm
  "preview rendering" placeholder rather than a broken control. A small idempotent sweep
  (`api/scripts/sweep_library_previews.py`, reusing the orphan-sweep harness pattern)
  can re-run to backfill.
- CP1 verifies the derive-vs-duplicate decision against the real preview endpoints; if a
  listing's source has no previewable artifact, the add is rejected with a clear error.

## 8. Privacy Implications

- **Sensitivity.** Practice frequency and timing are personal data revealing routine and
  potentially mental-health context. Treated as sensitive throughout.
- **Strictly private.** No public route, no profile display, no share. Not included in
  `/u/:account_id` profiles, not in `/gallery`, not in any export.
- **Minimization.** Only the four documented columns are stored (┬з1.1). No biometrics, no
  audio, no sculpt logs, no analytics on practice content.
- **Retention + deletion.** User can `DELETE` any play (forget). Account deletion cascades
  (`ON DELETE CASCADE`) and removes all history. Documented in `docs/HISTORY.md` with the
  retention/deletion policy and the explicit "we never email/notify you about practice"
  promise.
- **No third-party sharing.** Apple Health / Google Health Connect is explicitly **v4.6**,
  not v4.5 тАФ no health-platform write path exists in this slice.

## 9. Admin Curation Flow

`src/admin/LibraryCuration.tsx`, a new section in the existing `/admin` panel (gated by
the v0.8 `x-admin-key`, lazy-loaded so it stays out of mobile builds like `AdminPage`):

1. **Add to library** тАФ pick a listening session (by id/slug), set intention,
   length category, character tags, optional curator note. Submit тЖТ `POST /admin/library`
   тЖТ preview sweep.
2. **Manage editor's recent picks** тАФ toggle `editor_pick`, view current set + history
   (picks ordered by `editor_pick_at`).
3. **Reorder / archive / remove** тАФ `PATCH` for order & pick state, `DELETE` to archive.

Mirrors the existing `FeaturedPick` admin ergonomics (gallery) so the curator UX is familiar.

## 11. Risks

- **Feature creep toward engagement loops.** The single biggest risk. Mitigated by the
  calm checklist (┬з5), the stats single-site with intentionally omitted fields (┬з3), and
  the CI lexical gate. Any future "continue your practice" / per-session-goal idea is
  **deferred** and must pass a calm review before design.
- **Library quality if editorial throughput is slow.** Editorial-only is a deliberate
  quality gate; if it becomes a bottleneck, community contributions are a flagged
  **v4.5.1** follow-up тАФ not pulled into v4.5.
- **Preview derivation edge cases.** A source artifact with no preview (or a failed one)
  must not surface a broken card; handled by the pending-state placeholder + sweep (┬з7),
  and add-time rejection when no previewable source exists.
- **Cross-device history correctness.** Reads must aggregate over `owned_anon_ids`;
  covered by a CP1 test that logs plays on two anon ids under one account and asserts a
  unified list.
