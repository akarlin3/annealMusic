# AnnealMusic v1.1 — Plan (Checkpoint 0)

The largest slice since v1.0: **five new physical sub-models** (`membrane`,
`bowed`, `mallet`, `edge`, `bell`), the **Additive** engine (6th top-level), and
the **Wavetable** engine (7th top-level). Engine count 4 → 6; physical sub-model
count 3 → 8.

This plan is grounded in the actual v1.0 codebase: the `AnnealEngine` interface
(`src/audio/engines/types.ts`), the four shipped engines, the pure-TS physical
DSP + worklet split (`physical-dsp/` + `physical-worklets/`, bundled to
`public/worklets/physical.js` by `vite.worklet.config.ts`), the Zustand store
(`src/state/params.ts`), the v6 share schema (`src/share/`), the schema manifest
generator (`schema/gen-manifest.ts`), and the Playwright preview renderer.

> **Diagnosis-first.** Where the mega-prompt's wording conflicts with codebase
> reality, this plan flags the conflict and proposes the codebase-consistent
> path rather than bending the implementation. Two such conflicts are resolved
> below (worklet lazy-loading granularity §B.5; physical sub-model params reusing
> the pre-provisioned generic slots §B.4).
>
> **This is a HARD STOP.** Nothing in CP1–CP4 is implemented until explicit "go."

## Section A — Engine-interface audit

### A.1 Stress test of `AnnealEngine`

The interface (`types.ts`) is: `start / stop / getOutputNode / setSharedParams /
setEngineParams / setPartialDetune / getPartialCount / getPartialFrequencies`
plus optional `setErrorHandler`, and `capabilities { densityLockedWhilePlaying,
params, crossfadeMs? }`. The orchestrator owns the context, post-fx, drift loop,
and crossfade. I walked both new engines and the five sub-models against it.

**Additive** (16 native `OscillatorNode`s): fits cleanly. It is structurally the
sine engine with a richer per-partial gain/freq map. `setPartialDetune`,
`getPartialFrequencies`, `getPartialCount` map 1:1 to sine's implementation.
**One real gap:** it wants **16** partials, but two things cap the system at 8
(see A.2/A.3). No method-shape change needed.

**Wavetable** (per-partial dual/quad `OscillatorNode` + lazy JSON asset load):
fits, reusing two patterns already in the tree —
1. **Async asset load before nodes attach.** Granular already loads its source
   buffer lazily and the physical engine already attaches worklet nodes *after*
   an async `register()` resolves, under cover of the orchestrator's bus fade,
   while exposing partial metadata synchronously. Wavetable's JSON fetch + decode
   uses the identical "metadata sync, nodes async" shape. **No interface change.**
2. **Glitch-free internal rebuild on a structural param change.** The physical
   engine already rebuilds all voice nodes when `ph.model` changes
   (`teardownVoices()` → `build()`), inside the live engine, with the
   orchestrator's bus masking the seam. Wavetable's table/morphTarget swap reuses
   exactly this in-engine rebuild pattern. **No interface change.**

**Temptations to bypass the interface, and the ruling on each:**

| Temptation | Ruling |
| --- | --- |
| Additive reads density past 8 by going around the shared `density` param | **No.** Extend the lattice + add a `maxPartials` capability (A.2). Density stays the single shared partial-count source. |
| Wavetable fetches its JSON bank itself with a hard import path | **No.** Injectable loader like granular's `loadSourceByIndex` so it is unit-testable with a mock; lazy + memoized per table id. |
| Wavetable adds a `rebuild()` to the public interface | **No.** Rebuild is private engine state, same as physical's model swap. |
| Either engine needs a synchronous "partials are ready" signal | **No.** `getPartialCount/Frequencies` already return synchronously from metadata; audio attaches later. The orchestrator never assumes nodes exist at `start` return (physical proved this). |

**Verdict:** the interface absorbs both engines with **exactly one additive
change** — `capabilities.maxPartials?` (A.2). No method signatures change. This
is the same "additive-only" outcome v1.0 achieved with `setErrorHandler`.

### A.2 Capability extension — `maxPartials`

**Two facts cap partials at 8 today:**

1. `HARMONICS` in `src/types/audio.ts` has **8 ratios**: `[1,1.5,2,2.5,3,4,5,6]`.
   Every engine does `HARMONICS.slice(0, density)`. There is no 9th–16th ratio.
2. `density` is a **shared** param (`AnnealMusicParams.density`), bounded `2..8`,
   `step 1`, `lockWhilePlaying` in `CONTROL_DEFS`. `clampParam` clamps to `[2,8]`
   globally. The ControlPanel slider reads `def.max` (= 8).

**Decision — adopt path (a) from the prompt: declare the ceiling via
capabilities; UI reads it.** Add to `AnnealEngineCapabilities`:

```ts
/** Max partials this engine supports. Omitted ⇒ DEFAULT_MAX_PARTIALS (8). */
readonly maxPartials?: number;
```

Concrete changes this drives:

- **Lattice extension.** Extend `HARMONICS` from 8 → 16 ratios (append a
  musically sensible continuation: `7, 8, 9, 10, 12, 14, 16, 24` — a thinning
  upper harmonic series; final values tuned in CP2). Engines that cap at 8 simply
  never slice past index 8, so **sine/fm/granular/physical are unaffected** (they
  declare `maxPartials: 8` or omit it).
- **Shared `density` bound raised to 16**, but each engine clamps internally:
  `effectiveDensity = min(shared.density, caps.maxPartials ?? 8)`. This keeps
  **one** shared partial-count param (no per-engine density param — that would
  fork the store and the schema) while honoring per-engine ceilings.
- **UI slider max is per-engine.** `ControlPanel` density row computes
  `sliderMax = engineCapabilities(engineId).maxPartials ?? def.max`. Selecting
  Additive raises the slider to 16; switching away re-clamps the displayed value
  via the engine's internal `min(...)` (the stored value can stay 16 — non-additive
  engines just ignore the excess, same as a value of 8 today).
- **`getPartialCount()` returns the effective (clamped) count**, so the drift
  loop and visualizer see the truth.

**Per-engine `maxPartials` table:**

| Engine | maxPartials | Rationale |
| --- | --- | --- |
| sine | 8 | unchanged |
| fm | 8 | unchanged |
| granular | 8 | lattice partials; grain density is its own `gr.density` |
| physical | 8 | CPU (modal banks); unchanged |
| **additive** | **16** | the "sine but richer" spec |
| **wavetable** | **8** | 2–4 oscillators/partial; 8 keeps node count sane (D.3) |

Justification for path (a) over hard-coding per-engine UI: the schema generator,
the validator, and any future engine all read capabilities; hard-coding "if
additive then 16" in the React layer would be a third source of truth for a fact
the engine already owns. Capability-as-source-of-truth matches the v0.7 schema
principle.

### A.3 URL schema v7 spec

`SCHEMA_VERSION 6 → 7`. `SUPPORTED_SCHEMA_VERSIONS` gains `7` (v1–v6 still
decode). The v7 changes are **deliberately minimal** because v1.0 left room:

**Engine id space** (`EngineId` union, `ENGINES`, `ENGINE_ORDER`,
`ENGINE_LABELS`, `ENGINE_URL_NS`):

| Engine | `e=` selector | URL namespace |
| --- | --- | --- |
| additive | `e=additive` | `ad` |
| wavetable | `e=wavetable` | `wt` |

**`ad.*` keys** (additive engine params):

| Key | Min | Max | Default | Step | Notes |
| --- | --- | --- | --- | --- | --- |
| `ad.tilt` | −12 | +6 | −3 | 0.1 | dB/oct spectral rolloff |
| `ad.evenOdd` | −1 | +1 | 0 | 0.01 | −1 odd-only … +1 even-emphasized |
| `ad.inharmonic` | −0.05 | +0.05 | 0 | 0.001 | stretch exponent (C.2) |
| `ad.phase` | 0 | 2 | 0 | 1 | enum: 0 random, 1 aligned, 2 inverted |

(`ad.phase` is an enum encoded as an int param so it rides the existing numeric
`EngineParams` bag — same trick as `ph.model` and `gr.source`. The UI renders a
dropdown; `fmt` maps `0/1/2 → random/aligned/inverted`.)

**`wt.*` keys** (wavetable engine params):

| Key | Min | Max | Default | Step | Notes |
| --- | --- | --- | --- | --- | --- |
| `wt.table` | 0 | 7 | 0 | 1 | active table id (index into the 8-table bank) |
| `wt.position` | 0 | 1 | 0 | 0.01 | frame position; drifts autonomously |
| `wt.morphTarget` | −1 | 7 | −1 | 1 | second table id; `−1` = no morph |
| `wt.morph` | 0 | 1 | 0 | 0.01 | blend `table`→`morphTarget` |
| `wt.tilt` | −12 | +6 | 0 | 0.1 | per-partial gain tilt (additive's form) |

(`wt.morphTarget = −1` encodes "null"; the engine treats any negative value as
no-morph and skips the morph oscillator layer entirely — the CPU win in D.3.)

**`ph.model` enum widens** from `0..2` → `0..7`:
`0 string, 1 tube, 2 plate, 3 membrane, 4 bowed, 5 mallet, 6 edge, 7 bell`.
Because the `model` param's `max` is derived as `PHYSICAL_MODELS.length - 1`,
appending the five ids updates the bound automatically. **No new `ph.*` keys**
(see §B.4 — the two generic slots `ph.reed`/`ph.inharm` already exist in v6).

**Backward compatibility:** a v6 link with `ph.model=2&ph.inharm=0.7` loads
identically (still plate). `e=additive`/`e=wavetable` are simply absent from v6
links → default `sine`. Old links never carried `ad.`/`wt.` keys, so nothing to
migrate. Decode of `ad.`/`wt.` keys under `version < 7` is dropped-with-warning
exactly like the existing `version < N` guards in `decodeState`.

**Schema-as-source-of-truth:** after this section locks, `npm run gen:schema`
emits `schema/manifest.v7.json`; the Python validator (`api/app/validation.py`,
fed by the manifest) picks up the new engines/keys; the CI drift guard
(`api/tests/test_validation.py` + the manifest contract test) catches any miss.

## Section B — Physical sub-models

All five are **continuously excited** (no impulsive strike) — the v1.0 ambient
philosophy. Each is a thin `AudioWorkletProcessor` wrapping a pure-TS DSP class
(unit-tested directly in jsdom), bundled into the existing single worklet
artifact. Shared params `excitationLevel` (×gain×4 → exciter level), `damping`,
`brightness`, per-partial `detune` map exactly as in string/tube/plate so the
engine-level wiring in `PhysicalEngine.build()` / `setEngineParams()` needs no
per-model branches beyond model selection and the existing `plate`-style
structural `post({modes})`.

### B.1 membrane — 2D circular-membrane modal bank

- **DSP:** modal bank (reuses the generalized `ModalBank`, §B.6) tuned to the
  eigenfrequencies of an ideal circular membrane: `f_{m,n} = f0 · (j_{m,n} /
  j_{0,1})`, where `j_{m,n}` are the zeros of the Bessel function `J_m`. First
  ~12 modes from the standard table: ratios `1.000, 1.594, 2.136, 2.296, 2.653,
  2.918, 3.156, 3.501, 3.600, 3.652, 4.060, 4.154` (normalized to the
  fundamental `j_{0,1}`). Strongly inharmonic → drone-like, covers ground nothing
  else does.
- **Excitation:** continuous filtered noise into the bank (same `NoiseExciter`).
- **Shared-param mapping:** `excitationLevel` → exciter level; `damping` → mode Q
  (long ring at low damping); `brightness` → exciter cutoff + per-mode gain
  rolloff (identical to plate).
- **Sub-model params (2):** `tension` (rides `ph.reed` slot, §B.4) stretches the
  upper modes (multiplies `j_{m,n}` spacing, simulating skin tension, 0.5..2×);
  `modeStretch` (rides `ph.inharm`) blends membrane ratios toward harmonic for a
  tuned-drum feel.
- **CPU @ 8 partials:** 12 biquads × 8 = 96 biquads. Under the plate ceiling
  (160). ✅
- **Ref:** Cook, *Real Sound Synthesis for Interactive Applications* (modal
  membranes); Bessel-zero table (Abramowitz & Stegun).

### B.2 bowed — Karplus-Strong + bow friction (stick-slip)

- **DSP:** extends the `KarplusStrong` delay line with a bow-friction junction in
  the loop: the bow imparts energy via a nonlinear friction curve of the
  string-vs-bow relative velocity (stick-slip), rather than only injecting
  filtered noise. New class `BowedString` (subclass/composition over the existing
  delay-line core to avoid duplicating the loop — §B.5 anti-drift).
- **Excitation:** **continuous** bow pressure `kp` and velocity `vb` (a steady
  bow, not a stroke) — the stick-slip nonlinearity self-sustains the oscillation.
- **Shared-param mapping:** `excitationLevel` → bow force floor; `damping` →
  loop feedback (string decay); `brightness` → loop low-pass cutoff.
- **Sub-model params (2):** `bowPressure` (`ph.reed`) sets the friction-curve
  width (thin/glassy → broad/scratchy); `bowVelocity` (`ph.inharm`) sets `vb`
  (affects brightness + pitch-flattening of the slip cycle).
- **CPU @ 8 partials:** one delay line + a scalar friction lookup per sample ×
  8 ≈ cheaper than plate. ✅
- **Ref:** Smith, *Physical Audio Signal Processing* (CCRMA), "Bowed Strings /
  friction-driven oscillation"; McIntyre–Schumacher–Woodhouse.

### B.3 mallet — vibraphone/marimba modal bank

- **DSP:** generalized `ModalBank` tuned to ideal-bar eigenmodes. A free–free bar
  has inharmonic mode ratios `1, 2.756, 5.404, 8.933, …`; tuned percussion
  undercuts the bar to pull mode 2 to a near-integer (`1 : 4 : 10` for marimba,
  `1 : 4 : ~9.2` for vibraphone). Use `1, 3.984, 9.41, 15.6, …` (vibe-style),
  ~8 modes.
- **Excitation:** continuous **low-rate stochastic mallet "tremolo"** — a slow
  (1–8 Hz) amplitude-modulated noise burst train into the bank, so the bars
  shimmer as a sustained roll rather than discrete strikes.
- **Shared-param mapping:** `excitationLevel` → roll intensity; `damping` → bar Q
  (vibe bars ring long); `brightness` → exciter cutoff + mode rolloff.
- **Sub-model params (2):** `malletRate` (`ph.reed`) sets the tremolo rate;
  `materialHardness` (`ph.inharm`) sets exciter spectral hardness + mode-2 ratio
  (wood marimba ↔ metal vibe).
- **CPU @ 8 partials:** 8 biquads × 8 = 64 biquads + a slow LFO/noise gate. ✅
- **Ref:** Cook, *Real Sound Synthesis* (struck bars, tuned percussion);
  Fletcher & Rossing, *The Physics of Musical Instruments* (bar modes).

### B.4 edge — cylindrical waveguide with edge-tone (jet) excitation

- **DSP:** open–open cylindrical waveguide (two rails) with a **jet/edge-tone**
  excitation at the mouth instead of the tube's reed: an air jet with a
  delay-dependent deflection nonlinearity (the jet drive proportional to
  `sin`/cubic of the jet displacement) plus filtered-noise breathiness. New class
  `EdgeTone` (composition over the existing `Waveguide` rail core — §B.5; the
  reed junction is swapped for the jet junction, the closed end becomes open).
  Fills the air-column gap left by the reed-only `tube`.
- **Excitation:** continuous **jet velocity** + filtered-noise breath.
- **Shared-param mapping:** `excitationLevel` → jet drive; `damping` → bore loss;
  `brightness` → bore low-pass + breath cutoff.
- **Sub-model params (2):** `jetVelocity` (`ph.reed`) shifts the edge-tone regime
  (which harmonic the jet locks to → overblowing); `breathiness` (`ph.inharm`)
  sets the noise/tone balance.
- **CPU @ 8 partials:** two delay lines + a jet nonlinearity per sample × 8 ≈ tube
  cost. ✅
- **Ref:** Smith, *PASP* (CCRMA), "Flute / jet-drive waveguide"; Verge &
  Fabre flue-instrument models (jet deflection only — perceptual, not a full
  aeroacoustic solve).

### B.5 bell — bell-ratio modal bank

- **DSP:** generalized `ModalBank` tuned to church/handbell partial ratios:
  hum, prime, tierce (minor third), quint, nominal, and upper partials ≈
  `0.5, 1.0, 1.2, 1.5, 2.0, 2.5, 2.6, 3.0, 4.2, …`. The sub-octave hum + minor
  third is what makes a bell unmistakable and most distinct from plate.
- **Excitation:** continuous filtered noise into the bank (struck-bell *drone*).
- **Shared-param mapping:** identical to plate/membrane.
- **Sub-model params (2):** `inharmonicity` (`ph.inharm`) stretches/compresses
  the ratio set around the prime (warm bell ↔ clangorous); `warmth` (`ph.reed`)
  tilts mode gains toward the low partials (hum-forward) vs. the nominal.
- **CPU @ 8 partials:** ~10 biquads × 8 = 80 biquads. ✅
- **Ref:** Fletcher & Rossing, *The Physics of Musical Instruments* (bell partial
  structure); Cook, *Real Sound Synthesis* (modal banks).

### B.6 Worklet code structure & the modal-bank helper

- **Five new files** under `src/audio/engines/physical-worklets/`:
  `membrane-processor.ts`, `bowed-processor.ts`, `mallet-processor.ts`,
  `edge-processor.ts`, `bell-processor.ts` — each a thin wrapper exactly like
  `plate-processor.ts` (declare `parameterDescriptors`, marshal k-rate params into
  the DSP class, `render(out)`). `entry.ts` gains five `import` lines so the
  single bundle registers all 8 processors. `PROCESSOR_BY_MODEL` gains five rows.
- **Pure DSP files** under `src/audio/engines/physical-dsp/`:
  `membrane.ts`, `bowed.ts`, `mallet.ts`, `edge.ts`, `bell.ts` — each exporting a
  DSP class + its pure param-mapping functions (unit-tested in `dsp.test.ts`).
- **Modal-bank helper (anti-drift, the headline refactor).** `ModalBank` in
  `plate.ts` currently **hardcodes** the plate eigenfrequency formula in its
  private `eigenFreq(n)`. Membrane, mallet, and bell all want a modal bank with a
  *different* eigenfrequency source. Per the heuristic-drift rule (one
  implementation, not three copies), **generalize `ModalBank`**:
  - Move it to `physical-dsp/modal-bank.ts` (re-exported from `plate.ts` for
    back-compat), and inject the eigenfrequency source: a function
    `(n: number, baseFreq: number, stretch: number) => number` **or** a fixed
    ratio table `number[]`. Plate's `f0·sqrt(1+B·n²)` becomes `plateEigen`;
    membrane/mallet/bell pass `membraneRatios`/`malletRatios`/`bellRatios` tables
    (with their stretch params applied).
  - Keep the biquad design, Q/damping mapping, brightness gain rolloff, and
    `next()/render()` loop **unchanged and shared** — only the eigenfrequency
    input varies. Plate, membrane, mallet, bell are then four configurations of
    **one** `ModalBank`.
  - Bowed and edge do **not** use the modal bank; they generalize the existing
    `KarplusStrong` delay-line core and `Waveguide` rail core respectively
    (extract the shared loop primitives rather than copy-paste — same anti-drift
    rule applied to the waveguide/delay families).

### B.7 ⚠ Conflict flagged — worklet "lazy per sub-model"

The mega-prompt (CP1.3) says "worklet module loading is lazy **per sub-model**."
The codebase reality (`vite.worklet.config.ts`, `entry.ts`, `registerPhysicalModules`)
is **one** self-contained IIFE bundle registered by **one** `addModule`, memoized
per `AudioContext`, lazy on first *physical-engine* select. Splitting into eight
per-sub-model bundles would: multiply the build outputs, duplicate the shared DSP
across bundles (re-introducing the very drift the v1.0 single-bundle design
eliminated), and add eight `addModule` round-trips. **Recommendation: keep the
single-bundle, lazy-on-first-physical-select model.** All 8 processors register
together; switching sub-models is a pure in-engine rebuild (no new module load).
This is strictly better than per-sub-model lazy loading and preserves the
single-source-of-truth for shared DSP. Flagged here for explicit sign-off at the
gate.

## Section C — Additive engine

### C.1 Final param set

Per §A.3 `ad.*` table: `tilt` (−12..+6 dB/oct, def −3), `evenOdd` (−1..+1, def 0),
`inharmonic` (−0.05..+0.05, def 0), `phase` (enum 0/1/2, def 0=random).
`maxPartials: 16`, `densityLockedWhilePlaying: true`. No `crossfadeMs` override
(native nodes start instantly — default 600 ms is fine).

### C.2 Implementation sketch — `computePartialSpec`

Pure, table-testable: `computePartialSpec(params, partialIndex, root, spread,
shape) → { freq, gain, detune }`.

```
n        = partialIndex (0-based; partial number N = n + 1)
ratio    = HARMONICS[n] ^ spread                       // lattice + spread, as today
// inharmonic stretch: partial N freq = root · N^(1 + inharmonic)
stretch  = Math.pow(n + 1, 1 + inharmonic) / (n + 1)   // 1.0 when inharmonic = 0
freq     = root · ratio · stretch
// spectral tilt: dB/oct rolloff relative to the fundamental
octaves  = Math.log2(freq / root)
tiltGain = 10 ^ ((tilt · octaves) / 20)
// even/odd balance: N odd → boosted at evenOdd<0, N even → boosted at evenOdd>0
isOdd    = (n + 1) % 2 === 1
eoGain   = isOdd ? (1 - max(0, evenOdd)) : (1 - max(0, -evenOdd))   // equal-power variant in impl
gain     = baselineShape(n) · tiltGain · eoGain        // baselineShape = partialShape(n).baselineOffset
detune   = 0                                           // drift loop fills this
```

`phase` is not part of the per-partial spec (it's an oscillator-start concern):
`aligned` starts all oscillators at phase 0 (sharp transient on engine start),
`inverted` alternates 0/π, `random` randomizes — applied once at `start()` via
`OscillatorNode` start-time phase emulation (a short per-osc delay or a
`PeriodicWave` with a phase offset; `random` is the default diffuse field).

Engine structure mirrors `SineEngine`: per partial an `OscillatorNode(sine)` →
per-partial `GainNode` (driven by `computePartialSpec.gain` + the shared
baseline/LFO shape) → engine `out`. `setEngineParams` recomputes gains/freqs for
all live partials with `setTargetAtTime`. `setSharedParams(root/spread)` and
`setPartialDetune` map 1:1 to sine.

### C.3 Density extension to 16 (UI)

Path (a), already specified in §A.2: the density slider's `max` reads
`engineCapabilities(engineId).maxPartials ?? 8`. When Additive is active the
Pitch-group Density slider runs 2..16; its `lockWhilePlaying` behavior is
unchanged. No per-engine branch in React beyond reading the capability. Justified
in §A.2 (capability-as-source-of-truth; avoids a third source).

## Section D — Wavetable engine

### D.1 Wavetable file format

JSON per table in `public/wavetables/<id>.json`:

```jsonc
{
  "id": "evolving-harmonic",
  "name": "Evolving Harmonic",
  "frames": [ [c0, c1, …, cK], … ]   // frames[i] = Fourier magnitude coeffs for frame i
}
```

Fixed per bank: **32 frames × 64 coefficients** (the `real` array for
`createPeriodicWave`; `imag` all-zero ⇒ cosine-phase, deterministic spectra for
the FFT test in CP3). 32×64 floats ≈ 8 KB raw / ~3–5 KB as compact JSON per
table → bank well under the 500 KB budget. A small index
(`public/wavetables/index.json`) lists `[{id, name}]` in `wt.table` order.

### D.2 Bank composition (8 tables)

Synthesized offline by `scripts/build-wavetables.ts` (deterministic, checked in;
CI re-runs and diff-guards — same pattern as `gen:schema`). No third-party
audio assets ⇒ no `LICENSES.md` change.

1. **evolving-harmonic** — frame 0 sine → final frame full sawtooth (harmonics
   fade in across frames). The "hello world" morph.
2. **vocal-formant** — three drifting formant peaks (vowel a→e→i across frames).
3. **fm-sweep** — spectra of a 2-op FM tone with rising modulation index per
   frame (bright metallic sweep).
4. **organ-stack** — drawbar-style octave + fifth harmonic stacks, registration
   shifting per frame.
5. **square-pwm** — square wave with pulse-width sweeping 50%→5% (odd-harmonic
   emphasis morph).
6. **bell-partials** — inharmonic-ish coefficient set (within the harmonic
   periodic-wave constraint, approximated) for a glassy table.
7. **resonant-sweep** — a single resonant peak sweeping up the harmonic series
   per frame (wah-like).
8. **soft-saw-to-pulse** — gentle saw morphing to a narrow pulse; the "warm pad"
   default (`wt.table=0` could be this or evolving-harmonic; final pick in CP3).

### D.3 Per-partial dual-oscillator crossfade (audio graph for one partial)

Position-through-table needs interpolation between two **adjacent frames**;
`PeriodicWave` is immutable, so each frame is its own `setPeriodicWave`. Morph
between tables adds a second pair. **Worst case is 4 oscillators/partial** (frame
i, frame i+1 of `table`; frame i, frame i+1 of `morphTarget`). Per partial:

```
                 framePair(table):
  Osc(table,frame_i)   → gainA ┐
  Osc(table,frame_i+1) → gainB ┤→ tableGain(1-morph) ┐
                 framePair(morphTarget):              ├→ partialGain → engine out
  Osc(mt,frame_j)      → gainC ┐                      │   (shape · tilt)
  Osc(mt,frame_j+1)    → gainD ┤→ mtGain(morph) ──────┘
```

- `gainA/gainB` = equal-power crossfade on `frac = position·(frames−1) − i`.
- `tableGain/mtGain` = equal-power crossfade on `wt.morph`.
- All four oscillators share frequency (the partial's lattice freq) and the same
  `detune` from the drift loop.

**CPU / node budget.** Wavetable `maxPartials = 8` (not 16). With morph **off**
(`wt.morphTarget = −1`, the default), the morph pair is **not instantiated** → 2
osc/partial = **16 oscillators**. With morph **on** → 4 osc/partial = **32
oscillators**. 32 `OscillatorNode`s + ~24 gains is well within budget (sine runs
8 osc + LFOs; granular schedules ~120 grain voices). This confirms the prompt's
"verify against budget" concern: the 16-partial×4-osc = 64-node worst case **does
not occur** because wavetable caps at 8 partials and skips the morph layer when
unused. Documented for the CP4 stress matrix.

### D.4 Glitch-free table swap

Changing `wt.table` or `wt.morphTarget` mid-session rebuilds the oscillator set
(new `PeriodicWave`s). Reuse the physical engine's in-engine rebuild seam:

1. Lazy-load + cache the new table JSON (injectable loader; memoized per id).
2. Build the new oscillator/gain subgraph for every partial **at gain 0**,
   started but silent.
3. Equal-power crossfade old subgraph → new subgraph over ~60–120 ms (a private
   internal fade, distinct from the orchestrator's engine-level crossfade).
4. After the fade, stop + disconnect the old oscillators.

Because the load is async, `start`/swap exposes partial metadata synchronously
(freqs/count) and attaches audio when the JSON resolves — identical to physical's
`build()`-after-`register()`. Load failure surfaces via `setErrorHandler` (toast),
never silent (v1.0 pattern). `wt.position`/`wt.morph` changes are **not**
structural — they only move gains, no rebuild.

### D.5 Autonomous `wt.position` drift

Same trick as granular's `positionCenter` random-walk: a slow (~0.1/s) clamped
random walk on `position` on a ~100 ms tick, so static patches keep moving. The
walk is internal to the engine (not the global drift loop); it nudges the stored
`wt.position` and re-balances the frame crossfade gains.

## Section F — Risks

### F.1 CPU stress

Worst-case combos to test on a documented reference machine (CP4):

| Combo | Concern |
| --- | --- |
| Additive @ 16 partials + arc + input + 1 frozen loop slot | 16 osc + LFOs + grain loop + input filter |
| Wavetable @ 8 partials, morph ON + arc + input + 1 frozen loop | 32 osc + 2× crossfade + grain loop |
| Each new sub-model @ 8 partials + arc + input + 1 frozen loop | modal banks: membrane 96 / mallet 64 / bell 80 biquads; bowed/edge delay lines |

**Throttling strategy (already provisioned):** modal-bank processors accept a
structural `{ modes }` message (plate uses it today) to lower mode count under
pressure; the membrane/mallet/bell processors inherit this via the generalized
`ModalBank`. Additive/wavetable throttle by the user's density slider. Document
measured CPU%; if any combo exceeds budget on the reference machine, lower the
default mode counts (not the max) and note it.

### F.2 Server-side preview rendering (CP4 verification plan)

Headless Chromium (Playwright, `renderer_playwright.py` + `render.html` +
`src/render/headless.ts`) runs AudioWorklets natively, so the five new sub-models
"just work" (same single bundle). The two native-node engines need no worklet at
all. **The one thing to verify explicitly:** wavetable fetches JSON from
`public/wavetables/` — confirm the renderer's static-serving path includes that
directory (it already serves `public/worklets/`). CP4 publishes **one patch per
new sub-model (5) + one per new engine (2) = 7 previews** and asserts each renders
non-silent. Any failure is fixed at the renderer level, not the engine level.

### F.3 Tradeoffs

- **Single shared `density` + per-engine `maxPartials` clamp** vs. a per-engine
  density param: chosen the former (one store field, one schema key, one slider)
  at the cost of a `min(...)` clamp in each engine. The alternative forks state
  three ways.
- **Reusing `ph.reed`/`ph.inharm` as generic sub-model slots** vs. adding named
  keys per model: chosen reuse — **zero new physical schema keys**, full v6
  back-compat, shorter URLs — at the cost of the two slider *labels* changing per
  selected sub-model (a UI lookup table, not a schema cost). The alternative adds
  ~10 `ph.*` keys that are unused for 7 of 8 models.
- **Single worklet bundle** vs. per-sub-model lazy bundles (§B.7): chosen single
  bundle (no shared-DSP duplication) against the prompt's literal "lazy per
  sub-model" wording. Flagged for sign-off.
- **Wavetable caps at 8 partials, morph layer instantiated only when used:**
  trades the theoretical 16-partial wavetable for a safe node budget; per-partial
  *independent* position/morph is explicitly out of scope (post-v1.1).
- **`HARMONICS` extended 8 → 16:** a shared-type change touching every engine's
  slice; mitigated because engines capped at 8 never read the new indices.
