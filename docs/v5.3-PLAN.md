# AnnealMusic v5.3 · Session Datalogging Plan

This document outlines the architecture, data schema, format options, and integration strategy for adding comprehensive session datalogging to AnnealMusic v5.3.

---

## 1. Final Data Schema v1

The logged tick object conforms to a strict TypeScript interface `SessionLogRecord`. This schema is designated as **v1** and will remain stable indefinitely.

```typescript
export interface SessionLogRecord {
  // --- Timing ---
  timestamp: number; // AudioContext.currentTime in seconds since session start
  wallTime: string; // ISO 8601 string of the real-world time

  // --- Sculpt Params (v1.0+) ---
  params: {
    rootFreq: number; // Base carrier frequency in Hz
    spread: number; // Spectral spread factor (-1..1)
    density: number; // Harmonic density / partial count (1..32)
    coupling: number; // Kuramoto coupling strength (0..1)
    drift: number; // Drift random walk speed (0..1)
    brightness: number; // Filter cutoff warp (0..1)
    space: number; // Convolver wet mix gain (0..1)
    volume: number; // Master channel volume (0..1)
  };

  // --- Session Metadata ---
  metadata: {
    mode: 'sketch' | 'drone';
    engineId: string; // Active engine ('sine', 'granular', etc.)
    engineParams: Record<string, any>; // Active engine's internal parameters
    tuning: {
      system: 'equal' | 'just' | 'pythagorean' | 'custom';
      referenceA4Hz: number;
      sclId?: string;
    };
    schemaVersion: string; // State schema version (e.g. 'v20')
    logSchemaVersion: string; // Datalog schema version ('1.0')
  };

  // --- Drift Internal State ---
  drift: {
    meanDetune: number; // Mean detune across all partials (cents)
    orderParameter: number; // Kuramoto phase coherence r(t) (0..1)
    partials: number[]; // Per-partial current detune value (cents)
  };

  // --- Audio Engine State ---
  partials: {
    frequencies: number[]; // Per-partial current absolute frequency (Hz)
    amplitudes: number[]; // Per-partial current linear amplitude
  };

  // --- Audio Analysis Features ---
  features: {
    rms: number; // Root Mean Square amplitude of final block
    spectralCentroid: number; // Center of mass of current spectrum (Hz)
    spectralFlux: number; // Difference between current and previous frame
    zcr: number; // Zero Crossing Rate of the audio block (0..1)
    spectrum?: number[]; // Optional: 512-bin magnitude spectrum (clamped dB)
  };

  // --- Event Flags ---
  event: string | null; // 'session-start' | 'session-stop' | 'engine-swap' | 'arc-segment-boundary' | 'midi-input' | 'osc-input' | 'ring-buffer-overflow' | null
  eventData?: Record<string, any>; // Context metadata for the event
}
```

## 2. Format Library Choices (h5wasm vs. JSONL + CLI Conversion)

### Analysis of WASM Format Libraries:

- **`h5wasm` (HDF5 in WebAssembly):** Exposes C-level HDF5 library. Relies on heavy WASM files (~1.8MB gzipped) and complicates Vite builds due to custom FS bindings.
- **`parquet-wasm` / `apache-arrow`:** Heavy Rust WASM compilation (~2.5MB gzipped). Complicates package installation and requires advanced bundler configuration for ESM files.

### Decision:

To maintain a high-performance web bundle with **zero bloat**, we will:

1. **Natively implement JSONL and CSV writers in-browser** using lightweight, zero-dependency, and highly optimized TypeScript.
2. In-app downloads will natively support **JSONL** and **CSV**.
3. Expose a **CLI converter utility** (`annealmusic convert <jsonl-file> -f <format> -o <output>`) in `tools/cli` that uses full Node-native / standard bindings to output **HDF5** and **Parquet** formats for researchers.
4. Support writing all formats (`jsonl`, `csv`, `hdf5`, `parquet`) natively inside CLI batch renders (driven in Node).

This keeps the production browser bundle clean and small, while providing full scientific output features where they are most natural (CLI rendering and local post-processing scripts).

## 4. Logging Mode Definitions & Bandwidth Estimates

| Mode                 | Target Rate | Inclusions                                     | Est. Bandwidth | Memory Cap (100MB) |
| -------------------- | ----------- | ---------------------------------------------- | -------------- | ------------------ |
| **Lightweight**      | 10 Hz       | Params + features, no partials/spectrum        | ~1 KB/s        | ~27 hours          |
| **Standard**         | 50 Hz       | All sculpt + drift + partial states + features | ~10 KB/s       | ~2.7 hours         |
| **Full**             | 50 Hz       | Standard + 512-bin spectrum                    | ~50 KB/s       | ~33 minutes        |
| **Research-Extreme** | block rate  | Standard + spectrum + raw audio chunks         | ~1.5 MB/s      | ~1.1 minutes       |

## 5. In-App Datalogger Panel UI

The Datalogger Panel will be styled as a premium glassmorphic UI card on `/research` page:

- **Live Status Badge:** `IDLE` (gray), `RECORDING` (pulsing emerald), `OVERFLOW WARNING` (pulsing amber).
- **Real-Time Stats:** Ticks logged, elapsed time (MM:SS), memory capacity bar (current / 100MB).
- **Controls:**
  - **Mode Select:** Lightweight, Standard, Full, Research-Extreme.
  - **Rate Select:** Slider or button group (10Hz, 25Hz, 50Hz, 100Hz).
  - **Record Button:** Big action button with micro-animations.
  - **Format Select:** JSONL vs. CSV dropdown.
  - **Download Button:** Visible when recording stops and ticks > 0.

## 6. CLI Flag Integration

`tools/cli` render commands (`render` and `listening`) will support:

- `--log-format <format>`: Output format choice (`jsonl` | `csv` | `hdf5` | `parquet`, default: `jsonl`).
- `--log-out <path>`: Destination path (e.g., `session.jsonl`).
- `--log-rate <rate>`: Sample rate in Hz (default: `50`).
- `--log-mode <mode>`: Lightweight, Standard, Full, Research-Extreme.

## 7. OSC Bridge Integration

The `tools/osc-bridge` will accept a `--log-out <file>` flag:

- Bridges incoming/outgoing OSC messages to the specified file as a JSONL log.
- Record format:
  ```json
  {
    "timestamp": 1782398471239,
    "direction": "in",
    "address": "/anneal/params/brightness",
    "args": [0.65]
  }
  ```

## 8. Pyodide Integration Preview (JSON-RPC)

Exposes datalogger methods through `BridgeServer`:

1. `anneal.datalog.start(mode)`: Starts datalogging at the specified mode.
2. `anneal.datalog.stop()`: Stops datalogging.
3. `anneal.datalog.snapshot(limit)`: Returns a structured array of the latest N ticks from the ring buffer.
4. `anneal.datalog.stream()`: Returns subscription ID, then broadcasts each logged tick via `anneal.datalog.onTick` JSON-RPC notifications.

---

## 9. Performance Budget & Audio Thread Protection

- **Audio Thread Protection:** All calculation of RMS, centroid, flux, and ZCR is calculated on the main thread by copying values from Web Audio's `AnalyserNode` asynchronously.
- **Low GC Overhead:** Tick objects are created with flat structures where possible. Analysis buffers are allocated once and reused.
- **Ring Buffer Capping:** Strictly constrained to 100MB. Ticks are logged in a circular fashion (oldest dropped first) with warnings dispatched when overflow occurs.
- **CPU Footprint:** standard logging runs in <2% CPU overhead on typical research laptops.

## 11. Risks & Mitigations

1. **GC pauses causing dropouts:** mitigated by pre-allocating analysis scratchpads and doing all computations asynchronously off-thread relative to Web Audio DSP.
2. **Memory exhaustion on long sessions:** mitigated by a strict 100MB ring buffer cap that discards older records.
3. **Format compatibility:** The single schema in `schema.ts` acts as the source-of-truth. Tabular formats (CSV) flatten arrays by converting them to JSON strings or naming them `partial_freq_0`, `partial_freq_1`, etc.
