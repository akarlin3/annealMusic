# AnnealMusic v3.6 · Arrangement View · Implementation Plan

This document outlines the design, architecture, and implementation steps for replacing the minimal v3.0 timeline editor with a production-quality, DAW-style **Arrangement View** (multi-track timeline with zoom/scroll, snap modes, region operations, and automation tracks).

---

## 1. Track-Type Model & URL v14 Schema

v3.6 introduces a multi-track data model. A `Piece` contains heterogeneous track types that structure composition and playback parameters across time:

1. **Segment Track** (always present) – defines core structural blocks (`fixed`, `arc`, `open`, `transition`, `meta-arc`).
2. **Notation Track** (v3.2; up to one per piece) – holds `notation` notes.
3. **Pulse-Config Track** (optional, visible if `tempoBpm` is set) – tempo and pulse configuration overrides across time.
4. **Automation Tracks** – one per automated parameter, consisting of piecewise-linear curves.
5. **Meta-Arc Preview Track** (read-only) – displays the procedurally generated curves of `meta-arc` segments.

### 1.1 Data Structures

We will extend `src/piece/types.ts` with automation-specific types:

```typescript
export type InterpolationMode = 'linear' | 'exponential' | 'hold';

export interface AutomationPoint {
  id: string;
  timeMs: number;
  value: number;
  interpolation: InterpolationMode;
}

export interface AutomationTrack {
  id: string;
  paramKey: string; // e.g. 'brightness', 'space', 'rootFreq'
  points: AutomationPoint[];
}

export interface Piece {
  // Existing fields...
  tempoBpm: number | null;
  segments: PieceSegment[];
  notation?: NotationNote[];
  movements?: Movement[];

  // NEW v3.6 fields
  automationTracks?: AutomationTrack[];
}
```

### 1.2 URL Serialization v14

To share and load compositions offline with automation curves, we will bump the schema version to `14` in `src/share/schema.ts` and add `14` to `SUPPORTED_SCHEMA_VERSIONS`.

Automation tracks will be serialized into the URL fragment using namespaced query parameters:
`auto.{paramKey}={timeMs},{value},{interpolationCode};{timeMs},{value},{interpolationCode};...`

Where `interpolationCode` is:

- `l` = `linear`
- `e` = `exponential`
- `h` = `hold`

Example parameter in URL:
`&auto.brightness=0,0.5,l;1500,0.8,e;3000,0.2,h&auto.space=0,0.1,l;5000,0.6,h`

### 1.3 Backend Persistence

To persist the new `automationTracks` array on the backend without requiring a Postgres/SQLite database schema migration, we will:

1. Extend `PieceCreate`, `PieceUpdate`, and `PieceOut` schemas in `api/app/schemas.py` to include `automation_tracks: list[dict] | None = None`.
2. Store `automation_tracks` inside the existing JSONB `defaults_state` column under the hidden key `_automation_tracks`.
3. In `to_out` in `api/app/routers/pieces.py`, unpack `_automation_tracks` and expose it as `automation_tracks` at the root of `PieceOut` so that the frontend receives a clean, standard response.
4. Update `create_piece` and `update_piece` in `api/app/routers/pieces.py` to handle the incoming `automation_tracks` body and save it into `defaults_state["_automation_tracks"]`.

## 2. Automation Data Model & Playback Integration

Automation acts as a continuous controller layer overriding segment default parameter values at the playhead time.

### 2.1 Curve Evaluation

We will implement a pure evaluation function `evaluateAutomation(points: AutomationPoint[], timeMs: number): number | null`:

1. If `points` is empty, return `null`.
2. Sort points by `timeMs` ascending.
3. If `timeMs <= points[0].timeMs`, return `points[0].value`.
4. If `timeMs >= points[last].timeMs`, return `points[last].value`.
5. Find adjacent points `P_i` and `P_{i+1}` such that `P_i.timeMs <= timeMs < P_{i+1}.timeMs`.
6. Calculate normalized progress `t = (timeMs - P_i.timeMs) / (P_{i+1}.timeMs - P_i.timeMs)`.
7. Interpolate based on `P_i.interpolation`:
   - **linear**: `P_i.value + t * (P_{i+1}.value - P_i.value)`
   - **hold**: `P_i.value`
   - **exponential**:
     If `P_i.value !== 0 && P_{i+1}.value !== 0 && (P_i.value * P_{i+1}.value > 0)`:
     `P_i.value * Math.pow(P_{i+1}.value / P_i.value, t)`
     Else, fallback to **linear** to avoid complex number roots or division by zero.

### 2.2 Playback Resolution & Priority

In `PiecePlayer.ts`, parameter resolution at any global playhead time `timeMs` will be evaluated using the following priority order:
$$\text{Notation (highest)} > \text{Automation} > \text{Segment-Level State} > \text{Piece Defaults (lowest)}$$

- **Segment State**: Transitions, Arcs, Meta-Arcs, and Fixed segment parameters computed exactly as in v3.5.
- **Automation Override**: If an automation track exists for parameter `K` and evaluates to a non-null value `V` at `timeMs`, `V` overrides the segment state for `K`.
- **Notation Override**: If there is a notation note playing at `timeMs`, the note's pitch overrides the resolved frequency (`rootFreq`), superseding both automation and segment-level values.

## 3. Region Operations across Heterogeneous Tracks

Composers need to copy, paste, duplicate, and delete sections of the timeline. A selected region is defined by a time span `[startTimeMs, endTimeMs]`.

### 3.1 Region Action Semantics

- **Copy**: Extract all elements within `[startTimeMs, endTimeMs]` across all active tracks:
  - _Segment Track_: Copy all segments falling within or intersecting the selection.
  - _Notation Track_: Copy notation notes starting in the selection.
  - _Automation Tracks_: Copy all automation points in the selection (with times adjusted relative to `startTimeMs`).
- **Delete**: Clear all data within the selection in two modes:
  - _In-Place Clear_: Delete notes/points and leave silence.
  - _Ripple Delete_: Delete the selected range and shift all subsequent segments, notes, and automation points to the left by `(endTimeMs - startTimeMs)`.
- **Paste**: Insert copied content starting at the target playhead position `pasteTimeMs`:
  - _Pasting Automation_: Paste points onto respective automation tracks (re-mapping relative time offset).
  - _Pasting Notation_: Paste notes onto the notation track. If the piece doesn't have a notation track, paste creates it.
  - _Pasting Segments_: Insert segments on the segment track, splitting any overlapping segment at `pasteTimeMs` and shifting subsequent segments right (ripple paste).
- **Duplicate**: Identical to copying `[startTimeMs, endTimeMs]` and pasting it at `endTimeMs`.

## 4. Zoom, Scroll, & Snap Mechanics

### 4.1 Horizontal Zoom

Horizontal scaling maps timeline milliseconds to screen pixels:

- Let `baselineZoom` (1x) be "fit piece to screen," i.e., `clientWidth / totalDurationMs`.
- Zoom range is constrained to `0.5x` (zoomed out) up to `50x` (zoomed in).
- Smooth zoom via Cmd+Wheel or pinch-to-zoom gestures, keeping the playhead or mouse cursor centered.

### 4.2 Vertical Layout & Collapse

- Tracks can be individually expanded or collapsed.
- Expanded tracks (e.g. 80px-120px) show the full editor (grid, piano roll notes, or automation points).
- Collapsed tracks show a **12px-tall summary preview**:
  - _Notation_: A miniature, high-performance horizontal block showing notes.
  - _Automation_: A miniature SVG preview path of the curve.

### 4.3 Snap Modes

Snapping applies when dragging or drawing points, note onsets, and segment boundaries.
Modes cycle via keyboard shortcut `G` or `S`:

1. `off`: Freeform editing.
2. `grid`: Snap to tempo grid intervals (quarter note, eighth note) if a tempo is set.
3. `bar`: Snap to bar boundaries (4 beats).
4. `movement boundary`: Snap to start/end of movements.
5. `segment boundary`: Snap to start/end of segments.

## 5. High-Performance Virtual Rendering Strategy

DAWs must maintain a solid 60fps refresh rate even with hundreds of notes and thousands of automation points.

1. **Horizontal Viewport Culling**:
   Calculate the visible range `[visibleStartMs, visibleEndMs]` from `scrollLeft` and horizontal zoom factor:
   $$\text{visibleStartMs} = \frac{\text{scrollLeft}}{\text{zoomScale}}$$
   $$\text{visibleEndMs} = \text{visibleStartMs} + \frac{\text{viewportWidth}}{\text{zoomScale}}$$
2. **Track-Level Culling**:
   - _Notation Render_: Binary search notes list to render only notes intersecting the visible range.
   - _Automation Render_: Filter automation points to visible window. Draw lines to adjacent boundary points to keep continuous lines beautiful.
   - _Segment Render_: Render only visible segments.
3. **Canvas / SVG Drawing**:
   Use highly optimized SVGs with `vector-effect="non-scaling-stroke"` or lightweight HTML elements absolute-positioned based on calculated pixel coordinates.

## 6. Mobile Editor Scope

Mobile uses a streamlined gesture and screen layout:

- **Pan + Zoom**: Multi-touch zoom is mapped to single-finger pan and pinch zoom on the timeline canvas.
- **Header Collapse**: Track list and headers are minimized to maximize vertical space.
- **Ergonomics**: Single-tap selects a track, long-press or double-tap adds an automation point/note to minimize keyboard reliance.

---

## 7. Collaborative CRDT Sync & Performance

All track operations are synchronized in real time via Yjs:

- **Automation Point Throttling**: Broadcasters will throttle dragging updates at 100ms or apply changes locally at 60fps and sync the final coordinates only on `mouseup`. This prevents Yjs doc size explosion.
- **Delta Syncing**: Points are serialized inside a `Y.Array` or `Y.Map` by track, allowing efficient delta updates.

## 9. Risks & Mitigations

- **URL Length Limits**: Thousands of automation points could exceed standard browser URL length bounds (~2048 chars).
  - _Mitigation_: Implement simple Douglas-Peucker curve simplification on export to reduce unnecessary dense collinear points before serialization, or fall back to backend save links for extremely dense pieces.
- **Yjs Sync Lag**: Massive automation track Yjs sync could lag.
  - _Mitigation_: Debounce CRDT synchronization during dragging, ensuring only key frames and final values are synced.
