# Implementation Plan — v1.6 MIDI Integration

Bring MIDI control into **Anneal Ambiance v1.6**. The Web MIDI API exposes attached MIDI controllers and devices to the browser. This plan establishes how Anneal Ambiance will map MIDI inputs, handle notes, emit MIDI outputs, manage clock synchronization, and resolve browser-specific limitations on web platforms.

---

## 1. Browser Support & Platform Matrix

Web MIDI API support is inconsistent across modern browsers and completely absent in Capacitor web views. We establish a strict support gate and clear fallback user experiences.

### Compatibility Table

| Platform / Browser    | Web MIDI Support | Fallback / Notification UX                                      |
| :-------------------- | :--------------- | :-------------------------------------------------------------- |
| **Google Chrome**     | Native (Full)    | Exposed in UI; standard permission prompt.                      |
| **Microsoft Edge**    | Native (Full)    | Exposed in UI; standard permission prompt.                      |
| **Opera**             | Native (Full)    | Exposed in UI; standard permission prompt.                      |
| **Mozilla Firefox**   | Native (108+)    | Supported; permission prompt triggered.                         |
| **Apple Safari**      | None (2025/2026) | Clear "MIDI not supported" card with link to download Chrome.   |
| **iOS App (Cap)**     | None             | Blocked/Flagged; mobile MIDI deferred to a later custom plugin. |
| **Android App (Cap)** | None             | Blocked/Flagged; mobile MIDI deferred to a later custom plugin. |

### Feature Flagging & Safe Injection

- **Feature Gate**: A global check `const isMidiSupported = () => typeof navigator.requestMIDIAccess === 'function'`.
- **UI Availability**:
  - On **unsupported** engines/browsers, the MIDI configuration buttons/links are completely omitted or shown as a disabled state with a tooltip suggestion ("MIDI features require Chrome, Edge, or Firefox").
  - On **supported** platforms, a new "MIDI" button is added to the footer and main header alongside "Export Stems".
- **Capacitor Mobile Check**: `VITE_MOBILE` or standard `Capacitor.isNativePlatform()` flags will forcefully disable the MIDI route/modal to prevent silent failures in WKWebView or Chrome WebView.

## 2. Permission Flow & Gatekeeping

Accessing Web MIDI requires explicit user approval. We construct a premium, transparent permission UX that avoids abrupt browser prompts.

```
                  [User clicks MIDI Settings / Enable MIDI]
                                     │
                        Is permission already granted?
                         ├── YES ──> Initialize MIDI engine & list devices
                         └── NO
                                     │
                       Show Glassmorphic Permission Card
                       (Explains what MIDI access does)
                                     │
                             [Click "Grant Access"]
                                     │
                  navigator.requestMIDIAccess({ sysex: false })
                         ├── Approved ──> Save state; scan ports; notify user
                         └── Denied ────> Show "Permission Denied" troubleshooting info
```

### Permission states and UX:

1. **First-Call Prompt**: The user is shown an elegant, animated onboarding card explaining what MIDI is and why permission is needed. The actual browser API is only invoked when they click **"Enable MIDI"**.
2. **Access Parameters**: We permanently request `navigator.requestMIDIAccess({ sysex: false })`. System Exclusive (SysEx) access is completely locked to `false` to reduce the security permission warning surface (which otherwise warns the user about "dangerous hardware control").
3. **Persistent Permission**: Once accepted, standard browsers persist the permission for the origin. The onboarding card is automatically bypassed in future sessions.
4. **Permission Denied UX**: If the user blocks the prompt, the interface displays a visual warning:
   - "MIDI Access Blocked by Browser".
   - Clear instructions on how to click the browser lock/settings icon in the URL bar to reset the permission.
   - A retry button.

## 3. PlatformBridge Extension

We evaluate if `PlatformBridge` needs a dedicated wrapper for MIDI access.

- **Recommendation**: **Direct access**. Since MIDI is purely web-scoped in v1.6 and we have explicitly deferred native mobile MIDI, extending the platform bridge adds unnecessary boilerplate and layer indirection.
- **Bridge-Safe Storage**: To keep options open for a future Capacitor Preferences plugin, the mapping persistence layer in `src/midi/storage.ts` will leverage standard `localStorage` on the web, but encapsulate it cleanly:
  ```typescript
  export const saveMappings = async (
    key: string,
    value: string,
  ): Promise<void> => {
    localStorage.setItem(key, value);
  };
  export const loadMappings = async (key: string): Promise<string | null> => {
    return localStorage.getItem(key);
  };
  ```

## 4. Input Mapping Model

We will build a flexible, multi-dimensional mapping engine that translates high-frequency 7-bit MIDI CC fader streams into precise, smoothed parameter adjustments.

### `MappingSet` Data Structure

```typescript
export interface MidiMapping {
  paramKey: string; // Key of the target parameter (shared or engine-specific)
  isEngineParam: boolean; // Flag to differentiate useParamStore.params vs engineParams
  min: number; // Configured minimum range (maps from CC value 0)
  max: number; // Configured maximum range (maps from CC value 127)
  curve: 'linear' | 'exponential' | 'logarithmic'; // Transition curve type
}

export interface MappingSet {
  controllerId: string; // The physical MIDI device name / manufacturer
  channel: number; // MIDI channel filter (0 = Omni, 1..16 = specific channel)
  mappings: Record<number, MidiMapping>; // Key is the CC number (0..127)
}
```

### Parameter Curves

To support premium feeling response rates across parameters like frequency, we define three response curves:

1. **Linear**: Traditional 1:1 interpolation:
   $$y = y_{min} + \left( \frac{x}{127} \right) \cdot (y_{max} - y_{min})$$
2. **Exponential**: Best for filters and brightness where human hearing is logarithmic. More resolution at the bottom of the fader:
   $$y = y_{min} \cdot \left( \frac{y_{max}}{y_{min}} \right)^{\frac{x}{127}}$$ (clamped if $y_{min} \le 0$)
3. **Logarithmic**: More resolution at the top of the fader:
   $$y = y_{min} + (y_{max} - y_{min}) \cdot \log_{10}\left( 1 + 9 \cdot \frac{x}{127} \right)$$

### Per-CC Min/Max Clamping

Users can configure range boundaries (e.g. setting `min = 100` and `max = 500` for `rootFreq` to limit fader bounds to a comfortable octave band).

## 6. MIDI Learn State Machine

The MIDI Learn flow allows manual assignments of hardware CCs to any param slider in seconds.

```
       [Idle]
         │  (Click "Learn" button next to slider, or press 'L' on slider focus)
         ▼
    [Waiting for CC] ──> (Visual feedback: Amber pulse glow, slider says "Listening...")
         │
         ├── CC Received ──> Capture CC # and Controller ID
         │                     │
         │                     ▼
         │               [Confirm Map] (Replaces any existing map for this CC)
         │                     │
         ▼                     ▼
       [Idle] <────────── [Confirmed] (Visual flash of emerald, mapping saved)
```

- **Cancellation**: Cancel learn mode at any time by pressing `Esc`, clicking outside the slider, or clicking the Learn button again.
- **Collisions**: If the wiggled CC is already bound to another parameter, a toast confirms: "Reassigned CC [N] from [Old Param] to [New Param]".

## 7. Note-to-Root Frequency Mapping

In addition to faders, users can plug in a keyboard to play Anneal Ambiance.

### Pitch-to-Frequency Calculations

We use the standard A4 = 440 Hz Equal Temperament convention:
$$f = 440 \cdot 2^{\frac{d - 69}{12}}$$
where $d$ is the MIDI note number ($0 \le d \le 127$).

### Pre-MIDI Root Preservation

- **Behavior**: When the user presses a MIDI key, the app records their current pre-MIDI root frequency.
- **Held Note Priority**: If they hold a note, the root changes to that pitch. If they play multiple notes, we apply **last-note-priority** (monophonic behavior).
- **Note-Off Behavior (Release)**:
  - **Option A (Persist)**: The frequency remains locked to the pitch of the last released note.
  - **Option B (Return)**: Releasing the key returns the root frequency to the original pre-MIDI slider value.
  - **Config**: Configurable via a toggle in settings: "Note Release: [Sustain | Return]". Default is **Sustain** (persist) to preserve calm ambient shifts.
- **Clamping**: Notes outside A0 (21) to C8 (108) are ignored to prevent standard engines from overloading or collapsing into inaudibility.

## 8. Velocity Mapping

Velocity (MIDI keyboard striking force, 0–127) can dynamically modulate synthesis properties.

- **Standard Default Targets**:
  - **Sine / FM / Granular Engines**: Velocity is ignored (kept ambient and static) by default to avoid abrupt dynamic jumps.
  - **Physical Engine**: Velocity maps to `excitationLevel` ($v/127$), creating striking mallet responses or harder bowed sounds based on how hard keys are hit.
- **User Config**: In settings, users can assign velocity to modulate any of the 8 parameters (e.g. mapping velocity to `space` or `brightness` on note-on triggers).

## 9. CC Output Schema

To support recording sessions directly in external DAWs, Anneal Ambiance emits high-resolution CC streams as parameters shift (either via the UI or internal walks).

### Table of Emitted CCs

We use CC numbers selected from standard General MIDI CC ranges or the "undefined" range to ensure safety:

| Parameter  | Emitted CC | Purpose / Description                       |
| :--------- | :--------- | :------------------------------------------ |
| rootFreq   | CC 74      | Sound Brightness / Cutoff (GM convention)   |
| spread     | CC 75      | Sound Control 6                             |
| density    | CC 76      | Sound Control 7                             |
| coupling   | CC 77      | Sound Control 8                             |
| drift      | CC 78      | Sound Control 9                             |
| brightness | CC 71      | Filter Resonance (GM resonance)             |
| space      | CC 72      | Release Time (GM conventional space/reverb) |
| volume     | CC 7       | Channel Volume (GM standard volume)         |

- **Output Throttle**: Emitting MIDI messages at 100+ events per second will crash slow browser tabs and overwhelm virtual MIDI drivers. We implement a **60Hz throttle** per CC channel using a requestAnimationFrame or interval accumulator.

## 10. MIDI Clock Output (Master Clock)

Anneal Ambiance can sync hardware drum machines, Eurorack gear, or DAW sequencers by acting as the tempo master.

- **BPM Range**: 30 to 240 BPM (Default: 60 BPM).
- **Pulses Per Quarter Note (PPQN)**: Standard 24 PPQN MIDI clock rate.
- **Tick Calculation**:
  $$\text{Interval (ms)} = \frac{60000}{\text{BPM} \times 24}$$
  For 60 BPM, the interval is exactly $41.67\text{ ms}$ (24 messages per second).
- **Transport Messages**:
  - **System Real-Time Clock**: CC `0xF8` (decimal 248) sent every tick.
  - **Start**: CC `0xFA` sent when the Session `Begin` action triggers.
  - **Stop**: CC `0xFC` sent when the Session `Settle` or stop action triggers.

## 11. Output Device Selection

- **Connection Scan**: We scan `MIDIAccess.outputs` and render a dropdown picker in settings.
- **Fallback**: The output defaults to "None". Users must explicitly activate an output device to avoid spamming other synthesizer setups.
- **Hot-Plug updates**: An event listener on `MIDIAccess.onstatechange` triggers react state updates within 200ms of any physical plug/unplug event, keeping picker dropdowns live.

## 14. Performance Safeguards

1. **CC Input Debounce**: Incoming CC inputs execute at the audio processor target rates. Since the standard engines utilize linear ramp smoothing (`setTargetAtTime` over 50ms), we do not need custom input debouncers, avoiding parameter stepping or clicking.
2. **CC Output Rate Limit**: A 60Hz limit prevents overloading. If the user moves a slider, we update the engine synchronously but cache the value and let a `60Hz` setInterval/timer pump the CC output to external devices.
3. **Audio Thread Isolation**: Web MIDI messages operate entirely on the main DOM thread. There are zero blockages or allocations sent to the Web Audio AudioWorklet threads, protecting audio fidelity against dropouts or crackles.

## 15. Testing Strategy

### Unit Tests (`vitest`)

- **Curve Math**: Verify CC value (0..127) translations into param bounds using linear, exponential, and logarithmic formulas.
- **Auto-Mapping**: Verify controller ID string matching results in correct initial mappings.
- **MIDI Notes**: Verify MIDI note values (e.g. 60 -> C4 -> 261.63Hz) map to frequency numbers accurately.
- **Clock Pulse**: Verify PPQN timers emit exact count at varying BPM rates.

### Browser-Level Mocks

- Mock `navigator.requestMIDIAccess` to simulate port addition, port deletion, hot-plugs, and MIDI messages (Notes, CC, clock).

## 16. Risks & Mitigation Plan

- **Risk**: User has multiple controllers wiggled at once during Learn mode.
  - _Mitigation_: We process only the _first_ CC wiggled after learn is activated, and immediately drop out of learn mode or enter a confirm stage, preventing overlapping collisions.
- **Risk**: Double-mapping. Two faders mapped to the same CC.
  - _Mitigation_: The input controller stores mappings by CC number. Therefore, a single CC can map to only one parameter at a time. Wiggling a new parameter on that CC overrides the old parameter and triggers a visual toast explaining the replacement.
- **Risk**: Safari Web App confusion.
  - _Mitigation_: On Safari, a visually striking header card warns: "Safari does not support the Web MIDI API. Please open Anneal Ambiance in Google Chrome or Microsoft Edge to connect external instruments and controllers."
