# Implementation Plan — AnnealMusic v6.2 · Audio Clip Library

This document is the **Checkpoint 0** deliverable for v6.2. It finalizes the design before any
code is written. v6.2 adds a **curated library of short audio examples** (5–60 s clips) that
lessons reference by slug, a new **`audio-clip` lesson step type**, and a **clip-retrieval**
capability the v6.1 generation pipeline can use to anchor concepts in sound.

> Branch: `claude/v6.2-audio-clips-r6viu`
> Prereqs shipped: v6.1 (LLM lesson generation pipeline, per-step prompts + validators + cache),
> v1.7 (embedding pipeline, schema-in-prompt, validation/retry), v4.3 (Opus bell library),
> v0.9 (Opus source library), v1.5 (24-bit WAV export).

Three facts that drive design decisions below:

1. **Cosine search is done in Python** over candidate rows, not via the SQL `<=>` operator. The
   ivfflat index exists but the read path fetches rows and scores in app code. For a ≤60-row clip
   library this is _more_ than fast enough; the index is forward-looking (§7).
2. **The bridge has no audio suspend/resume.** The embedded app owns its `AudioContext`. v6.2 must
   add `suspendEngine`/`resumeEngine` RPC methods so clips don't fight the live engine (§6).
3. **Lessons reference clips by `slug`, not UUID.** Slugs are the stable, portable wire id (like
   `sources/registry.ts` `index`/`id`), so a lesson spec/JSON stays valid across DB reseeds.

## 2. Clip metadata, manifest, seed + embedding pipeline

### Committed manifest = source of truth

The shipped library is described by a committed JSON manifest, `api/data/clip_library.json`
(mirrors how `api/data/lesson_examples.json` is the committed source for few-shot data). Each entry:

```jsonc
{
  "slug": "karplus-archetype",
  "title": "Textbook Karplus Pluck",
  "description": "A textbook Karplus pluck — bright, percussive, with a clean decay.",
  "duration_ms": 8000,
  "file": "clips/karplus-archetype.opus", // resolves to /public/clips/…
  "track_affinity": ["synthesis-fundamentals"], // track slugs
  "concept_tags": ["karplus-strong", "string", "physical-modeling", "pluck"],
  "license": "original-by-you",
  "attribution": null, // required string when license ∈ {CC0,CC-BY,licensed-third-party}
}
```

The manifest carries **no embedding** — embeddings are derived (§ below), kept in the DB only,
exactly like patches/pieces (never committed).

### Seed → DB

`api/scripts/seed.py` gains `seed_audio_clips()`: upsert each manifest entry into `audio_clips`
keyed by `slug`, computing `description_embedding` from `description + " " + concept_tags` via the
configured `EmbeddingClient` (Mock in tests/CI → deterministic; OpenAI in prod). Re-seeding is
idempotent (slug unique); a changed description re-embeds. `storage_key` is set to
`public:clips/<slug>.opus` for manifest clips (§4 storage convention).

### Embedding at admin-add time

The admin upload endpoint (`POST /admin/clips`) computes the embedding on create/patch (reuses the
v1.7 pipeline), so newly authored clips are immediately retrievable before they are exported into
the committed manifest. Embedding text = `description + tags` (the same recipe used at seed, so
seed and live upload agree). This satisfies the brief's "embedded at clip-add time" + the
`clip_embeddings` requirement — implemented as a column on `audio_clips`, not a side table (§3
rationale).

## 3. Data model + migration (`0020_v6_2_audio_clips`)

Revises `0019_v6_1_lesson_generation`. Dialect-aware (PG vs SQLite), matching repo style. One new
table; **no change to `lessons`/`lesson_steps`** (the `audio-clip` step lives in the existing
`lesson_steps.config` JSON — same as every other step type).

```sql
CREATE TABLE audio_clips (
  id            UUID PRIMARY KEY,                 -- GUID(), _uuid default
  slug          TEXT UNIQUE NOT NULL,             -- URL-safe; the wire id lessons reference
  title         TEXT NOT NULL,
  description   TEXT NOT NULL,
  duration_ms   INTEGER NOT NULL,
  storage_key   TEXT NOT NULL,                    -- 'public:clips/<slug>.opus' OR object-store key
  track_affinity TEXT[] NOT NULL DEFAULT '{}',    -- UUIDArray→TextArray; JSON on SQLite
  concept_tags  TEXT[] NOT NULL DEFAULT '{}',
  license       TEXT NOT NULL,                    -- enum-checked (CK constraint)
  attribution   TEXT,                             -- NOT NULL-enforced in app when license≠original-by-you
  description_embedding  vector(1536),            -- VectorType(1536)
  archived_at   TIMESTAMPTZ,                      -- soft-delete (DELETE = archive)
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX uq_audio_clips_slug ON audio_clips(slug);
CREATE INDEX idx_clip_embeddings ON audio_clips
  USING ivfflat (description_embedding vector_cosine_ops)
  WHERE archived_at IS NULL;                      -- PG only; skipped on SQLite
-- CK constraint: license IN ('CC0','CC-BY','original-by-you','licensed-third-party')
```

**Decisions vs the brief's draft schema**

- **`clip_embeddings` table → folded into `audio_clips.description_embedding`.** A 1:1 side table
  buys nothing here (no versioning of embeddings is required, and patches/pieces already store the
  vector inline). Calling this out as the one deliberate deviation from the brief's data-model
  sketch.
- The ivfflat index + partial `WHERE archived_at IS NULL` is created **PG-only** (guarded by
  `dialect.name == 'postgresql'`), like `0006`/`0010`.
- `license` gets a `CHECK` constraint (the four-value enum); `attribution` requiredness is enforced
  in the schema layer (a `CHECK` can't express "required unless original-by-you" portably).

New SQLAlchemy model `AudioClip` in `api/app/models.py` follows the `Patch`/`LibraryListing`
conventions (GUID PK, `VectorType(1536)`, `archived_at`).

## 4. Storage & streaming (reconciling static vs API)

The brief lists **both** static hosting in `public/clips/` _and_ a `GET …/clips/:slug/audio`
streaming endpoint. They coexist via one rule on `storage_key`:

- **Shipped (manifest) clips** → committed to `public/clips/<slug>.opus`; `storage_key =
"public:clips/<slug>.opus"`. Served as a static asset (consistent with bells/sources, works on
  Firebase static hosting, lazily loaded by the player). Total ~10–20 MB across 51 clips.
- **Admin-uploaded, not-yet-released clips** → bytes go to object storage via `StorageClient.put`;
  `storage_key` is the object key. Served through the API streaming endpoint.

The metadata endpoint resolves an `audio_url` for the client:

```
storage_key starts with "public:"  →  audio_url = "/<rest>"            (static)
otherwise                          →  audio_url = "/api/v1/clips/<slug>/audio"  (streamed)
```

`GET /api/v1/clips/:slug/audio` streams Opus (`content-type: audio/opus`) — `302` to a presigned
URL when the storage client provides one (mirrors how previews are served), else `StreamingResponse`
of `StorageClient.get`. Range requests are passed through for seek support.

**Authoring → release flow:** author in-app → upload via admin (object storage, retrievable
immediately for testing) → on release, `tools/export-clips.mjs` pulls approved clips into
`public/clips/`, appends to `clip_library.json`, and flips `storage_key` to the `public:` form. CP3
runs this for the 51 clips.

## 5. `audio-clip` step type

A **new, distinct step type** (not folded into `demo`, per the brief). Extended in lock-step across
the three definition sites:

- `src/learn/LearnApp.tsx`: `type: 'text' | 'demo' | 'prompt' | 'reflection' | 'audio-clip'`.
- `api/app/schemas.py`: `StepType = Literal[..., "audio-clip"]`; `LessonSpecStep` gains
  `clip_topic: str | None` (the authoring brief — what the clip should illustrate).
- `src/learn/stepTypes/StepContainer.tsx`: `case 'audio-clip': return <AudioClipStep …/>`.

**Step `config` shape** (what generation/override writes, what the player reads):

```jsonc
{
  "clip_id": "karplus-archetype", // slug (portable). Validated to exist + be non-archived.
  "intro_text": "Listen to a textbook Karplus pluck — bright, percussive, with a clean decay.",
  "outro_text": "Notice the gentle high-frequency roll-off as the delay-line filter softens repeats.",
  "auto_advance": false,
  "loop": false,
  "start_ms": 0, // optional sub-clip window
  "end_ms": null,
}
```

**Validation** (`validate_audio_clip()` in `lesson_generation.py`, and at spec-POST time in
`lesson_admin.py`): `clip_id` resolves to a non-archived `audio_clips.slug`; `intro_text` non-empty
(≤ 280 chars); `outro_text` optional (≤ 280); `auto_advance`/`loop` boolean; `start_ms`/`end_ms`
within `[0, duration_ms]` with `start < end`. A reference to a missing/archived clip is a hard
failure (→ retry / admin fix), never silently dropped.

## 6. Lesson player UX + embedded-app audio pause

`src/learn/stepTypes/AudioClipStep.tsx` renders, in order:

1. **Intro text.**
2. On mount (or on first Play), **`bridge.suspendEngine()`** so the embedded synth's `AudioContext`
   is suspended and clips don't fight the live engine.
3. A **waveform display + transport** (play / pause / seek / loop). Audio plays from the **parent
   (lesson) frame** via an `HTMLAudioElement`/WebAudio pointed at `audio_url` (lazy-loaded on first
   reference; cached thereafter). Waveform is drawn from a decoded peaks summary (cheap, mono).
4. On clip end → **outro text** appears; if `auto_advance` → advance, else show **Continue**.
5. On step exit / unmount → **`bridge.resumeEngine()`** (idempotent; safe if the engine was never
   running) and `releaseConstraints()` per existing flow.

**Bridge additions** (the one genuinely new transport capability):

- `src/research/bridge/BridgeClient.ts`: `async suspendEngine(): Promise<boolean>`,
  `async resumeEngine(): Promise<boolean>` (JSON-RPC like the existing `start/stopSession`).
- The **embedded-app host handler** (the side that answers `loadPatch`/`startSession`) gains
  `suspendEngine`/`resumeEngine` → `audioContext.suspend()` / `.resume()`. Idempotent; tracks
  whether _it_ suspended (so it doesn't resume an engine the user had paused).

This is the cleanest place to honor "embedded app pauses cleanly during clip playback" — playback
and the live engine live in two different audio contexts (parent frame vs iframe), so suspending
the iframe context fully de-conflicts them.

## 7. Clip retrieval (ONE shared module)

Per rules of engagement ("retrieval logic lives once, shared between admin search and LLM
retrieval"), all ranking lives in **`api/app/services/clip_retrieval.py`**:

```python

    # candidates = non-archived audio_clips
    # score = w_e * embedding_sim + w_t * tag_jaccard + w_a * affinity_match
    ...
```

Scoring signals (the brief's three):

1. **Embedding similarity** — if `query_text` given, embed it and `1 - get_cosine_dist(...)` against
   `description_embedding` (reusing `get_cosine_dist`; exact in-Python scan over ≤60 rows).
2. **Tag intersection** — Jaccard of `tags` against `concept_tags`.
3. **Track affinity** — boost when `track ∈ track_affinity`.

Weights are module constants (`W_EMBED=0.6, W_TAGS=0.3, W_AFFINITY=0.1`), tuned in CP3 against the
smoke query _"physical modeling string"_ → expects `karplus-archetype`, `physical-engine-archetype`,
`physical-bowed-string`. Returns top-`limit` `ClipMatch{slug, title, description, score}`.

Consumed by both `GET /admin/clips/search` and the generation pipeline (§8) — never reimplemented.

## 8. LLM clip retrieval + generation-prompt extension

Two paths, both built on §7's `search_clips`:

**(A) Authored `audio-clip` step (primary, CP2).** Spec author adds an `audio-clip` step with a
`clip_topic`. Generation:

1. `search_clips(query_text=clip_topic, tags=<lesson tags>, track=<lesson track>, limit=3)`.
2. A small LLM call (`_audio_clip_system()`) is given the 3 candidates (slug + description) + lesson
   context and returns **`{ "clip_id": <one of the 3 | null>, "intro_text": …, "outro_text": … }`**.
   `null` ("none materially helps") → step marked `generation_failed` with a clear message so the
   admin revises the `clip_topic` or hand-authors. The prompt instructs: _pick a clip only when it
   materially helps; prefer `null` over a weak match._
3. `validate_audio_clip()` (§5) confirms the chosen `clip_id` is one of the candidates and exists.

**(B) Spontaneous suggestion on a `text` step (CP2, admin-gated).** The `text` system prompt is
extended so the model **may** emit an optional `audio_clip_suggestion: <concept query>` alongside
its markdown. The pipeline resolves it via `search_clips` and, rather than auto-inserting a step,
**surfaces the top candidate to the admin** in the step editor as _"suggested clip: X — accept?"_.
This honors "the LLM can request a clip relevant to concept X" while keeping the _"LLM choosing
inappropriate clips"_ risk behind a human gate (see §14). Auto-insertion is a flagged open decision.

New `kind` on `ai_generations`: `lesson-audio-clip`. Cache key reuses `lesson_cache_key(...)` with
`step_type="audio-clip"`; cost is the small picker call (~300 in / 80 out, ≈ $0.001). Few-shot
pairs for the picker land in `lesson_examples.json` under `audio-clip` (3 examples incl. one
`null`). `LESSON_PROMPT_VERSION` bumps to `v6.2.0` (any prompt/example edit ⇒ version bump, per the
existing CI hash-lock).

## 10. API surface

```
# Public
GET    /api/v1/clips/:slug              -- metadata (+ resolved audio_url)
```

## 14. Risks & mitigations

| Risk                                                                | Mitigation                                                                                                                                                                                                                                      |
| ------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Clip library quality drift** (inconsistent loudness/length/style) | A per-clip QA checklist + a normalization pass in `export-clips.mjs` (target LUFS, 96 kbps mono, trim silence); manifest is reviewed in one diff at release; counts/bands fixed in §1.                                                          |
| **LLM picks an inappropriate clip**                                 | Retrieval returns only top-3 _candidates_; the picker may (and is told to prefer) `null`; spontaneous suggestions are **admin-gated**, not auto-inserted (§8); `validate_audio_clip` rejects out-of-set ids.                                    |
| **Licensing edge cases**                                            | CP0 call: ship 100 % homages + CC0 references, **zero** licensed canonical works; four-value license enum + CK; CI gate (§11) blocks any clip lacking a license/attribution; homage descriptions say _"in the style of"_, never claim a source. |
| **Static vs streamed audio confusion**                              | One `storage_key` convention (`public:` prefix) + a single `audio_url` resolver (§4); the same player transport handles both.                                                                                                                   |
| **Embedded app and clip audio fight**                               | Two separate audio contexts; `suspendEngine`/`resumeEngine` bridge calls (idempotent, ownership-tracked) around playback (§6).                                                                                                                  |
| **Retrieval logic duplicated** (admin vs LLM drift)                 | Single `clip_retrieval.search_clips`; both callers go through it; a test pins the _"physical modeling string"_ expectation.                                                                                                                     |
| **Embedding cost/availability on seed**                             | Mock client in CI/tests (deterministic); real embeddings computed once at admin-add/seed; manifest carries no embedding so reseed regenerates cleanly.                                                                                          |
| **Brief's `clip_embeddings` table vs inline column**                | Deliberate, documented deviation (§3) — inline `description_embedding` matches patches/pieces; flagged as open decision below.                                                                                                                  |
