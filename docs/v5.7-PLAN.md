# AnnealMusic v5.7 · v5 Closeout · Technical Implementation Plan

This plan outlines the design, architecture, and step-by-step roadmap for closing out the **AnnealMusic v5 Research Arc**, transitioning it from a suite of innovative features into a highly credible, production-ready research instrument.

---

## 1. Doc Generator Choice & Architecture

### Choice: VitePress

We select **VitePress** as the documentation-as-code generator for the research API reference site.

### Justification:

1. **Perfect Tech Stack Synergy**: AnnealMusic is a high-performance web application built with React, TypeScript, and Vite. Using VitePress (built on Vite + Vue) ensures instant hot module reloading (HMR) and uses the identical bundler setup (Vite) as the main application and its sub-builds.
2. **Minimal Overhead**: Highly optimized, lightweight, and requires minimal custom configurations compared to Docusaurus (heavy Webpack/React-based site) or MkDocs (Python-based, which introduces another environment/runtime dependency into our JS/TS-centric repo).
3. **Markdown-First**: Write pure Markdown for research-oriented documents while supporting embedding raw Vue components directly if needed.

## 2. Recipe Book Catalog (15+ Working Examples)

Each recipe contains a structured description, code implementation, expected outputs (spectral/behavioral), and links to run the script in `/research`.

### Composer / Generative Musician:

1. **"Generate 100 ambient drone variations and listen to the gallery"**
   - _Prose:_ Loop through parameter grids in the Python Scripting Panel to spawn drone variations with varied coupling, drift, and density.
   - _Code:_ Python script setting states and logging variations.
2. **"Use Python to create a random walk through a constrained parameter space"**
   - _Prose:_ Brownian motion over the coupling, brightness, and drift coordinates, keeping the engine within pleasant physical bounds.
   - _Code:_ Python async loop walking parameters.
3. **"Drive AnnealMusic from SuperCollider over OSC"**
   - _Prose:_ Complete SuperCollider code sending to `/anneal/control/root` and subscribing to state updates on `/anneal/state/root`.
   - _Code:_ SuperCollider code block.

### Music Technologist:

4. **"Build a custom controller layout in TouchOSC and bind to sculpt params"**
   - _Prose:_ Setting up an OSC mapping for TouchOSC layout driving `/anneal/control/brightness` and `/anneal/control/space`.
   - _Code:_ OSC layout mapping configuration file description + bind script.
5. **"Bridge AnnealMusic to a Eurorack system via OSC → CV-MIDI converter"**
   - _Prose:_ Routing OSC state outputs to hardware MIDI interfaces (like CV/Gate converters) to let oscillator sync drive analog modular systems.
   - _Code:_ Python OSC client bridging state to MIDI out.
6. **"Run AnnealMusic headless on a Raspberry Pi as an installation backend"**
   - _Prose:_ Instructions on running the Node-based CLI `render` tool headlessly inside a systemd daemon on Linux.
   - _Code:_ CLI configuration script and systemd unit service file.

### Music Cognition Researcher:

7. **"A consonance-dissonance rating experiment from scratch"**
   - _Prose:_ Set up stimulus blocks that compare perfect fifths/octaves with highly dissonant tritones/minor seconds, collecting 7-point Likert ratings.
   - _Code:_ Python experiment runner using `Experiment` and `LikertResponse` APIs.
8. **"A 2AFC pitch-direction experiment with counterbalancing"**
   - _Prose:_ Present consecutive dyads (pitch A vs B) and collect two-alternative forced choices (higher vs lower) with randomized trials.
   - _Code:_ Python experiment runner using `ForcedChoice` block with counterbalancing.
9. **"Continuous-response valence rating with the visualizer hidden"**
   - _Prose:_ Present a 2-minute drone segment and track continuous coordinate pointer movements mapping subject valence, with all visualizers hidden to prevent visual bias.
   - _Code:_ Python experiment runner utilizing `Continuous` feedback logging.

### MIR / ML:

10. **"Generate a 10,000-clip dataset of varied ambient textures with controlled spectral features"**
    - _Prose:_ High-throughput dataset creator generating short wav files named with their exact generative seed and parameter bounds.
    - _Code:_ CLI batch script / python loop running offline.
11. **"Extract librosa-style features over a 30-min generative session and cluster"**
    - _Prose:_ Recording continuous spectral centroid, spectral flux, and RMS energy via the datalogger, then clustering the states in Python.
    - _Code:_ Python notebook utilizing the datalogger output CSV/JSONL to run K-Means clustering.
12. **"Train a simple classifier to predict engine type from spectral features"**
    - _Prose:_ Collecting spectral telemetry from FM, waveguide, and sine engines, then training an off-the-shelf classifier (`scikit-learn`) to predict active synthesis engines.
    - _Code:_ Python machine learning code block.

### Computational Science Adjacent:

13. **"Sonify a 1D diffusion equation by mapping the field to root + drift"**
    - _Prose:_ Solve a 1D heat diffusion equation over discrete grid points and map field concentration variables to frequency and drift parameters in real-time.
    - _Code:_ Python script implementing numerical solver + parameter mapping.
14. **"Implement Kuramoto oscillator synchronization analysis on AnnealMusic's actual oscillator state"**
    - _Prose:_ Extract live phase data of the partials from the datalogger, compute the order parameter $r(t) = \frac{1}{N}\left|\sum_{j=1}^N e^{i\theta_j}\right|$, and verify phase transitions.
    - _Code:_ Python script doing phase calculations and plotting phase alignment.
15. **"Render hour-long meditative drones for clinical trial pilot studies"**
    - _Prose:_ Generating massive, high-fidelity wav files with deterministic slow parameter drift curves designed for mindfulness clinical trials.
    - _Code:_ Python batch/render script generating high-resolution audio files.

### Advanced Section:

## 4. Doc Site URL, Base Path & Deployment

### Deployment Strategy:

- **Base Domain:** `https://docs.annealmusic.org`
- **Docs Root Path:** `/research/`
- **CD pipeline target:** The existing CI/CD workflow will be configured to build VitePress using `npm run docs:build` (which builds to `docs-site/.vitepress/dist`).
- **Firebase Multisite Target:** We will add a multisite target inside `firebase.json` mapping `annealmusic-docs` to the docs site directory. To keep it completely robust and self-contained (preventing manual `firebase target:apply` blocks on local machines), we will also support copying the docs static assets into `dist/research/docs/` so that it functions seamlessly under both subdomains and single-domain routes.

## 5. CI Notebook Execution Setup

To prevent "documentation rot", where code examples become obsolete as APIs shift, we implement automated testing for our Jupyter/Pyodide examples:

- **Test Script (`scripts/test-notebooks.py`):** A Python script using `nbformat` and `ipykernel` to execute every `.ipynb` file in `examples/notebooks/` headlessly in a virtual environment.
- **CI Action integration:** Add a notebook testing step into `.github/workflows/ci.yml` that sets up Python, installs dependencies, and runs `python scripts/test-notebooks.py`.

## 9. Key Risks & Mitigations

1. **Notebook CI Flakiness:**
   - _Risk:_ Heavy packages (like `pandas`, `scikit-learn`, `numpy`) could fail to install or timeout in simple CI containers.
   - _Mitigation:_ The CI environment will use `uv` for ultra-fast, cached dependency resolution. Notebooks will run with low sample lengths/durations (e.g. 1 second renders) during testing to minimize CPU overhead.
2. **Search Index Maintenance:**
   - _Risk:_ Algolia search requires continuous scraping and api keys which can rot.
   - _Mitigation:_ We will implement VitePress's built-in, local client-side search index. It is extremely fast, works offline, and requires zero external credentials.
3. **Citation Accuracy:**
   - _Risk:_ Hardcoded version numbers in citation strings drift from the actual package version.
   - _Mitigation:_ The citation CLI and Python utilities will dynamically read the version from `package.json` and export it synchronously.
