Status: **diagnosis-first plan. HARD STOP — awaiting explicit "go" before CP1.**

Branch: `feat/v0.9-granular-engine` (the mega-prompt's name; the working branch
for this environment is `claude/determined-feynman-RWdwp`).

Goal: add a third selectable synthesis engine — **granular synthesis from a
curated bank of source buffers** — that slots into the v0.3 `AnnealEngine`
abstraction alongside `sine` and `fm`. Refactor the v0.6 freeze granular code

## 1. Refactor audit of v0.6 `GranularPlayer`

### Current functionality (`src/loop/GranularPlayer.ts`)

One `GranularPlayer` re-synthesizes **one** captured `AudioBuffer` into an
endless drone for a frozen loop slot. It:

1. Owns a `LookaheadScheduler` (`src/loop/scheduler.ts`) and a cached
   `hannWindow(256)` (`src/loop/windows.ts`).
2. `pump(until)` schedules grains while `nextGrainTime < until`; per grain it
   advances `center` by `interval * CENTER_DRIFT_RATE` (0.25) modulo buffer
   duration — a **per-grain center auto-scan** through the buffer.
3. `scheduleGrain(when)`: grain duration = `min(sizeMs/1000, dur*0.9)`; position
   = `center + jitter*dur` where `jitter = (rand*2-1) * effectivePosJitter`;
   creates a `BufferSourceNode` + `GainNode`, applies the Hann curve via
   `setValueCurveAtTime`, `start(when, offset, grainDur)`,
   `stop(when+grainDur+0.02)`, GC's on `onended`.
4. `pitchJitter` → `src.detune.value` random per grain.
5. **Loop-pedal-specific**: `setDriftModulation(meanNorm)` + `driftCoupled` →
   `effectivePosJitter()` widens position jitter by `DRIFT_POS_GAIN*|meanDrift|`.

Consumed by `LoopSlot.freeze()` (`src/loop/LoopSlot.ts:238`):
`new GranularPlayer(ctx, buffer, slotGain)` → `setDriftModulation` →
`start({...grain, driftCoupled})`; live param edits via `setParams`; `stop()` on
unfreeze/clear; `isRunning()`.

### What becomes `GrainCloud` vs. what stays loop-specific

`GrainCloud` is the **mechanism**: scheduler + Hann window + grain triggering
from a source buffer at an explicit cluster `positionCenter`, with `pitchOffset`

- `pitchJitter` and `gain`. It is **policy-free** — it does not know about
  mean-drift coupling or how the center moves on its own.

Two behaviors stay **out** of the core, in consumers:

- **Per-grain center auto-scan** (v0.6) — preserved by a small, documented
  addition `centerDriftRate` to the cloud params (default `0`), applied per grain
  exactly as today (`center += interval * centerDriftRate`). v0.6 wrapper passes
  `0.25`; the v0.9 engine passes `0` and moves the center itself. Keeping the
  scan _inside_ the pump (rather than re-deriving it on a timer in the wrapper)
  is what guarantees byte-identical v0.6 timing. (Deviation justified in §2.)
- **Drift-coupled jitter widening** — stays in the v0.6 wrapper
  (`GranularPlayer`), which keeps `setDriftModulation` + `driftCoupled` and pushes
  an already-widened `positionJitter` into the cloud via `setParams`. The cloud
  never sees `meanDrift`.

So `GranularPlayer` is **kept as a thin wrapper** (not deleted): same public API
(`constructor(ctx, buffer, output, random?)`, `start(GranularStartParams)`,
`setParams`, `setDriftModulation`, `stop()`, `isRunning()`), internally a single
`GrainCloud` plus the two loop-pedal policies. `LoopSlot` is **unchanged**.

### Regression test plan (v0.6 must be identical)

- All existing `src/loop/*.test.ts` pass **unchanged** (`GranularPlayer.test.ts`,
  `scheduler.test.ts`, `windows.test.ts`, `LoopSlot.test.ts`,
  `SeamLoopPlayer.test.ts`).
- Determinism: tests inject `random = () => k`; the wrapper must forward the same
  injected RNG into the cloud so grain positions/pitch are bit-identical.
- Offline-render parity smoke (new, optional): render N seconds of a fixed buffer
  - fixed RNG through old vs. new code paths and assert sample-equal (documented
    in CP1 report; gated behind the existing audio mock).
- Manual: freeze a captured loop, A/B against pre-refactor build — identical.

## 2. Final `GrainCloud` interface

```ts
// src/audio/granular/GrainCloud.ts
export interface GrainCloudParams {
  source: AudioBuffer;
  sizeMs: number; // grain length (clamped to source*0.9)
  density: number; // grains/sec for THIS cloud
  positionJitter: number; // 0..1 of source duration
  pitchJitter: number; // cents, random per grain
  positionCenter: number; // 0..1 cluster center in the source
  pitchOffset: number; // cents, constant detune for this cloud (partial pitch)
  gain: number; // linear output gain for this cloud
  centerDriftRate?: number; // per-grain center auto-scan (×); default 0  [ADDED]
}

export class GrainCloud {
  constructor(ctx: AudioContext, random?: () => number);
  start(params: GrainCloudParams): void; // idempotent-guarded like v0.6
  setParams(partial: Partial<GrainCloudParams>): void; // applies on NEXT grain
  setPitchOffset(cents: number): void; // convenience for detune path
  stop(fadeSeconds?: number): Promise<void>; // fade then teardown, no clicks
  getOutputNode(): AudioNode; // a GainNode (the cloud's gain)
  isRunning(): boolean;
}
```

Deviations from the proposal, justified:

1. **`source` moves into `GrainCloudParams`** (proposal had it there too) but the
   buffer is fixed for the cloud's lifetime; `setParams({source})` triggers a
   clean internal swap (used by the engine's source-change path, §3). Kept in the
   params bag rather than the constructor so the engine can build clouds before a
   source has finished loading and start them when it arrives.
2. **`centerDriftRate` added.** Required to preserve v0.6's exact per-grain center
   scan without leaking loop-pedal policy into the engine. Default `0` means "no
   autonomous scan" — the v0.9 engine drives `positionCenter` itself.
3. **`stop(fadeSeconds?)` returns `Promise<void>`** (proposal already had this) to
   match the orchestrator/engine `stop` contract and to allow a click-free fade
   (offline-tested in CP1).
4. **`random?` constructor arg** — needed for deterministic tests, mirroring
   v0.6.
5. Own output is a `GainNode` so `gain` and stop-fades are AudioParam ramps (no
   click), and the engine can route each cloud into its summing node.

`setParams` applies on the **next scheduled grain**, never retroactively (grains
already scheduled in the look-ahead window keep their parameters) — matches v0.6
and is asserted in CP1 tests.

Shared low-level modules move to the granular home (single source of truth):

- `src/loop/scheduler.ts` → `src/audio/granular/scheduler.ts`
- `src/loop/windows.ts` → `src/audio/granular/windows.ts`

Importers updated: `GranularPlayer`, `SeamLoopPlayer` (uses `equalPowerFadeIn/
Out`), and their tests. `SeamLoopPlayer` will import the equal-power fades from
`@/audio/granular/windows`. (Cross-module import `loop → audio/granular` is
acceptable: `windows`/`scheduler` are low-level audio utilities, and the
heuristic-drift rule wants exactly one home for the windowing math.)

## 3. Final granular engine interface (`GranularEngine implements AnnealEngine`)

- **Partials.** `N = shared.density` clouds over `HARMONICS`, exactly like
  `sine`/`fm`. Partial `i` "frequency" = `rootFreq * HARMONICS[i]^spread`.
  `densityLockedWhilePlaying: true` (structural — one cloud per partial; matches
  sine/fm so the crossfade's partial-count assumption holds).
- **Frequency → grain playback rate.** A grain played at detune `c` cents plays
  the source at rate `2^(c/1200)`. We map partial pitch to `pitchOffset`:

  `pitchOffset_i = 1200 * log2(freq_i / refFreq)`

  where `refFreq = source.fundamentalHz ?? freq_0` (the lowest partial). With a
  tagged fundamental the bank aligns to musical pitch; untagged (field
  recordings) → the lowest partial plays at native rate and higher partials are
  offset by the harmonic ratio. Sources carry an **optional** `fundamentalHz`
  (§4). At max `spread`, the 8th partial (ratio 6) is ~+31 semitones → playback
  rate ~6×; clamped so grain duration in source samples stays valid.

- **Drift detune.** `setPartialDetune(i, cents)` → cloud `i`
  `setPitchOffset(pitchOffset_i + cents)`. Same path sine/fm use; the orchestrator
  drift loop is engine-agnostic and needs no change. Detune is smoothed by
  re-reading on the next grain (grain granularity ≈ 1/density s ≤ 250 ms), which
  is smooth enough for the slow drift (±60 cents over seconds).
- **Per-partial vs. global params.** `sizeMs`, `density`, `positionJitter`,
  `pitchJitter`, `positionCenter` are **shared across the bank** (one value, fan
  out to every cloud via `setParams`) — matches the prompt. `sourceId` is global.
  `gain` is per-cloud, set from the same `partialShape(i)` baseline the sine/fm
  engines use, so loudness across partials matches the other engines.
  **`positionCenter` autonomous drift**: the engine runs a clamped random walk
  (≈ `0.01/s`, step on the existing 50 ms-ish cadence) on `positionCenter` and
  fans the result to all clouds, so a static patch still moves. Clouds'
  `centerDriftRate` stays `0` (the engine owns center motion).
- **Source change mid-session.** `setEngineParams({source})` (numeric index, §9)
  → engine loads the buffer (lazy, §5) → `cloud.setParams({source})` on every
  cloud. Old grains finish their Hann tails; new grains read the new buffer →
  no hard cut (verified CP4).
- **Grain count / CPU sanity.** Total simultaneous grains ≈
  `partials × density × (sizeMs/1000)`. At the prompt's worst case
  `8 × 40 × 0.2 = 64` overlapping grains → ~128 live nodes (BufferSource+Gain),
  with **320 grains/sec created+destroyed** — the churn, not the steady count, is
  the real cost. Defaults are chosen well under this (§10). Per-partial `density`
  default `14`; at default `density=6` partials that's ~84 grains/sec created,
  ~17 peak overlaps — comfortable.

New optional capability (proposed interface extension):

```ts
interface AnnealEngineCapabilities {
  densityLockedWhilePlaying: boolean;
  params: readonly EngineParamDef[];
  crossfadeMs?: number; // [ADDED] preferred incoming-crossfade window
}
```

Granular returns `crossfadeMs: 800`. The orchestrator reads
`engineCapabilities(incoming).crossfadeMs ?? CROSSFADE_MS` for the **incoming**
ramp + swap timer; `sine`/`fm` omit it → unchanged 600 ms. Backward compatible
(optional field). (§ CP3.5.)

## 4. Source bank decision

**Licensing path: (c) generate every source algorithmically (synthesis).**
Rationale: it is the only path I can fully stand behind in this environment —
no third-party downloads to vet, no attribution I can't verify, reproducible,
small, and it matches the app's identity (it _is_ a synthesis app). Every source
is **original work, released CC0**. A committed, deterministic generator
(`scripts/gen-sources.ts`, run offline) renders each source via
`OfflineAudioContext` → WAV → `ffmpeg -c:a libopus -b:a 96k` → `.opus`. ffmpeg/
opus is an **asset-build** dependency only, never a runtime/CI dependency; the
`.opus` files are committed. The registry's `license` field is freeform so
future CC0/CC-BY-SA/commissioned sources drop in without code change.

Proposed bank (8 sources; all CC0, original/synthesized):

## 5. Asset loading

- **Lazy.** Nothing loads until granular is selected **and** a source is picked.
  Selecting the engine pre-warms the default source (index 0) only.
- **Loader** (`src/audio/sources/loader.ts`): `loadSource(ctx, id) →
Promise<AudioBuffer>`. `fetch('/sources/<id>.opus')` →
  `ctx.decodeAudioData`. In-memory cache `Map<id, Promise<AudioBuffer>>` keyed by
  id (per session; dedupes concurrent requests; second request returns cached).
  Fetch/decode failure → reject; UI shows an inline error and keeps the prior
  source playing (no silent field, no crash).
- **Loading-state UX.** Source picker tiles show a skeleton/spinner on the
  selected tile while its buffer loads. Engine audio keeps playing the previous
  source through the load; on arrival the engine swaps the buffer into the clouds
  (§3) — the field never stutters.
- **Caching.** Browser HTTP cache via Firebase Hosting headers — add a
  `/sources/**` rule: `Cache-Control: public, max-age=31536000, immutable`
  (filenames are stable, content-addressed by id; safe to pin). No Service Worker
  in v0.9 (the in-memory map + HTTP cache are sufficient; SW deferred).

## 6. Server-side preview rendering compatibility

The renderer is **Option B** (Playwright/Chromium, `renderer_playwright.py`)
driving `render.html` + `src/render/headless.ts`, which builds the **real**
`Orchestrator` and records the analyser via `MediaRecorder`. It is the production
runtime, so granular DSP is identical to the client.

**Decision: keep Option B; sources fetched from the same origin — no renderer
code change.** The harness page is served from the Vite/host origin
(`render_harness_url`), and `public/sources/*` is served on that same origin, so
the granular engine's `fetch('/sources/<id>.opus')` resolves inside Chromium
exactly as in the client. The only requirements:

1. The render harness origin serves `public/sources/` (it does — Vite serves
   `public/`, and a built `dist/sources/` is served by `vite preview`/Hosting).
2. Decode happens via the real `AudioContext` already used by the orchestrator.
3. `headless.ts` already hydrates engine + engine-params from the payload; once
   `gr.source` decodes to a source index and the engine lazy-loads it, the
   preview renders the chosen source with no harness change.

**Tests:** extend `api/tests/test_preview.py` / `test_render.py` with a granular
payload (`e=granular&gr.source=0&...`) and assert a non-empty preview is produced
(the existing preview tests stub/exercise the render path). End-to-end manual:
publish a granular patch → preview renders → preview plays (CP4 smoke matrix).

Risk: Chromium Opus decode of our generated files — covered by the COMPAT note in
CP4; generator targets standard Ogg/Opus 96 kbps which Chromium decodes natively.

## 9. URL schema v5 — exact keys, defaults, bounds

`SCHEMA_VERSION = 5`; `SUPPORTED_SCHEMA_VERSIONS = [1,2,3,4,5]`. Older schemas
load unchanged (v1–v4 paths untouched). New keys, **only meaningful when
`e=granular`**:

| URL key          | meaning                                        | type | default | bounds                  |
| ---------------- | ---------------------------------------------- | ---- | ------- | ----------------------- |
| `gr.source`      | source **index** into the append-only registry | int  | `0`     | `0 .. SOURCES.length-1` |
| `gr.size`        | grain size (ms)                                | int  | `120`   | `30 .. 300`             |
| `gr.density`     | grains/sec per partial                         | int  | `14`    | `4 .. 40`               |
| `gr.posJitter`   | position jitter                                | 2-dp | `0.30`  | `0 .. 1`                |
| `gr.pitchJitter` | per-grain pitch jitter (cents)                 | int  | `0`     | `0 .. 100`              |
| `gr.posCenter`   | position center                                | 2-dp | `0.50`  | `0 .. 1`                |

Two schema mechanics this requires:

1. **Per-engine URL namespace (`urlNs`).** Engine id stays `granular` (so
   `e=granular`), but its params serialize under `gr` to honor the prompt. Add an
   optional `urlNs` to the engine (default = id; sine→`sine`, fm→`fm`,
   granular→`gr`). `encodeEngineParams` uses `urlNs`; `decodeState` resolves a
   parsed namespace back to an engine id via a reverse map. `gen-manifest.ts`
   keys the `engines` map by `urlNs`, so the **server validator needs no logic
   change** for the numeric grain params — it already validates `engines[ns]`.
2. **`gr.source` is the one non-numeric-fitting field.** 🔶 **Decision: encode it
   as a stable integer index, not a string id.** This keeps `EngineParams =
Record<string, number>` intact — _the v0.3 abstraction holds with no change to
   its type model._ The engine maps index → registry id internally. Cost: the
   registry becomes **append-only / never-reordered** (a documented invariant,
   same discipline as schema versions), so old share links keep resolving. The
   alternative — a string `gr.source=glasspad` — would force a string into the
   numeric param bag, the encoder, the store, the manifest, and the Python
   validator's numeric path; that is a real abstraction change and is **not**
   taken. (See §11.) Even as an index, `gr.source` is validated as an enum/int
   bound `[0, len-1]` in both client decode and `validation.py` (add a `sources`
   list to the manifest for the count).

Round-trip is covered by new tests in `src/share/encode.test.ts` (granular params

- older-schema back-compat) and the schema-contract CI job (regenerate
  `manifest.v5.json`, assert no diff).

## 10. Performance budget

- **Target:** 8 partials × 20 grains/s on a 5-year-old laptop, glitch-free.
  That's 160 grains/s created, ~32 peak overlaps at 200 ms — well within Web
  Audio's comfortable range on such hardware.
- **Defaults** chosen below target: per-partial density `14`, size `120 ms`, at
  the app default `density = 6` partials → ~84 grains/s, ~17 peak overlaps.
- **Worst case** (8 × 40 × 300 ms) ≈ 96 overlaps / 320 grains/s — allowed but
  above the comfort target.
- 🔶 **Fallback strategy: trust the user, with a soft cap, not a hard device
  lock.** Justification: a hard "device-limited" gate needs reliable device
  capability detection (there isn't one) and risks silently neutering the
  instrument. Instead: (a) keep defaults conservative; (b) the engine enforces a
  **total simultaneous-grain ceiling** (skip scheduling a grain if the live count
  exceeds a cap, e.g. ~120) so an extreme patch degrades gracefully (sparser
  texture) instead of glitching/crashing — this is invisible at normal settings;
  (c) reuse the existing low-`deviceMemory` console warning pattern
  (`orchestrator.ts:313`) but do not block. Measured on the target laptop class
  in CP4; if it exceeds budget, lower the default density and document.

## 11. Risks & tradeoffs — is the v0.3 abstraction holding?

**Mostly yes — it holds at the structural level, and strains at exactly one
point: a non-numeric param.**

- ✅ **Lifecycle, output routing, detune, partial count, crossfade** all fit with
  zero change. Granular is "N voices over the harmonic lattice, each consuming
  the drift detune" — the same shape as sine/fm. The drift loop, orchestrator
  crossfade, and visualizer (`getPartialFrequencies`) work unmodified.
- 🔶 **`EngineParams = Record<string, number>` cannot express `sourceId`.** This
  is the one real strain and the most important decision in this plan. Rather
  than widen the param model to `number | string` (which ripples through encode/
  decode, the store, the manifest, and the Python validator), I encode `source`
  as a **stable integer index** into an **append-only registry**. The abstraction
  holds unchanged; the cost is the registry-ordering invariant. **This is the #1
  item to confirm before CP1.**
- 🔶 **Namespace ≠ id (`gr` vs `granular`).** Honoring the prompt's `gr.*` keys
  needs a small `urlNs` generalization (default = id). Minor, additive, and
  keeps sine/fm URLs byte-identical. Alternative: use `granular.*` and deviate
  from the prompt — rejected (URLs are a compatibility contract; honor the spec).
- 🔶 **`crossfadeMs` capability.** A new optional field so granular can request an
  800 ms swap. Additive, backward compatible. Without it, granular start-up
  jitter might be audible under the 600 ms window.
- 🟢 **`centerDriftRate` on `GrainCloud`.** Additive param to preserve exact v0.6
  timing; default 0 means engine consumers are unaffected.
- **Schema/validator blast radius.** v5 touches TS schema, the generated
  manifest, and `api/app/validation.py` (+ its tests). Mitigated by the existing
  schema-contract CI job, which will fail loudly on any drift.
- **Asset licensing.** Synthesized CC0 removes all third-party legal risk and is
  CI-enforced (every registry entry must have a non-empty `license`); the
  tradeoff is "synthetic, not field-recorded" sources — acceptable for v0.9, and
  the registry is built to accept licensed/recorded sources later.
- **CPU churn** is the real performance risk (grain node allocation), addressed by
  conservative defaults + a soft grain ceiling (§10).

### Flags / optional scope (per the prompt's "flag if you want it")

- **Multi-source granular** (different source per partial): _not_ wanted for v0.9;
  the index-based `gr.source` is global. Easy to add later (per-cloud source).
- **Reverse playback grains:** not wanted for v0.9; trivial future `playbackRate
< 0` extension on the cloud.

   vs. licensed Freesound CC0/CC-BY-SA recordings (more "real," needs external
