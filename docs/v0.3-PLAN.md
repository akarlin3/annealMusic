### Tangles flagged

1. **`setParams` mixes two concerns.** `rootFreq`/`spread` are voice-frequency
   params (engine-owned post-refactor); `brightness`/`space`/`volume` are post-fx
   (orchestrator). They're handled in one method today. → Split across
   `Orchestrator.setSharedParams` (post-fx + fan-out) and
   `engine.setSharedParams` (voice freqs only).
2. **Partials connect directly into `filter`** (line 108). The engine must not
   know about the post-fx filter. → Engine exposes one `getOutputNode()`; the
   orchestrator wires `engineOut → crossfadeGain → filter`.
3. **Global fade lives on `master`.** With crossfade, amplitude fades must be
   **per-engine** (two engines alive at once). → Move fade automation to a
   per-engine bus gain owned by the orchestrator; `master` becomes a static mix
   bus. (See §4.)
4. **Drift reads `osc.detune` directly** and stores `detune` on the live
   `PartialVoice`. → Orchestrator keeps its own pure drift state array (count
   from `engine.getPartialCount()`); engine applies cents in `setPartialDetune`.
   Note: `driftStep` only reads `detune` and the mean — it **never reads
   `ratio`** — so drift is already structurally engine-agnostic.

## 2. Final engine interface — refined

`src/audio/engines/types.ts`:

```ts
export type EngineId = 'sine' | 'fm';

/** Shared params == the existing AnnealMusicParams (rootFreq…volume). */
export type SharedParams = AnnealMusicParams;

/** Engine-specific param bag. Flat number map; keys validated via capabilities. */
export type EngineParams = Record<string, number>;

export interface EngineParamDef {
  readonly key: string;
  readonly label: string;
  readonly min: number;
  readonly max: number;
  readonly step: number;
  readonly default: number;
  readonly fmt: (v: number) => string;
}

export interface AnnealEngineCapabilities {
  readonly densityLockedWhilePlaying: boolean;
  readonly params: readonly EngineParamDef[];
}

export interface AnnealEngine {
  readonly id: EngineId;
  readonly capabilities: AnnealEngineCapabilities;

  /** Build internal nodes + start sources. Throws if called while started. */
  start(ctx: AudioContext, shared: SharedParams, engine: EngineParams): void;

  /** Ramp own output to 0 over fadeSeconds, then stop + dispose. */
  stop(fadeSeconds?: number): Promise<void>;

  /** Single node the orchestrator routes into post-fx. */
  getOutputNode(): AudioNode;

  setSharedParams(partial: Partial<SharedParams>): void;
  setEngineParams(partial: Partial<EngineParams>): void;
  setPartialDetune(index: number, cents: number): void;
  getPartialCount(): number;

  /** Current per-partial fundamental frequencies (Hz) — drives the visualizer. */
  getPartialFrequencies(): number[];
}
```

**Changes from the proposal & justifications:**

- **Added `getPartialFrequencies(): number[]`.** Non-negotiable: `Visualizer.tsx`
  (line 59) reads it today to align the spectrum overlay to the live partial
  pitches. Without it on the interface the visualizer breaks on FM. It's
  engine-knowledge (carrier freq for FM), so it belongs on the engine; the
  orchestrator surfaces it by delegating to the active (incoming) engine.
- **`EngineParams = Record<string, number>`** rather than a generic. Keeps the
  store and URL layer simple, every engine param is a scalar slider, and it
  avoids `as` casts on the interface (the rule). Bounds/defaults live in
  `capabilities.params`, which is the single source of truth.
- **`stop(fadeSeconds?)` clarified:** ramps the engine's **own** output node to 0
  over `fadeSeconds` (default ~0.02s safety ramp to avoid an osc-stop click),
  then stops sources and disposes. The **audible** crossfade/settle fade is the
  orchestrator's bus gain (§4); `stop`'s ramp is just click-avoidance. This
  resolves the "who owns the fade" ambiguity: **orchestrator owns audible
  fades, engine owns click-safe teardown.**
- Kept everything else from the proposal verbatim.

## 4. Crossfade design — exact node graph

Static post-fx chain (built once at orchestrator boot, lives until stop):

```
[engine bus] → filter(lowpass) → dryGain ─┐
                       └→ convolver → wetGain ─┴→ master → masterVol → analyser → destination
```

Each **active engine** is connected through its own orchestrator-owned bus gain:

```
engineA.getOutputNode() → busA(gain) ┐
engineB.getOutputNode() → busB(gain) ┴→ filter → …(post-fx)…
```

**First start** (no previous engine): build engine, create `busA` at gain 0,
connect, `engine.start(...)`, ramp `busA: 0 → 1` over **3.0s** (preserves v0.2's
"bloom" fade-in — verified identical behavior).

**Swap (A → B)** over `CROSSFADE_SECONDS = 0.6`:

1. If `targetId === activeId` → **no-op** (return; no rebuild, no glitch). See §8.
2. Build `engineB` from `ENGINES[targetId]()`; create `busB` at gain 0; connect
   `engineB.out → busB → filter`; `engineB.start(ctx, shared, engineParamsForB)`.
3. Seed B's detune from the current orchestrator drift state (so the texture's
   detune is continuous across the swap), then `setPartialDetune` for each.
4. At `t0 = ctx.currentTime`: `busA.gain.setValueAtTime(1, t0)` then
   `linearRampToValueAtTime(0, t0 + 0.6)`; `busB.gain.setValueAtTime(0, t0)` then
   `linearRampToValueAtTime(1, t0 + 0.6)`. Both engines audible during the window.
5. After 600ms (timer): `await engineA.stop(0.02)`; disconnect + drop `busA`.
   `engineB`/`busB` become active.

**Click avoidance:** (a) `linearRamp` from an explicit `setValueAtTime` anchor —
no implicit jumps; (b) equal-power is unnecessary here because both engines share
the per-partial amplitude shape and a 600ms linear sum stays well under unity
clip on the lattice (verified by the no-clip test, CP2); (c) the engine's own
`stop(0.02)` ramp prevents a tail click when oscillators halt; (d) swap-to-same
is a no-op so there's never a rebuild discontinuity.

**Why bus gains, not `master`:** two engines must fade independently and
simultaneously; a single `master` can't express that.

## 5. FM param set (incl. feedback — in scope for v0.3)

Topology (per partial): `carrier` sine at `rootFreq·ratio^spread`; `modulator`
sine at `carrier × modRatio`; `modGain = modIndex × carrierFreq` (Hz) →
`carrier.frequency` (additive FM). **Feedback:** `modulator → fbGain → modulator.frequency`,
`fbGain = feedback × modFreq × FB_SCALE`.

| param | range | step | default | rationale |
| ----- | ----- | ---- | ------- | --------- |
| `modRatio` | 0.5 – 4 | 0.01 | **1.0** | 1.0 = classic harmonic FM (richest, most musical default); sub-1 gives sub-harmonic shimmer, up to 4 covers bright metallic spectra without aliasing at our 55–220 Hz roots. |
| `modIndex` | 0 – 10 | 0.05 | **2.0** | 0 = pure sine (graceful "FM that sounds like sine"); ~2 is a warm, present default; 10 is very bright but bounded so the spectrum stays controllable. |
| `feedback` | 0 – 1 | 0.01 | **0.0** | 0 default = identical to no-feedback FM (safe baseline). Mapped through `FB_SCALE` (≈ a few × modFreq, clamped) so 1.0 reaches the chaotic/noisy edge without runaway. Defaulting to 0 keeps first-listen predictable. |

`FB_SCALE` chosen conservatively so `feedback=1` approaches but doesn't blow past
the classic single-operator feedback "sawtooth→noise" threshold; exact constant
tuned by ear in CP2 and documented inline.

Detune updates apply to the **carrier**; the modulator frequency is recomputed as
`carrierFreq × modRatio` so the ratio (and thus timbre) tracks pitch.

## 6. State + UI plan

**Store (`src/state/params.ts`)** — add, alongside `params`:

```ts
engineId: EngineId;                              // default 'sine'
engineParams: Record<EngineId, EngineParams>;    // per-engine, retained on switch
setEngine: (id: EngineId) => void;
setEngineParam: (id: EngineId, key: string, value: number) => void;
```

`engineParams` is initialized from each engine's `capabilities.params` defaults
(`{ sine: {}, fm: { modRatio: 1, modIndex: 2, feedback: 0 } }`). Switching back to
a previously-used engine restores its retained params (requirement met). Engine
param writes clamp to the engine's declared bounds.

**Density lock via capability:** today density's lock is hard-coded
(`CONTROL_DEFS … lockWhilePlaying: true`). v0.3: the control panel reads the
**active engine's** `capabilities.densityLockedWhilePlaying`. Both sine and FM are
structural (`true`), so behavior is identical to v0.2 — but the wiring is now
capability-driven for future engines (e.g. a granular engine that can re-grain
live could set `false`).

**Engine selector UI** — segmented control directly under the header, above the
visualizer. Two segments `SINE` / `FM`, mono caps (`tracking-[0.2em]`, same warm
palette). Active segment marked with a filled warm-amber background + brighter
text; inactive are muted with `1px solid #44403c`. Keyboard a11y:
`role="radiogroup"` / `role="radio"`, `aria-checked`, arrow-key navigation,
`Enter`/`Space` to select; visible focus ring. Click/select → `setEngine(id)` →
orchestrator crossfade (only while playing; when stopped it just updates the
store and the next Begin starts on that engine).

**Engine param panel** — fourth control group `ENGINE`. Heading shows the active
engine name in subdued caps; renders a `Slider` per `capabilities.params` entry,
driven by `engineParams[engineId]`. **Sine (no params): the group hides entirely**
(decision — chosen over an empty-state string, since an empty group is visual
noise and sine is the common case; the grid simply shows three groups). The group
mounts/unmounts on switch with a short opacity/height transition to avoid a layout
jump. Engine params are **not** locked while playing (live-sculptable).

## 7. URL schema v2 spec

- `SCHEMA_VERSION = 2`.
- Payload adds `e=<id>` and namespaced engine params `fm.modRatio`,
  `fm.modIndex`, `fm.feedback`. Shared params unchanged. Example:

  ```
  #s=2:e=fm&rootFreq=110&spread=1.00&density=6&coupling=0.30&drift=0.50&brightness=0.50&space=0.40&fm.modRatio=1.00&fm.modIndex=2.00&fm.feedback=0.00
  ```

- **Decode dispatch by version** (today `readStateFromHash` returns `null` for any
  version ≠ 1; this is rewritten to accept 1 **and** 2):
  - `s=1:` → decode shared params; `engineId = 'sine'`; engine params = sine
    defaults (none). **v1 URLs keep working** (requirement).
  - `s=2:` → decode shared; read `e` (missing/unknown → `'sine'` + warning);
    decode each `fm.*` namespaced param (clamped to the engine's bounds, unknown
    namespaced keys → warning).
  - version > 2 → `null` (existing "link from a newer version" toast).
- Encode always writes v2: shared params + `e=<active>` + the active engine's
  namespaced params. (Engines with no params contribute only `e=`.)
- `decodeParams` return type grows to
  `{ params, engineId, engineParams, warnings }`. `CopyLinkButton` /
  `buildShareUrl` / `App` hydration updated accordingly. Volume still never
  encoded.

## 8. Migration

1. Extract post-fx + lifecycle + drift loop from `AnnealMusicEngine` →
   `Orchestrator`. The orchestrator exposes the **same external surface the React
   layer uses today**: `start()`, `stop()`, `isRunning()`, `getAnalyser()`,
   `getPartialFrequencies()`, `setSharedParams()` (renamed from `setParams`), plus
   new `setEngine(id)`, `setEngineParams(partial)`. `useAudioEngine` /
   `useAnnealMusic` / `Visualizer` swap `AnnealMusicEngine` → `Orchestrator` (the
   ref type changes; method names mostly identical — `setParams` → `setSharedParams`
   is the one rename, contained to the hook).
2. Move sine-bank construction → `SineEngine`. Behavior byte-for-byte identical
   (same node graph, same constants, same `setTargetAtTime` time-constants).
3. `ENGINES = { sine: () => new SineEngine() }` in CP1; add `fm` in CP2.
4. **All existing v0.2 tests pass unchanged** (drift, share, params, smoke — none
   touch the engine class directly, so they're unaffected by the refactor; the
   share tests change only in CP3 when schema bumps to v2).
5. **Same-engine swap = no-op** (documented choice): `setEngine('sine')` while sine
   is active returns early. CP1 test asserts this (no rebuild). Rationale: a
   rebuild would cause a needless 600ms dip for zero user benefit.
6. **Rapid-switch guard = coalescing latest-wins queue** (documented choice):
   while a crossfade is in flight, a new request stores only the latest target;
   on completion, if it differs from the now-active engine, run one more
   crossfade. Intermediate targets are dropped. Simpler than a full FIFO and
   matches user intent (they want the last thing they clicked).

## 9. Risks — does the interface hold for granular (v0.9) / physical (v1.0)?

- **Granular needs sample buffers.** `start(ctx, shared, engineParams)` gives no
  hook to inject an `AudioBuffer`. → Acceptable to **defer**: a granular engine
  can own its buffer loading internally (fetch/record) and expose load state via
  engine params or a later interface extension. The shared-param + output-node +
  detune contract still holds (grains can be detuned per "partial"). **No interface
  change now**; documented as a known future extension point (likely an optional
  `loadBuffer(buf)` method added non-breakingly).
- **Physical modeling needs IIR/feedback delay banks.** These are just internal
  nodes behind `getOutputNode()`. `setPartialDetune` maps to delay-line
  length/pitch. The interface holds. The one tension: physical models may want
  **excitation events** (pluck/strike) rather than continuous drones —
  that's a new method (`excite()`), addable non-breakingly later. Safe to defer.
- **`getPartialCount` assumes a fixed partial model.** Granular (grain clouds) may
  not map cleanly to "N partials." Drift still works if the engine reports a
  sensible count (e.g. number of detunable streams). If an engine has no
  per-partial detune concept, it can report `0` and ignore `setPartialDetune`
  (drift loop already no-ops on count 0). **Holds.**
- **FM feedback (now in scope):** the only added risk is **instability/clipping**
  at high `feedback`. Mitigation: bounded `FB_SCALE`, `feedback` default 0, and a
  CP2 no-clip assertion. Flagged but contained.

**Conclusion:** no interface changes needed now; two future non-breaking
extension points identified (`loadBuffer`, `excite`). Documented rather than
pre-built (YAGNI — building them now would be speculative).

## 10. Tradeoffs flagged

1. **`EngineParams = Record<string, number>`** (untyped keys) vs per-engine typed
   param interfaces. Chose the map for simplicity + no interface casts + trivial
   URL/store round-trip; bounds live in `capabilities.params`. Cost: a typo in an
   engine param key isn't a compile error (mitigated by tests asserting each
   engine's params match its capabilities).
2. **Orchestrator owns audible fades; engine `stop` only does a click-safe ramp.**
   Slightly diverges from the proposal's implied "engine fades itself out," but
   it's the only coherent way to crossfade two engines. Documented in §2.
3. **Sine engine-param group hides entirely** (vs empty-state text). Less
   discoverable that sine "has no params," but cleaner. Reversible.
4. **No-op on same-engine swap** + **latest-wins coalescing**: simplest correct
   behavior; a full queue is overkill for a 2-segment control.
5. **Testing Web Audio in jsdom (significant — see below).**

### ⚠️ Testing constraint (must decide before CP1/CP2)

jsdom implements **no Web Audio API** — no `AudioContext`, no
`OfflineAudioContext`. The current engine has therefore **never** been unit-tested
with real audio (only `drift.ts` pure math + share + params are tested). This
directly affects the spec's CP2 ask for `OfflineAudioContext` FFT-render
assertions, which **cannot run as written**.

**Proposed approach (no new runtime deps):** add a small hand-rolled
`MockAudioContext` test helper (`src/test/audioMock.ts`) that records node
creation, connections, and `AudioParam` automation calls (`setValueAtTime`,
`linearRampToValueAtTime`, `setTargetAtTime`). Then:

- Orchestrator/engine lifecycle, swap-to-same no-op, drift→`setPartialDetune`
  fan-out, and FM frequency tracking → asserted **structurally** against the mock
  (e.g. "carrier `.frequency` target == rootFreq·ratio^spread", "modulator freq ==
  carrier×modRatio", "busA ramps 1→0 and busB ramps 0→1 over 0.6s, overlapping").
- The spec's "both engines audible / no zero-gap / no clipping" becomes an
  assertion over the **recorded gain ramp envelopes** (sum of busA+busB stays in
  (0, ≤~1] across the window), not a real FFT.

**Alternative:** add a dev-dep mock library (e.g.
`standardized-audio-context-mock`) or run a real-audio smoke in a headless
browser (Playwright). Heavier; I'd avoid for v0.3.

**Need your call** on: hand-rolled mock (recommended) vs. dev-dep mock vs.
browser-based audio test. I'll default to the hand-rolled mock unless told
otherwise.
