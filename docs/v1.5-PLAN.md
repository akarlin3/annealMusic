# Implementation Plan — v1.5 DAW Export / Stems

This implementation plan details the technical architecture, specifications, file structures, render pipeline, and user experience for rendering and exporting high-fidelity multi-stem sessions in **AnnealMusic v1.5**.

---

## 1. Stem Set Finalization

Each stem is a discrete, sample-accurate mono or stereo WAV file representing one voice in the texture for the full duration of the render, time-aligned to $t=0$.

### The 13-Stem Standard Set

The maximum set consists of up to 13 stems + additional opt-in per-partial stems:

| Stem Name   | Format | Source Node (Tap Point)                                   | Post-FX Chain Processed? | Description                                                                        |
| :---------- | :----- | :-------------------------------------------------------- | :----------------------- | :--------------------------------------------------------------------------------- |
| `engine`    | Stereo | Active Engine Output Node (`engine.getOutputNode()`)      | No                       | Raw synthesis engine mix (pre-filter/pre-reverb).                                  |
| `engine-fx` | Stereo | Post-FX Master Volume Output (`masterVol`)                | Yes                      | Active engine output routed through filter + reverb.                               |
| `input`     | Mono   | Input processing chain tap (`inputVoice.getCaptureTap()`) | No                       | Processed input voice (highpass, compressor, shaper, drift filter) before post-fx. |
| `input-fx`  | Stereo | Post-FX Master Volume Output (`masterVol`)                | Yes                      | Input voice output (`monitorGain`) routed through filter + reverb.                 |
| `loop-A`    | Mono   | Loop Slot A gain output node (`loopSlots.A.slotGain`)     | No                       | Raw Loop A playback, post-grain modulation, before post-fx.                        |
| `loop-A-fx` | Stereo | Post-FX Master Volume Output (`masterVol`)                | Yes                      | Loop A output routed through filter + reverb.                                      |
| `loop-B`    | Mono   | Loop Slot B gain output node (`loopSlots.B.slotGain`)     | No                       | Raw Loop B playback, before post-fx.                                               |
| `loop-B-fx` | Stereo | Post-FX Master Volume Output (`masterVol`)                | Yes                      | Loop B output routed through filter + reverb.                                      |
| `loop-C`    | Mono   | Loop Slot C gain output node (`loopSlots.C.slotGain`)     | No                       | Raw Loop C playback, before post-fx.                                               |
| `loop-C-fx` | Stereo | Post-FX Master Volume Output (`masterVol`)                | Yes                      | Loop C output routed through filter + reverb.                                      |
| `master`    | Stereo | Post-FX Master Volume Output (`masterVol`)                | Yes                      | Complete session mix (engine + input + all active loops).                          |

### Filtering / Skipping Empty Stems

- **Rules of Engagement**: If a stem source has never been connected or captured during a session (e.g. `inputVoice` has `connected = false` or a `loopSlot` has `state === 'empty'`), that stem and its corresponding `-fx` variant are skipped entirely.
- **Master Stem Rule**: The `master` stem is always rendered.
- **Per-Partial Engine Stems (Opt-In)**: Up to 16 additional mono files. Available only for engines implementing `getPartialOutputs(): AudioNode[]` (specifically `sine` and `fm`). Off by default.

## 2. Per-Engine Stem Isolation & Offline Compatibility

### Extended Interface: `AnnealEngine`

We extend the engine interface in `src/audio/engines/types.ts` with the optional `getPartialOutputs` capability:

```typescript
export interface AnnealEngine {
  // ... existing fields ...
  /** [Optional] Expose individual partial output nodes for per-partial stem capture. */
  getPartialOutputs?(): AudioNode[];
}
```

### Engine Isolation Specifications

- **Sine Engine (`sine`)**: Exposes `getPartialOutputs(): AudioNode[]` returning the array of oscillators' gain nodes (`this.partials.map(p => p.g)`). Cleanly isolatable.
- **FM Engine (`fm`)**: Exposes `getPartialOutputs(): AudioNode[]` returning the carrier gain nodes (`this.voices.map(v => v.g)`). Cleanly isolatable.
- **Granular Engine (`granular`)**: Exposes `getPartialOutputs(): AudioNode[]` returning `this.partials.map(p => p.cloud.getOutputNode())`. Runs cleanly in `OfflineAudioContext`.
- **Physical Engine (`physical`)**: Exposes `getPartialOutputs(): AudioNode[]` returning `this.partials.map(p => p.node?.node).filter(Boolean)`.

### `OfflineAudioContext` Compatibility & AudioWorklets

- **AudioWorklets in OAC**: On modern Web Audio implementations, worklets register and run flawlessly inside an `OfflineAudioContext`.
- **Lazy Registration**: We leverage `registerPhysicalModules(ctx)` on the OAC instance before starting the physical engine offline.
- **Supported Sample Rates**: We support standard sample rates: 44.1kHz, 48kHz (default), and 96kHz.

## 3. Determinism Strategy

To ensure identical byte-level WAV renders across consecutive renders of the same patch, we seed all random sources during offline renders.

### The Seeded PRNG

We implement a high-performance **Mulberry32** pseudo-random number generator to replace `Math.random` in offline mode:

```typescript
export function createPRNG(seed: number): () => number {
  return () => {
    let t = (seed += 0x6d2b79f5);
    t = Math.imul(t ^ (t >>> 15), t | 1);
    t ^= t + Math.imul(t ^ (t >>> 7), t | 61);
    return ((t ^ (t >>> 14)) >>> 0) / 4294967296;
  };
}
```

### Randomness Sources & Seeding

1. **Lattice Drift Loop**: The Ornstein–Uhlenbeck detune walk (`driftStep`) utilizes the seeded PRNG at every checkpoint.
2. **Granular Drift**: Autonomous `posCenter` walks and grain parameters in the `GranularEngine` utilize the seeded PRNG.
3. **Seed Persistence**: The seed (a 32-bit integer generated at click-time) is recorded in `session.json` to allow exact replication of the session by re-rendering.
4. **Realtime Capture Mode**: Documented as inherently non-deterministic due to live microphone input and manual loop capture times.

## 4. WAV Writer Specification

We implement a high-efficiency, zero-dependency `WavEncoder.ts` supporting standard WAV headers, BWF metadata, and iXML semantic chunks.

### WAV Header & Formatting

- **PCM 24-bit Mode**: Format tag `1` (Integer PCM). Three little-endian bytes per sample:
  ```typescript
  const intVal = Math.max(-8388608, Math.min(8388607, floatSample * 8388607));
  view.setUint8(offset, intVal & 0xff);
  view.setUint8(offset + 1, (intVal >> 8) & 0xff);
  view.setUint8(offset + 2, (intVal >> 16) & 0xff);
  ```
- **IEEE Float 32-bit Mode**: Format tag `3` (IEEE Float). Little-endian 32-bit floats.
- **Size Safety**: WAV's standard 32-bit chunk size cap is 4GB. Our strict 30-minute render limit at 96kHz 32-bit float stereo yields a maximum size of ~1.4GB, well below the limit.

### BWF `bext` Chunk Layout

The Broadcast Audio Extension (`bext`) chunk is written before the `data` chunk:

```
'bext' [ChunkSize]
  Description: 256 bytes (padded ASCII)
  Originator: 32 bytes (padded ASCII: "AnnealMusic v1.5.0")
  OriginatorReference: 32 bytes (padded ASCII: patch short hash / title)
  OriginationDate: 10 bytes (YYYY-MM-DD)
  OriginationTime: 8 bytes (HH-MM-SS)
  TimeReference: 8 bytes (0, all stems aligned to t=0)
  Version: 2 bytes (0)
  Reserved: 254 bytes (null-padded)
  CodingHistory: Variable length ASCII (empty)
```

### iXML Chunk Format

The `iXML` chunk contains a descriptive XML string:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<IXML>
  <PROJECT>AnnealMusic Session</PROJECT>
  <SPEED>
    <FILE_SAMPLE_RATE>48000</FILE_SAMPLE_RATE>
    <AUDIO_BIT_DEPTH>24</AUDIO_BIT_DEPTH>
  </SPEED>
  <TRACK_LIST>
    <TRACK>
      <CHANNEL_INDEX>1</CHANNEL_INDEX>
      <INTERLEAVE_INDEX>1</INTERLEAVE_INDEX>
      <NAME>engine</NAME>
      <FUNCTION>synthesis</FUNCTION>
    </TRACK>
  </TRACK_LIST>
  <USER>
    <ENGINE_TYPE>fm</ENGINE_TYPE>
    <STEM_NAME>engine</STEM_NAME>
  </USER>
</IXML>
```

## 5. Zero-Dependency ZIP Packaging

To eliminate external dependencies, Capacitor sync friction, and large bundle bloat, we implement a **pure TypeScript, uncompressed ZIP builder** (`ZipBuilder.ts`).

### Uncompressed (Store) Method

Audio files are already high-bandwidth binary content where DEFLATE algorithms offer very poor compression ratios for high CPU overhead. By storing files uncompressed (`Store` method = 0), we perform standard byte copies, achieving massive performance gains and instant exports!

### Directory Conventions

- Path: `<session-id>/<stem-name>.wav`
- Root contains:
  - `session.json` (Session manifest)
  - `README.txt` (DAW import instructions)

## 6. Render Pipeline & Memory Management

Stem renders are performed **sequentially** by default to maintain a strict memory ceiling, with an optional toggle for parallel rendering on desktop systems.

```
[User Clicks Export]
        │
        ├── Build session manifest & generate seed
        │
        ├── Sequential Loop:
        │     ├── Instantiate OfflineAudioContext
        │     ├── Rebuild duplicate audio graph for stem
        │     ├── script parameters at checkpoints (drift, arc)
        │     ├── Run OAC render
        │     ├── Encode resulting buffer to 24-bit/32-float WAV
        │     ├── Append bytes directly to ZIP accumulator
        │     └── Nullify context, buffer & graph nodes (GC release)
        │
        └── Generate ZIP blob & trigger share/download
```

- **Memory Cleanup**: Between each stem render, all references to the `OfflineAudioContext`, `AudioBuffer`, and `AudioNode` graph are completely unlinked to allow rapid Garbage Collection (GC) sweeps.
- **Time Alignment**: Leading silence for stems that start late (e.g. mid-session loop slots) is naturally generated by running the player inside the OAC timeline at its correct offset.

## 7. Realtime Capture Mode

Realtime capture mode is available for dynamic live jams with live mic performance or loop pedaling that cannot be reproduced offline.

- **MediaStream Tap**: We establish a `MediaStreamAudioDestinationNode` tap at each stem's source node in the active live graph.
- **Accumulators**: A separate Web Audio recorder / worklet accumulator gathers PCM float samples concurrently.
- **Synchronization**: One master trigger starts and stops recording on all active accumulators simultaneously, guaranteeing sample-accurate synchronization.
- **CPU Headroom**: Limited to a maximum of 12 stems on desktop and 8 on mobile to avoid CPU stutter.

## 8. User Interface & Experience

### The Export Dialog (`ExportDialog.tsx`)

A gorgeous, modern glassmorphic dialog:

- **Render Mode**: Offline (default) or Realtime.
- **Duration Selector**: Defaults to natural arc length or 5 minutes for open jam.
- **Quality Settings**:
  - Sample Rate: 48kHz (default), 44.1kHz, 96kHz (warns size).
  - Bit Depth: 24-bit (default), 32-bit float.
- **File Size Estimator**: Computes size estimate prior to render:
  $$\text{Size} = \text{Stems} \times \text{Duration (sec)} \times \text{Sample Rate} \times \text{Bit Depth} \times \text{Channels}$$

### Progress Display (`ExportProgress.tsx`)

- Sleek checkmarked list for each stem as it completes rendering.
- Circular progress halo with remaining render time.
- **Cancel button**: Immediately aborts render, nullifies active contexts, and returns to the patch interface.

## 9. Mobile Shell Support (Capacitor)

- **Filesystem Save**: On mobile shells (iOS and Android), the finished ZIP is saved into Capacitor's cache directory via `Filesystem.writeFile`.
- **Share Sheet**: Capacitor's `Share.share` opens the native sharing panel to let users save the ZIP directly to Files, AirDrop, or messaging apps.
- **Mobile Caps**: We cap default render duration to **10 minutes** on mobile (user-overrideable with warnings) to avoid WebView OOM suspensions.

## Open Questions for User Review

> [!IMPORTANT]
> **Zero-Dependency ZIP Builder vs JSZip**
> We propose building a lightweight, zero-dependency, uncompressed ZIP builder in pure TypeScript (`ZipBuilder.ts`).
>
> - **Pro**: Zero bundle overhead, works flawlessly inside standard Capacitor WebViews, extremely fast because it bypasses slow DEFLATE compression on heavy audio streams.
> - **Con**: Uncompressed ZIP files will have the exact same size as the sum of all stems, but since audio stems are already uncompressible PCM, the size difference is negligible.
>   We recommend this zero-dependency option. Please let us know if you prefer importing the full `jszip` package instead.

> [!WARNING]
> **Worklet Loading in OfflineAudioContext (iOS Safari)**
> iOS Safari WebViews have occasionally exhibited issues registering AudioWorklet modules on an `OfflineAudioContext` asynchronously.
>
> - **Mitigation**: We implement a reliable, step-wise `register` hook that lazily initializes the modules on the active context, backed by synchronous fallback detection. If worklet registration fails on the OAC, we fall back to realtime capture or single-file export seamlessly.
