# AnnealMusic v3.1 · Tempo + Pulse Engine · Specification & Plan

This document details the design, mathematics, audio specifications, and rollout plan for the opt-in tempo layer and the new **Pulse** engine in AnnealMusic.

---

## 1. Tempo Data Model & URL v9 Schema

### Data Model

Tempo is a piece-level property. It is modeled as a nullable floating-point number:

- **`tempoBpm: number | null`** on the `Piece` interface.
- A value of `null` represents a **tempoless** piece (default, identical behavior to v3.0).
- When `tempoBpm` is set, segment durations and transitions are represented in milliseconds, but can snap to beat boundaries.

### URL Schema Bumps to v9

To support sharing of tempo-enabled ambient compositions, the URL scheme is bumped to version 9 (`#s=9:...`).
The following new serialization elements are introduced:

1. **Piece Tempo:** Serialized as `tempo=<bpm>` (or omitted if tempoless).
2. **Grid-Lock Toggle:** Stored per-engine inside the defaultsState engine params as `<ns>.grid_lock=1` (or `0` when disabled).
3. **Tempo-Locked Segments:** Stored in the segment's config as `seg<idx>.tempoLocked=1` (or `0` when disabled).

**Backward Compatibility Policy:**

- Payloads with schema version <= 8 will load with `tempoBpm = null` and `grid_lock = false`.
- Custom parsers map old formats seamlessly, ensuring zero regression for existing compositions.

## 2. Timeline Grid Overlay UI

When a Piece has `tempoBpm` configured, the timeline editor renders a visual grid overlay behind segments:

- **Ticks:** Small vertical lines representing beats.
- **Accent Ticks:** Larger, brighter vertical lines representing bar boundaries (assuming standard 4/4 time, every 4 beats).
- **Scale Math:**
  - Let $T_{\text{beat}} = 60 / \text{BPM}$ (seconds per beat).
  - Visual beat spacing: $W_{\text{beat}} = T_{\text{beat}} \times \text{PX\_PER\_SEC} \times 10$ pixels.
  - Draw a subtle tick at $n \times W_{\text{beat}}$, and a highlighted tick when $n \pmod 4 == 0$.
- **Snapping:** Drag-resizing a segment will snap the duration to the nearest beat boundary:
  $$\text{snappedMs} = \text{round}\left(\frac{\text{durationMs}}{T_{\text{beat}} \times 1000}\right) \times T_{\text{beat}} \times 1000$$

## 3. Pulse Engine DSP Spec

The **Pulse Engine** is the 5th synthesis engine. It generates short percussive grains locked to subdivisons of the tempo.

### Synthesis Architecture

To maintain the ambient and organic character of AnnealMusic, the Pulse engine uses pitched percussion resembling struck plates or bells.

- **Excitation:** A brief 5ms filtered white-noise burst.
- **Resonators:** Six partials tuned to the harmonic lattice of `rootFreq` ($f_i = f_0 \times \text{HARMONICS}[i]^{\text{spread}}$).
- **Damping:** Each partial has an exponential decay envelope ($A_i(t) = A_i(0) e^{-d_i t}$), where damping $d_i$ scales with the partial frequency and tone.

### Parameter Controls

1. **`density` (Subdivision Index):** Selects active subdivisions of the tempo:
   - `whole` (1 strike per 4 beats)
   - `half` (1 strike per 2 beats)
   - `quarter` (1 strike per beat)
   - `eighth` (2 strikes per beat)
   - `sixteenth` (4 strikes per beat)
   - `triplet` (3 strikes per beat)
2. **`accent` (Boolean):** Emphasizes the first beat of each bar ($n \pmod{4 \times \text{subdivisions}} == 0$) by scaling the grain amplitude by $1.6$.
3. **`tone` (Filter Cutoff):** Maps 0..1 to lowpass cutoff ($200\text{ Hz}$ to $8000\text{ Hz}$), controlling high-frequency brightness.
4. **`swing` (0..1):** Shuffles paired subdivisions (eighth/sixteenths) by delaying every second pulse in a pair:
   $$\Delta t_{\text{swing}} = \text{swing} \times \frac{T_{\text{sub}}}{3}$$
5. **`humanize` (0..1):** Adds sample-accurate micro-timing jitter to mimic organic human play:
   $$\Delta t_{\text{jitter}} = \text{rand}(-1, 1) \times \text{humanize} \times 15\text{ms}$$

### AudioWorklet Scheduling

For 100% stable timing immune to main thread garbage collection pauses, grain triggering is handled directly inside the AudioWorklet processor thread by counting elapsed audio samples and triggering excitation bursts at computed sample offsets.

## 4. Grid-Lock Semantics

When `grid_lock` is enabled on the active engine, updates to `rootFreq` are quantized to the nearest future beat boundary.

### Hold-Until-Beat Semantics

To prevent melodic pitches from drifting asynchronously, the system adopts **Hold-Until-Beat**:

1. When a change to `rootFreq` is received via `setSharedParams()`:
   - Calculate the elapsed time since playback began: $t_{\text{elapsed}} = \text{currentTime} - \text{playStartAudioTime}$.
   - Find the next beat index: $n_{\text{next}} = \lceil t_{\text{elapsed}} / T_{\text{beat}} \rceil$.
   - Calculate the precise Web Audio time of the next beat:
     $$t_{\text{beat}} = \text{playStartAudioTime} + n_{\text{next}} \times T_{\text{beat}}$$
2. Schedule the oscillator frequency changes exactly at $t_{\text{beat}}$ using sample-accurate scheduling:
   - `setValueAtTime(currentFreq, t_beat - 0.005)`
   - `setTargetAtTime(targetFreq, t_beat, FREQ_TC)`
3. Non-pitch parameters (e.g. brightness, space, drift) continue to update instantly to preserve organic ambient sculpting.

## 5. Tempo-Locked Arcs and Transitions

- **Opt-in Toggle:** Arcs and transitions can set `tempoLocked: true` in their config.
- **Behavior:**
  - If `tempoLocked` is false, they run based on raw seconds (identical to v3.0).
  - If `tempoLocked` is true, their segment durations represent bars (1 bar = 4 beats).
  - The interpolation speed scales dynamically with the BPM. If the user speeds up the tempo, the arc runs faster; if they slow down, it runs slower.

---

## 6. MIDI Clock Integration

The MIDI Clock output (v1.6) is updated to support dynamic piece tempo sync:

- **`clockSyncToPieceTempo`:** A boolean toggle in the global MIDI configuration.
- **Transmission:**
  - When enabled and a piece tempo is set: MIDI clock pulses (24 PPQN) are sent at the piece's dynamic BPM.
  - When disabled or piece is tempoless: MIDI clock pulses are sent at the user-configured static BPM.

## 7. Risks & Mitigations

| Risk                                                          | Mitigation                                                                                                                                                                        |
| :------------------------------------------------------------ | :-------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Tempo breaks the ambient aesthetic for tempoless users.**   | **Opt-in Design:** By default, pieces remain tempoless (`tempoBpm: null`). The UI and audio engines behave exactly as in v3.0 unless a tempo is explicitly added.                 |
| **Worklet thread CPU overload due to six resonant partials.** | **Optimized DSP:** Resonators are implemented using extremely lightweight, highly optimized dampening envelopes and direct lookup formulas rather than heavy physical waveguides. |
| **MIDI Clock timing drift under browser main thread load.**   | **High-precision intervals:** MIDI clock pulse transmission uses high-priority timers coupled with thin event dispatches to keep jitter minimal.                                  |
