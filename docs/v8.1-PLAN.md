# Master Plan - AnnealMusic v8.1 (docs/v8.1-PLAN.md)

This document outlines the prioritized execution plan and risk register for the **v8.1 Code Health & Dependency Upgrades** milestone, compiled directly from findings in `docs/v8/AUDIT_CODE_HEALTH.md`.

---

## 1. Prioritized Audit Findings

Our execution is prioritized based on the severity of the Technical Debt:

1. **Schema Drift (Severity: High)**: Re-declaration of parameters in `OSCNamespace.ts` and the CLI's hardcoded reliance on `manifest.v20.json`.
2. **Dependency Outdatedness (Severity: High)**: Compiling client-side bundles under newer Vite 8, React 19, and TypeScript 6, ensuring React 19 concurrent modes work with existing audio worklets.
3. **Dead Code Accumulation (Severity: Medium)**: Flagged unused and duplicate exports bloating client bundles and backend routers.
4. **Type Coverage (Severity: Medium)**: Bringing TS coverage from `98.00%` to `>= 98.5%` and asserting backend `mypy` strictness.

---

## 2. Dependency Upgrade Order & Sequence

Dependency upgrades are grouped to resolve compilation gates cleanly:

1. **Vite 8 & TS 6 Upgrade**: Upgrading developer tools first allows us to detect styling and type mismatches immediately.
2. **React 19 & React Router 7 Upgrade**: Upgraded after developer tools to verify react state rendering compatibilities.
3. **Tailwind v4 & Zustand v5 Upgrade**: Upgraded to streamline the state and style compilers.
4. **Python Backend Core Upgrade**: Updating Starlette, Pydantic, and Boto3 in virtual environment.

---

## 3. Schema Unification Scope (v8.1 vs v8.4)

- **In-Scope for v8.1**:
  - Unifying the CLI validation commands to dynamically pull the highest available schema manifest.
  - Automating the synchronization of client-side `OSC_PARAM_TYPES` via `scripts/sync-schemas.mjs`.
  - Designating `schema/manifest.v21.json` as the absolute compile-time parameter truth.
- **Deferred to v8.4**:
  - Restructuring the fully composition-based `AnnealRenderer` (which bridges headless play, CLI render, and browser preview).
  - Unifying polymorphic schema specifications such as the sonification mapping templates and study export formats, which require structural refactoring of their respective layers.

---

## 4. Type Strictness Migration Plan

1. Enable typescript strict options (`strict`, `noImplicitAny`, `strictNullChecks`, `strictFunctionTypes`) in `tsconfig.app.json`.
2. Clean up explicit `any` typings inside `src/share/encode.ts` and dynamic event handlers inside the postMessage transport and OSC controllers.
3. Align backend FastAPI typing using `mypy --strict` assertions.

---

## 5. Dead-Code Removal Pass

We will remove the exact unused exports and duplicate defaults flagged in `knip` and `vulture` logs, confirming first that they are not referenced by:

- URL parameters-driven paths.
- Test suites.
- Headless preview scripts.

---

## 6. Risk Register

| Risk                                  | Impact | Likelihood | Mitigation Strategy                                                                            |
| ------------------------------------- | ------ | ---------- | ---------------------------------------------------------------------------------------------- |
| **React 19 Hooks & Ref changes**      | High   | Medium     | Sequential verification of all Tone.js sound player component lifecycle states.                |
| **Tailwind CSS v4 Compiler shift**    | Medium | Low        | Verify standard theme and border configs compile properly without missing tailwind-directives. |
| **Pydantic V2 Depreciation Warnings** | Low    | High       | Standardize on `model_validate` instead of `from_orm` where warnings are thrown.               |

---

## 7. Behavior Preservation & Regression Testing

- **Tests**: Automated testing using `npm run test` (716 tests) and backend `pytest` (328 tests) after each phase.
- **Manual Verification**: Launching the developer server to verify player, preset arc, and biofeedback features.

---

## 8. Order of Operations across CPs

- **CP 0**: Master planning, review, and hard-stop approval.
- **CP 1**: Package upgrades, lockfile updates, and compatibility fixes.
- **CP 2**: Schema unification via Node generator, OSC mapping consolidation, and CLI validate dynamic lookup.
- **CP 3**: Dead code removal, type coverage strictness assertions, walkthrough documentation, and tagging release v8.1.0.
