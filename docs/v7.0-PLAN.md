## 0. Diagnosis — how v7.0 sits on the existing codebase

The exploration of the v6.5 codebase establishes the patterns v7.0 must follow, and surfaces one
non-obvious tension that drives several design decisions:

**The `accounts` ↔ `users` split.** AnnealMusic has two identity tables (v1.3):

- `accounts` — authenticated identities (email magic-link / Google / GitHub OAuth). Have an
  `id` (UUID), `email`, `display_name`, etc.
- `users` — anonymous, per-device identities. **Every owned resource** (`patches`, `pieces`,
  `experiments`, `user_scripts`, `listening_sessions`, `captures`, `recordings`) FKs to
  `users.id`, **not** `accounts.id`.
- The bridge is `Identity.owned_anon_ids` (from `deps.get_identity`): the list of `users.id`
  values an account has claimed across its devices. Ownership checks today read
  `if resource.user_id not in identity.owned_anon_ids: raise forbidden()`.

A **Study requires auth** (per the rules of engagement), so `study_investigators.account_id`
correctly FKs to `accounts.id`. But the **resources** a study links are owned by `users.id`.
v7.0 must bridge the two: see §11. This is the single most important design decision in the
slice and it is settled here so CP1 doesn't have to.

**Confirmed patterns to replicate (from the codebase, not invented):**

## 1. Final data model

Refined from the mega-prompt sketch. Changes from the sketch are called out inline.

```sql
-- ── studies ─────────────────────────────────────────────────────────────────
CREATE TABLE studies (
  id                  UUID PRIMARY KEY,
  slug                TEXT NOT NULL UNIQUE,          -- [+] short public slug (new_slug()); for /s/<slug>
  title               TEXT NOT NULL,
  description         TEXT,
  abstract            TEXT,
  status              TEXT NOT NULL DEFAULT 'planning',
                      -- planning|pre-registered|active|data-collection|analysis|published|archived
  visibility          TEXT NOT NULL DEFAULT 'private',  -- [+] 'private' | 'public'
  preregistration_url TEXT,
  ethics_statement    TEXT,
  funding_sources     JSONB NOT NULL DEFAULT '[]',   -- [{ source, grant_number, role }]
  concept_doi         TEXT,                          -- [+] Zenodo concept DOI (all versions)
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  archived_at         TIMESTAMPTZ
);
CREATE INDEX idx_studies_visibility ON studies(visibility);

-- ── investigators ───────────────────────────────────────────────────────────
CREATE TABLE study_investigators (
  study_id    UUID NOT NULL REFERENCES studies(id) ON DELETE CASCADE,
  account_id  UUID NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  role        TEXT NOT NULL,                         -- pi|co-investigator|analyst|viewer
  added_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (study_id, account_id)
);
-- Invariant (enforced in app layer, see §3): every study has ≥1 'pi' at all times.

-- ── linked resources (polymorphic; no cross-table FK possible) ────────────────
CREATE TABLE study_resources (
  id            UUID PRIMARY KEY,
  study_id      UUID NOT NULL REFERENCES studies(id) ON DELETE CASCADE,
  resource_kind TEXT NOT NULL,
                -- patch|piece|listening_session|audio_clip|experiment|user_script|dataset|sonification
  resource_id   UUID NOT NULL,
  role          TEXT,                                -- stimulus|protocol|data|analysis
  added_by      UUID REFERENCES accounts(id) ON DELETE SET NULL,  -- [+] provenance
  added_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (study_id, resource_kind, resource_id)      -- [+] dedupe links
);
CREATE INDEX idx_study_resources_study ON study_resources(study_id);

-- ── immutable versions / snapshots ────────────────────────────────────────────
CREATE TABLE study_versions (
  id            UUID PRIMARY KEY,
  study_id      UUID NOT NULL REFERENCES studies(id) ON DELETE CASCADE,
  version_label TEXT NOT NULL,                       -- 'v1.0' | 'preregistered' | 'published'
  doi           TEXT,                                -- Zenodo version DOI (minted on publish)
  snapshot_json JSONB NOT NULL,                      -- full immutable snapshot (see §4)
  created_by    UUID REFERENCES accounts(id) ON DELETE SET NULL,  -- [+]
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (study_id, version_label)                   -- [+] labels unique within a study
);
CREATE INDEX idx_study_versions_study ON study_versions(study_id);

-- ── provenance / audit ────────────────────────────────────────────────────────
CREATE TABLE study_audit_log (
  id          UUID PRIMARY KEY,
  study_id    UUID NOT NULL REFERENCES studies(id) ON DELETE CASCADE,
  account_id  UUID REFERENCES accounts(id) ON DELETE SET NULL,
  timestamp   TIMESTAMPTZ NOT NULL DEFAULT now(),
  action      TEXT NOT NULL,                         -- see §5 enumerated set
  before      JSONB,
  after       JSONB
);
CREATE INDEX idx_study_audit_study ON study_audit_log(study_id, timestamp);

-- ── account profile extensions ────────────────────────────────────────────────
ALTER TABLE accounts
  ADD COLUMN orcid          TEXT,                     -- '0000-0002-3904-7128'
  ADD COLUMN affiliation_ror TEXT;                    -- 'https://ror.org/03t748b94'
```

**Refinements vs the sketch and why:**

- **`slug` + `visibility`** — the rules require "anyone can browse public studies + cite them."
  The public study **gallery UI** is deferred (v7.6), but the _data substrate_ for it ships now:
  a stable public slug (`/s/<slug>`) and a `visibility` flag. Default `private`.
- **`concept_doi` on `studies`, `doi` on `study_versions`** — mirrors Zenodo's two-tier model
  (one concept DOI for the study, one version DOI per snapshot). See §7.
- **`added_by` / `created_by`** — provenance is a first-class requirement; recording the actor on
  resource links and versions (not just the audit log) makes the snapshot self-describing.
- **`UNIQUE` constraints** — dedupe resource links and version labels.
- **No FK on `study_resources.resource_id`** — it is polymorphic across 8 tables; a single FK is
  impossible. Integrity is enforced at link time (resolve-and-verify, §11) and snapshots capture
  content at freeze time so later deletion of a source resource can't corrupt history.
- **Dialect portability** — `JSONB`→`sa.JSON()`, `TIMESTAMPTZ`→`DateTime(timezone=True)`,
  `UUID`→`GUID()` on SQLite, exactly as v6.4/v6.5 migrations do.

Migration file: `api/alembic/versions/0025_v7_0_studies.py` (down_revision `0024_v6_5_lesson_analytics`).
ORM models added to `api/app/models.py`.

## 2. Final API surface

All under `/api/v1`. Auth: studies are account-scoped, so write routes use `CurrentWriter` +
an account-resolved `Identity` (anon-only writers are rejected with `401 auth_required`).

```
# Studies CRUD
POST   /api/v1/studies                          create (becomes PI)            [auth]
GET    /api/v1/studies/me                        studies I'm an investigator on [auth]
GET    /api/v1/studies/:idOrSlug                 read (public if visibility=public, else investigators)
PATCH  /api/v1/studies/:id                       edit fields                    [pi|co-investigator]
DELETE /api/v1/studies/:id                       archive (soft; sets archived_at)[pi]

# Investigators
POST   /api/v1/studies/:id/investigators         add {account_email|account_id, role}  [pi]
PATCH  /api/v1/studies/:id/investigators/:acct    change role                   [pi]
DELETE /api/v1/studies/:id/investigators/:acct    remove                        [pi]

# Resources
GET    /api/v1/studies/:id/resources             list linked resources (resolved)  [read]
POST   /api/v1/studies/:id/resources             link {resource_kind, resource_id, role} [pi|co-investigator|analyst*]
DELETE /api/v1/studies/:id/resources/:resourceId unlink                          [pi|co-investigator]

# Versions / snapshots
POST   /api/v1/studies/:id/snapshot              freeze {version_label}          [pi|co-investigator]
GET    /api/v1/studies/:id/versions              list versions                   [read]
GET    /api/v1/studies/:id/versions/:versionId   retrieve a snapshot             [read]
POST   /api/v1/studies/:id/publish               mint DOI for a version          [pi]

# Provenance + citation
GET    /api/v1/studies/:id/audit                 audit log (paginated)           [investigators]
GET    /api/v1/studies/:id/citation?format=…     bibtex|apa|chicago (default bibtex) [read]

# Profile extensions
PATCH  /api/v1/account/me                        now also accepts {orcid, affiliation_ror} [auth]
```

\* `analyst` may link resources **only** with `role='analysis'` (their lane); see §3.

Conventions: `201` on create, `409 {error:'quota_exceeded'}` on quota, `403 forbidden` on
permission failure, `404` on missing/non-visible study, `422` on validation. CP0/CP1 may refine
exact codes; the matrix in §3 is authoritative for who-can-do-what.

## 3. Role + permission matrix

Roles: `pi`, `co-investigator`, `analyst`, `viewer`. Plus **anonymous** (no account) and
**non-investigator authed** users for public reads.

| Action                                                              | pi  | co-investigator | analyst | viewer | public (study=public) | public (study=private) |
| ------------------------------------------------------------------- | :-: | :-------------: | :-----: | :----: | :-------------------: | :--------------------: |
| Read study / resources / versions / citation                        |  ✓  |        ✓        |    ✓    |   ✓    |     ✓ (read-only)     |           ✕            |
| Read audit log                                                      |  ✓  |        ✓        |    ✓    |   ✓    |           ✕           |           ✕            |
| Edit study fields (title/abstract/status/funding/ethics/visibility) |  ✓  |        ✓        |    ✕    |   ✕    |           ✕           |           ✕            |
| Link resource (any role)                                            |  ✓  |        ✓        |    ✕    |   ✕    |           ✕           |           ✕            |
| Link resource (`role='analysis'` only)                              |  ✓  |        ✓        |    ✓    |   ✕    |           ✕           |           ✕            |
| Write/add analysis script as resource                               |  ✓  |        ✓        |    ✓    |   ✕    |           ✕           |           ✕            |
| Unlink resource                                                     |  ✓  |        ✓        |    ✕    |   ✕    |           ✕           |           ✕            |
| Create snapshot                                                     |  ✓  |        ✓        |    ✕    |   ✕    |           ✕           |           ✕            |
| Add / remove / re-role investigators                                |  ✓  |        ✕        |    ✕    |   ✕    |           ✕           |           ✕            |
| Publish (mint DOI)                                                  |  ✓  |        ✕        |    ✕    |   ✕    |           ✕           |           ✕            |
| Archive / delete study                                              |  ✓  |        ✕        |    ✕    |   ✕    |           ✕           |           ✕            |

**Invariants (app-enforced):**

1. **Always ≥1 PI.** A PI cannot be removed or downgraded if they are the last PI (`409 last_pi`).
2. **Creator is PI.** `POST /studies` inserts the creator as `pi`.
3. **Analyst lane.** Analyst writes are restricted to `study_resources` rows with
   `role='analysis'`; any other mutation → `403`.
4. **No self-escalation.** Only a PI changes roles; a PI may downgrade themselves only if another
   PI exists.

Implementation: a single helper `require_study_role(session, study_id, identity, min_role)` in
`api/app/routers/studies.py` (or `app/study_permissions.py`) used by **every** mutating route —
the heuristic-drift guard. It returns the investigator row or raises `forbidden()`/`auth_required()`.
Role ordering for `min_role`: `viewer < analyst < co-investigator < pi`, with the analyst-lane
carve-out handled explicitly at the resource route.

## 4. Snapshot semantics — frozen vs mutable

A **snapshot** (`POST /studies/:id/snapshot`) materializes the _entire current state_ of a study
into an immutable `study_versions` row. Snapshots are append-only and **never edited or deleted**
(matching the rule "snapshots are immutable").

**Frozen into `snapshot_json` (a self-contained, dereferenced bundle):**

```jsonc
{
  "schema": "study-snapshot/v1",
  "snapshot_at": "2026-05-31T12:00:00Z",
  "study": { "title", "description", "abstract", "status", "visibility",
             "preregistration_url", "ethics_statement", "funding_sources" },
  "investigators": [ { "account_id", "display_name", "orcid", "affiliation_ror", "role" } ],
  "resources": [
    { "resource_kind", "resource_id", "role", "added_by", "added_at",
      "title": "<resolved at freeze>",        // resolved metadata so deletion of source can't orphan it
      "content_hash": "sha256:…",             // hash of the resolved resource payload (reproducibility)
      "owner_user_id": "…" }
  ],
  "anneal_music_version": "7.0.0",
  "snapshot_label": "preregistered"
}
```

- **Resolved metadata + content hash** of each linked resource is captured at freeze time. This
  directly mitigates the "provenance gap if a resource is deleted before snapshot" risk (§10): a
  published version remains fully describable even if the underlying patch/experiment is later
  deleted. (v7.5 will extend this to full code+data bundling; v7.0 stores metadata + hash, not
  full payloads, to bound snapshot size.)
- The snapshot does **not** copy large binary payloads (audio, recordings) — only identifiers,
  titles, and content hashes. Keeps `snapshot_json` small (§10 size risk).

**Stays mutable (the live study):** everything in `studies`, `study_investigators`,
`study_resources` continues to be editable after a snapshot. Snapshots are point-in-time forks of
state, not locks. Editing the live study does not alter any existing version.

**Relationship to publish:** publishing (§7) selects a specific `study_versions` row (creating one
first if the caller passes a label) and mints its `doi`. A published version's `doi` and
`snapshot_json` are immutable; only the parent study's `concept_doi` and `status` change.

## 5. Audit log scope

`study_audit_log` records `(timestamp, account_id, action, before, after)` for **every state
mutation** of a study or its linked collection. Written through one helper —
`record_audit(session, study_id, account_id, action, before, after)` — called by every mutating
route (heuristic-drift guard: provenance is impossible to forget because the helper is the only
write path the routes use).

**Logged actions (`action` enum):**

| action                     | before                               | after                                |
| -------------------------- | ------------------------------------ | ------------------------------------ |
| `study.create`             | `null`                               | new study fields                     |
| `study.update`             | changed fields (old)                 | changed fields (new)                 |
| `study.archive`            | `{archived_at:null}`                 | `{archived_at:…}`                    |
| `study.publish`            | `{status, version_id}`               | `{status:'published', doi}`          |
| `investigator.add`         | `null`                               | `{account_id, role}`                 |
| `investigator.role_change` | `{role:old}`                         | `{role:new}`                         |
| `investigator.remove`      | `{account_id, role}`                 | `null`                               |
| `resource.link`            | `null`                               | `{resource_kind, resource_id, role}` |
| `resource.unlink`          | `{resource_kind, resource_id, role}` | `null`                               |
| `snapshot.create`          | `null`                               | `{version_id, version_label}`        |

**Not logged** (by design): reads of any kind (study GET, resource list, version retrieval,
citation generation, **audit-log reads themselves**) — reads carry no provenance value and would
balloon the table. Account profile edits (`orcid`/`ror`) are logged in the account's own history,
not per-study. `before`/`after` store only the **changed** fields, not whole-row snapshots, to
keep rows compact (whole-state capture is the snapshot's job, §4).

Audit reads (`GET /studies/:id/audit`) are restricted to investigators and paginated
(`?limit=&before=` cursor on `timestamp`).

## 6. Citation format support

Three formats, generated server-side from study + version metadata, with a TS parallel for the CLI.

- **BibTeX** — `@misc{...}` (or `@software` when a code resource dominates). Fields: author list
  (ORCID-annotated), title, year, `howpublished`/`publisher = {Zenodo}`, `doi`, `url`, `version`.
- **APA 7th** — `Author, A. A., & Author, B. B. (Year). *Title* (Version) [Data set/Study].
Zenodo. https://doi.org/<doi>`.
- **Chicago (author-date)** — `Author, First, and Second Author. Year. "Title." Version. Zenodo.
https://doi.org/<doi>.`

**Server:** `api/app/services/citation.py` — pure functions
`bibtex(study, version) -> str`, `apa(...)`, `chicago(...)`, plus `author_list(investigators)`
that orders PIs first, then co-investigators, and appends ` (ORCID: …)` where present. Driven by
`GET /studies/:id/citation?format=bibtex|apa|chicago`. Unit-tested in isolation (pure → easy).

**Client/CLI:** extend `tools/cite/` with `apa-generator.ts` + `chicago-generator.ts` alongside
the existing `bibtex-generator.ts`, and widen `tools/cli/src/commands/cite.ts` to accept a study
id/slug and a `--format` flag (defaults preserve today's behavior: project-level BibTeX).

If a study has no DOI yet (unpublished), citations are still generated using the canonical
`/s/<slug>` URL and an "Unpublished study" note, so works-in-progress remain citable.

## 7. Zenodo integration for study DOIs

Parallels v5.7's release-DOI pattern but is **runtime, API-driven** (v5.7 was static
`.zenodo.json` + GitHub→Zenodo webhook). New service: `api/app/services/zenodo.py`.

**Two-tier DOI model (Zenodo native):**

- **Concept DOI** — one per study (stored on `studies.concept_doi`), resolves to the latest
  version. Minted on first publish via Zenodo's deposition versioning.
- **Version DOI** — one per published `study_versions` row (stored on `study_versions.doi`).
  Each snapshot published gets its own citable version DOI.

**Publish flow (`POST /studies/:id/publish`):**

1. Pre-flight checklist (server-validated, mirrors UI in §8): abstract present, ethics_statement
   present, ≥1 PI, every investigator has an ORCID, a target `study_versions` row exists (create
   one from `version_label` if absent). Fail → `422 {error:'preflight_failed', missing:[…]}`.
2. Build Zenodo deposition metadata from the version's `snapshot_json` (title, creators with ORCID
   - ROR affiliation, description=abstract, keywords, license `AGPL-3.0-or-later`, funding,
     `upload_type=dataset`/`publication`).
3. Create deposition → upload `snapshot_json` as the artifact → publish → capture `doi` +
   `conceptdoi`. Persist to `study_versions.doi` and `studies.concept_doi`; set `status='published'`;
   write `study.publish` audit row.

**Robustness (Anthropic-recommended robust HTTP client):** a lazy-init `httpx.AsyncClient`
(matching `oauth.py`/`email.py`), wrapped in a small retry helper — bounded exponential backoff
(0.5s→1s→2s→4s, max 4 attempts) on `5xx`, `429` (honoring `Retry-After`), and connect/timeout
errors; **no retry** on `4xx` validation errors. Idempotency: store the in-progress
`deposition_id` so a retried publish resumes rather than double-mints.

**Config & test isolation:** `ZENODO_API_URL` (defaults to **sandbox** `sandbox.zenodo.org` in
dev/test) + `ZENODO_TOKEN`. Tests and the CP3 smoke matrix run against **Zenodo Sandbox**; if no
token is configured the service runs in a **stub mode** that returns a deterministic fake DOI so
CI never depends on a live external service (consistent with the email/LLM stub pattern). Real
minting only when `ZENODO_TOKEN` is set.

**ORCID + ROR:** stored on `accounts` (§1), surfaced in `PATCH /account/me`, validated with light
format checks (ORCID `\d{4}-\d{4}-\d{4}-\d{3}[\dX]`, ROR a `ror.org` URL). No live ORCID OAuth in
v7.0 — investigators self-enter; full ORCID OAuth linking is a candidate for a later v7 slice.

## 8. UI plan — Studies panel

New tab in the `/research` console (`src/research/ResearchApp.tsx`): add `'studies'` to the
`activeTab` union and a lucide `FlaskConical`/`Microscope` icon, following the existing
tab-routing + `fetch()` conventions (no new state library). All study UI lives in `src/studies/`.

- **`StudiesPanel`** — lists `GET /studies/me`, sorted by last activity (`updated_at`); "New
  study" button; empty-state for anon users prompting sign-in (studies require auth).
- **`StudyView`** — full detail: header (title, status badge, visibility toggle), investigator
  chips, resources grouped by `resource_kind`, version list, and an **audit-log sidebar**
  (`GET /studies/:id/audit`). Controls are shown/hidden per the §3 matrix using the caller's role
  returned in the study payload.
- **`InvestigatorManager`** (PI-only) — add by email, change role, remove; guards the last-PI
  invariant in the UI with a disabled state + tooltip.
- **`ResourceLinker`** — multi-kind picker. Fetches the caller's (and co-investigators') existing
  patches/pieces/experiments/scripts/listening-sessions/captures, lets the user pick a `role`
  (stimulus/protocol/data/analysis), button-add (drag-drop is a nice-to-have, not required for
  CP3). Analyst sees only the `analysis` role option.
- **`SnapshotDialog`** — version-label input + a preview of what will be frozen; calls
  `POST /studies/:id/snapshot`.
- **`PublishFlow`** — pre-flight checklist UI (abstract ✓, ethics ✓, ORCIDs ✓, version selected ✓)
  with each item green/red, then a "Mint DOI" button hitting `POST /studies/:id/publish`; shows the
  returned DOI on success.

Styling matches the research console's dark stone palette + lucide icons. API access via plain
`fetch` to `/api/v1/studies/...`; a thin `src/studies/api.ts` wraps the endpoints with typed
responses to avoid scattering fetch calls.

## 10. Risks & mitigations

| #   | Risk                                                                                         | Mitigation                                                                                                                                                                                                |
| --- | -------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 1   | **Zenodo API rate limits / outages** block publish                                           | Bounded exp-backoff + `Retry-After` honor; idempotent resume via stored `deposition_id`; **sandbox + stub** so CI never hits live Zenodo; publish failure leaves study un-`published` (no partial state). |
| 2   | **Snapshot size** for studies with many resources                                            | Snapshots store resolved **metadata + content hash**, never binary payloads; `snapshot_json` stays KB-scale. Full code+data bundling explicitly deferred to v7.5.                                         |
| 3   | **Provenance gap if a resource is deleted before snapshot**                                  | Resolve-and-capture at freeze time (title + hash in `snapshot_json`); `added_by` recorded at link time; deletion of a source resource can't corrupt an existing version.                                  |
| 4   | **accounts ↔ users ownership mismatch** (resources FK `users.id`, studies use `accounts.id`) | Settled in §11: link-time ownership check spans all investigators' `owned_anon_ids`; resources need not be owned by the linker, only by _some_ investigator.                                              |
| 5   | **Polymorphic `resource_id` integrity** (no FK across 8 tables)                              | Resolve-and-verify at link time; dedupe via `UNIQUE(study_id, kind, id)`; snapshots make history deletion-proof (risk 3).                                                                                 |
| 6   | **Permission drift** across many routes                                                      | One `require_study_role` helper + one `record_audit` helper used by every mutation; covered by an access-control test that sweeps each role × each action.                                                |
| 7   | **Last-PI lockout / orphaned study**                                                         | App invariant: ≥1 PI always; removing/downgrading the last PI → `409`.                                                                                                                                    |
| 8   | **ORCID/ROR data quality**                                                                   | Light format validation on write; publish pre-flight requires ORCID for all investigators. No live ORCID OAuth this slice (deferred).                                                                     |

## 11. The accounts ↔ users reconciliation (design decision)

This is the load-bearing decision; recording it so CP1 is mechanical.

- **Investigators are accounts.** `study_investigators.account_id → accounts.id`. Study writes
  require an authenticated account: routes use `CurrentWriter` and then resolve
  `identity.account_id`; if `None` → `401 auth_required`.
- **Resources are user-owned.** To link a resource, the server verifies the `resource_id` is owned
  by **any investigator on the study** — i.e. `resource.user_id ∈ ⋃(owned_anon_ids of all
investigator accounts)`. This realizes the spec's "pick from resources owned by **any**
  investigator." A co-investigator can therefore link the PI's patch, etc.
- **Public reads need no account.** `GET /studies/:idOrSlug` and `GET .../citation` succeed for
  anonymous callers when `visibility='public'`; otherwise `404` (not `403`, to avoid leaking
  existence of private studies).
- **Audit actor** = the acting `account_id` (nullable FK with `ON DELETE SET NULL` so deleting an
  account preserves the provenance trail with a tombstone).

Computing "union of all investigators' owned_anon_ids" requires a helper that, given a study,
gathers each investigator account's claimed `users.id`s (join `accounts → users` via the existing
claim linkage). This helper is the only new cross-table query and is unit-tested.
