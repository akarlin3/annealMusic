## 3. Authoring workflow tooling design (what CC builds in CP1)

The per-lesson human loop (brief ¬ß"Authoring workflow"): author spec (10‚Äď15m) ‚Üí generate ‚Üí
review (5‚Äď10m) ‚Üí refine/override ‚Üí approve+publish; ~30‚Äď60m/lesson. Tooling target:
make 55 lessons tractable and keep the heuristics (spec format, QA rules, DAG check)
defined **once**.

### 3.1 Spec generator (`SpecGenerator.tsx` + `services/spec_generator.py`)

- **Input:** topic + one-line outline (+ optional difficulty, track, suggested clips).
- **Output:** a *starting* `LessonSpec` JSON (id, track, title, objectives, difficulty,
  `step_outline`, `constraints_during_prompts`) that the author edits. **LLM-assisted, the
  author always edits** ‚ÄĒ never auto-publishes.
- Reuses `services/llm.py` (Haiku) with a dedicated `SPEC_PROMPT_VERSION`; output validated
  against the existing `LessonSpec` pydantic model before returning (reject + retry‚Č§1 on
  invalid). Honors `FRAMING.md`: when topic matches the framing-sensitive lexicon
  (432/solfeggio/healing/binaural/entrainment), the system prompt injects the honest-framing
  directive and seeds a reflection step that surfaces the evidence.
- New endpoint `POST /api/v1/admin/curriculum/spec-generate`.

### 3.2 Batch generation (`BatchGenerationDashboard.tsx`)

- Lists all specs and their `generation_status` (`pending`/`generating`/`ready`/`generation_failed`).
- **"Generate all pending"** enqueues one generation per pending/changed spec; results queue
  for review. Re-running an unchanged spec is a **free cache hit** (existing key includes
  spec hash), so batch runs are cheap and idempotent.
- Per-row: status, last-generated time, cost, error (if failed), re-queue button.
- Backed by `POST /api/v1/admin/curriculum/batch-generate` (accepts a list of spec ids or
  "all pending"); generation itself reuses `generate_lesson`. Serial with small concurrency
  cap to respect rate limits; progress polled via existing `/lessons` status.

### 3.3 Review dashboard (`ReviewDashboard.tsx`)

- Per-lesson row: status + QA badge (green/amber/red from ¬ß4).
- **Side-by-side:** spec (left) ‚ÜĒ generated steps rendered with the real step renderers
  (right), so review matches the learner view.
- Actions per lesson: **Approve & publish**, **Revise** (edit spec inline ‚Üí regenerate),
  **Regenerate** (force-fresh), and per-step **Manual override** (reuses
  `StepOverrideEditor` + existing override endpoints).
- "Publish" flips a `published` flag (see ¬ß6 versioning); only `ready` + QA-passing lessons
  can publish. Calm-by-design: no scores, no streaks ‚ÄĒ just per-lesson state.

### 3.4 Quality-check pipeline (`QualityChecks.tsx` + `services/curriculum_qa.py`)

See ¬ß4. Runs on demand (button) and automatically after each generation; results cached on
the lesson and surfaced as the review badge. Pure functions, no LLM, so it's fast and CI-able.

### 3.5 Prerequisite graph editor (`PrerequisiteGraphEditor.tsx`)

- Visual node-link view of all lessons (grouped/colored by track).
- Drag to connect prerequisite ‚Üí lesson; **cycle attempt is rejected in-editor** with a clear
  message (DFS back-edge check on every add).
- Reads/writes `prereqs.json` via `GET/PUT /api/v1/admin/curriculum/prereqs`; the DAG check
  (¬ß4) is the same code the editor calls, so the guard and the linter never drift.
- Lightweight SVG/HTML layout (no heavy graph lib) to stay within the calm, small-bundle
  ethos; layout is deterministic (track columns √ó depth rows).

### 3.6 Content versioning (¬ß6 requirement)

When a spec changes, the **old generation is preserved**; a new generation is produced; admin
compares and chooses which to publish. Implemented as a `lesson_generations` history row keyed
by spec-hash (details ¬ß6); the review dashboard exposes a compare view.

## 4. Quality-check rules (`services/curriculum_qa.py`)

Each rule returns `{id, level: error|warn, message}`. **Errors block publish; warns are
advisory.** All rules are pure (no network) and exercised by unit tests + reused in CI on the
seeded curriculum.

| # | Rule | Level | Detail |
| --- | --- | --- | --- |
| 1 | **Step-type coverage** | error | Lesson has at least the step types its spec declared, and at least one `demo` *or* `audio-clip` (every lesson must let you *hear* something) and one `reflection`. |
| 2 | **Audio clips exist** | error | Every `audio-clip` step resolved to a real, non-archived clip in the v6.2 library; flags lessons whose `clip_topic` retrieved nothing (‚Üí author a new clip, ¬ßCP2.6). |
| 3 | **Patch demos validate** | error | Every `demo` step's `patch` validates against the live schema manifest (reuses v6.1 demo validator). |
| 4 | **SVG diagrams sanitize** | error | Every generated SVG passes `svg_sanitizer`; mermaid passes the mermaid validator. |
| 5 | **Word-count sanity** | warn | Per-step body within `[min,max]` band by type (e.g. text 60‚Äď320 words, reflection prompt 15‚Äď80, demo description 20‚Äď120). Too-short ‚Üí thin; too-long ‚Üí wall-of-text. |
| 6 | **Prereqs form a DAG** | error | Whole-graph: no cycles (DFS), no self-edge, all referenced ids exist, root reachable. Run over `prereqs.json`. |
| 7 | **Spec/id integrity** | error | `id == track/slug`, track exists & not archived, difficulty in enum, 1‚Äď8 objectives. |
| 8 | **Framing compliance** | warn | If lesson topic/objectives hit the framing lexicon, assert the generated copy contains honest-framing language and no prohibited claim phrases (`DNA repair`, `clinically proven`, `cures`, `brainwave entrainment`, ‚Ä¶). The `432-solfeggio` lesson must explicitly state the evidence is absent. |
| 9 | **Difficulty monotonicity** | warn | A lesson's difficulty ‚Č• max(prereq difficulties) ‚ąí tolerance; flags an `intro` lesson gated behind an `advanced` one. |

Rule 6, 8 and the framing lexicon are defined **once** and shared by the spec generator,
the QA pipeline, the graph editor, and CI ‚ÄĒ the heuristic-drift guard the brief calls for.

## 5. Batch generation cost estimate

Model: Claude Haiku (`lesson_gen_model_id = claude-3-haiku-20240307`), priced **$0.25 / M
input, $1.25 / M output** (`estimate_haiku_cost`). Generation is **cached** by spec hash, so
the dominant cost is the *first* generation of each lesson plus review-driven regenerations.

Assumptions: 55 lessons √ó ~5 steps + ~1.5 diagram sub-steps ‚Čą **6.5 LLM calls/lesson**. Per
call ‚Čą 3,000 input (system + few-shot + neighbors) + ~500 output tokens ‚Čą **$0.0014/call**.

| Phase | Calls | Tokens (in/out) | Cost |
| --- | --- | --- | --- |
| Spec generation (55 specs, 1 call each) | 55 | ~1.5k / ~0.8k | **‚Čą $0.08** |
| First full generation (55 √ó 6.5) | ~358 | ~3k / ~0.5k | **‚Čą $0.49** |
| Review regenerations (~1.5√ó churn factor) | ~180 | ‚ÄĒ | **‚Čą $0.25** |
| Re-runs of unchanged specs | ‚ÄĒ | cache hits | **$0.00** |
| **Total (one-time, whole curriculum)** | | | **‚Čą $0.80‚Äď$1.20** |

Even with heavy iteration the slice stays **well under ~$2**. Cost is governed by the
existing 30-day trailing-cost guard in `lesson_generation.py`; batch runs respect it. (If the
model is later upgraded to Haiku 4.5, re-estimate ‚ÄĒ pricing differs ‚ÄĒ but order of magnitude
holds.) The real budget here is **human authoring time**: ~30‚Äď60 min/lesson √ó 55 ‚Čą
**27‚Äď55 hours**, spread across CP2 and committed incrementally.

## 7. Risks & mitigations

| Risk | Likelihood | Mitigation |
| --- | --- | --- |
| **Editorial throughput** ‚ÄĒ 27‚Äď55h of human authoring is the real critical path; slice could stall. | High | CP2 ships incrementally (commit per track / per batch); spec generator gives a fast first draft; batch generation + review dashboard cut the per-lesson overhead; **MERGE GATE: CP1 ships independently** so tooling lands before all content is done. |
| **LLM generation quality drift across 55 lessons** ‚ÄĒ tone/quality varies; framing-sensitive topics get sloppy. | Med | Deterministic spec format + few-shot examples + `LESSON_PROMPT_VERSION`; QA pipeline (¬ß4) gates publish; framing lexicon (rule 8) is automated, not eyeballed; review dashboard renders the real learner view; regenerate is a free cache-miss when prompts bump. |
| **Prerequisite graph complexity / cycles** ‚ÄĒ 55 nodes, cross-track edges. | Med | Single `prereqs.json` source of truth; DAG check (rule 6) shared by editor guard + linter + CI; depth budget ‚Č§5; single root; intro lessons ‚Č§1 prereq so onboarding never gates. |
| **Audio-clip gaps** ‚ÄĒ lessons reference clips the v6.2 library lacks. | Med | QA rule 2 flags missing clips up front; CP2.6 authors the gap clips via existing `ClipManager`/upload path before publish. |
| **Calm-by-design regression** ‚ÄĒ review/QA dashboards or browser sprout scores/streaks/funnels. | Low/High-impact | Dashboards show state, not scores; no completion %, no gamification; `calm-by-design` CI lexical gate already covers `src/learn`; final calm review in CP3 smoke matrix. |

| **Heuristic drift** ‚ÄĒ spec format, QA rules, DAG, framing defined in multiple places. | Med | Each lives **once**: `LessonSpec` (schema), `curriculum_qa.py` (rules + DAG), `framing_lexicon.py` (framing). Editor and CI call the same functions. |
