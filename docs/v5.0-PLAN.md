# AnnealMusic v5.0 Plan: Research Surface Foundation

This document specifies the technical design, JSON-RPC 2.0 schemas, transport protocols, modular bundle layout, and performance budget for the **v5.0 Research Surface Foundation** milestone.

---

## 1. Final Bridge v1 API Surface Specification

Every method uses JSON-RPC 2.0 rules. Pushes/events from subscriptions are sent as JSON-RPC notifications (omitting `id`).

### State Observation

#### `anneal.state.get`

- **Description**: Returns the full parameter store state, including current creative mode, parameters, active engine, custom scales, and active tuning.
- **Parameters**: None.
- **Return Shape**:
  ```typescript
  {
    params: {
      rootFreq: number;
      spread: number;
      density: number;
      coupling: number;
      drift: number;
      brightness: number;
      space: number;
      volume: number;
    }
    engineId: string;
    engineParams: Record<string, any>;
    tuning: {
      system: string;
      referenceA4Hz: number;
    }
    mode: 'sketch' | 'drone';
  }
  ```

#### `anneal.state.subscribe`

- **Description**: Subscribes to changes on specific keys of the param store (e.g. `params`, `engineId`, `tuning`).
- **Parameters**:
  ```typescript
  {
    keys: ("params" | "engineId" | "tuning" | "mode")[];
  }
  ```
- **Return Shape**:
  ```typescript
  {
    subscriptionId: string; // GUID
  }
  ```
- **Pushed Notification Format**:
  Sent as a JSON-RPC notification when values change:
  ```json
  {
    "jsonrpc": "2.0",
    "method": "anneal.state.onChange",
    "params": {
      "subscriptionId": "sub_uuid_here",
      "key": "params",
      "value": { "rootFreq": 120.0 }
    }
  }
  ```

#### `anneal.state.unsubscribe`

- **Description**: Cancels an active state subscription.
- **Parameters**:
  ```typescript
  {
    subscriptionId: string;
  }
  ```
- **Return Shape**: `boolean` (true on success).

#### `anneal.engine.getSpectrum`

- **Description**: Fetches the latest Web Audio AnalyserNode FFT spectrum frame. High-performance subscriptions can pull this up to 60Hz.
- **Parameters**: None.
- **Return Shape**:
  ```typescript
  {
    spectrum: number[]; // Array of uint8 frequency bin values
  }
  ```

#### `anneal.engine.getPartials`

- **Description**: Retrieves the emergent phase-coupled partial values from the active oscillator engine (including frequency and amplitude).
- **Parameters**: None.
- **Return Shape**:
  ```typescript
  {
    partials: {
      freq: number;
      amp: number;
    }
    [];
  }
  ```

### State Mutation

#### `anneal.state.set`

- **Description**: Patches the core parameters. Clamps all values to their physical bounds.
- **Parameters**:
  ```typescript
  {
    params: {
      rootFreq?: number;
      spread?: number;
      density?: number;
      coupling?: number;
      drift?: number;
      brightness?: number;
      space?: number;
      volume?: number;
    };
  }
  ```
- **Return Shape**: `boolean` (true on success).

#### `anneal.state.setEngine`

- **Description**: Swaps the active sound synthesis engine dynamically (e.g. `'sine'`, `'waveguide'`, `'bowed'`).
- **Parameters**:
  ```typescript
  {
    engineId: 'sine' |
      'waveguide' |
      'bowed' |
      'pulse' |
      'fm' |
      'mallet' |
      'membrane' |
      'edge';
  }
  ```
- **Return Shape**: `boolean` (true on success).

#### `anneal.state.setTuning`

- **Description**: Updates the reference tuning system and A4 frequency.
- **Parameters**:
  ```typescript
  {
    tuning: {
      system: string;
      referenceA4Hz: number;
    }
  }
  ```
- **Return Shape**: `boolean` (true on success).

### Lifecycle Management

#### `anneal.session.start`

- **Description**: Resolves and starts the active listening session or standalone meditation timer.
- **Parameters**: None.
- **Return Shape**: `boolean` (true on success).

#### `anneal.session.stop`

- **Description**: Fades and stops the active listening session or standalone meditation timer.
- **Parameters**: None.
- **Return Shape**: `boolean` (true on success).

#### `anneal.session.status`

- **Description**: Returns the current status of the running session, along with elapsed and remaining durations.
- **Parameters**: None.
- **Return Shape**:
  ```typescript
  {
    status: 'idle' | 'playing' | 'paused' | 'ended';
    elapsedMs: number;
    remainingMs: number;
  }
  ```

#### `anneal.session.loadPatch`

- **Description**: Directly hydrates the parameters, engine, and tuning parameters from a pre-saved patch.
- **Parameters**:
  ```typescript
  {
    patch: {
      params: Record<string, number>;
      engineId: string;
      engineParams: Record<string, any>;
      tuning: {
        system: string;
        referenceA4Hz: number;
      }
    }
  }
  ```
- **Return Shape**: `boolean` (true on success).

#### `anneal.session.loadPiece`

- **Description**: Loads an arrangement piece, including segments and bell schedules.
- **Parameters**:
  ```typescript
  {
    piece: Record<string, any>;
  }
  ```
- **Return Shape**: `boolean` (true on success).

## 2. JSON-RPC Error Code Table

All bridge error responses strictly follow the JSON-RPC 2.0 error specification.

| Error Code | Error Message            | Description                                                                  |
| :--------- | :----------------------- | :--------------------------------------------------------------------------- |
| `-32700`   | `Parse error`            | Invalid JSON received by the server.                                         |
| `-32600`   | `Invalid Request`        | The JSON sent is not a valid JSON-RPC 2.0 request.                           |
| `-32601`   | `Method not found`       | The requested method does not exist on the bridge.                           |
| `-32602`   | `Invalid params`         | The method parameters are invalid or fail schema bounds.                     |
| `-32603`   | `Internal error`         | Internal bridge server error.                                                |
| `-32000`   | `Unauthorized`           | WebSocket or CLI token validation failed (for v5.1+).                        |
| `-32001`   | `Security Violation`     | Attempted action violates the safety policy (e.g. attempting account saves). |
| `-32002`   | `Engine Not Initialized` | The AudioContext or engine orchestrator is not yet bootsrapped.              |

## 3. Multi-Transport Architecture & Roadmap

The wire protocol is transport-agnostic (pure JSON-RPC 2.0). Different versions support different transports depending on context:

```
[ Research Panel / CLI / Script ]
           |
           v (JSON-RPC 2.0 payload)
+------------------------------------------+
| Transport Layer                          |
|  - v5.0: BroadcastChannel (Same-origin)  |
|  - v5.1: WebSocket (OSC Bridge)          |
|  - v5.2: stdio (CLI standard I/O)        |
+------------------------------------------+
           |
           v
   [ Bridge Server ] ---> [ Audio Orchestrator ]
```

- **v5.0 (BroadcastChannel)**: Standard communication channel named `anneal_music_bridge`. BroadcastChannel isolates scopes automatically to the same-origin, same-browser context, providing implicit security.
- **v5.1 (WebSocket)**: Introduces native secure WebSockets (`ws://` / `wss://`) to bridge incoming OSC commands, utilizing secure authentication tokens.
- **v5.2 (stdio)**: Enables command-line control of running headless node instances.

## 4. Versioning Policy

We commit to a strict semantic versioning policy for the Research Bridge:

1. **Bridge v1 Support**: Supported indefinitely.
2. **Additive-Only**: Changes within a major version (e.g., v1.1, v1.2) must be strictly additive. Breaking changes, parameter narrowing, or method removals are prohibited.
3. **Major Version Breaks**: Any breaking changes or structural updates will require a major bridge version bump (e.g., Bridge v2). In such cases, the server must support both active versions side-by-side during a deprecated transition window.

## 5. Performance Budget Verification Plan

We establish a strict latency and size budget:

- **JS/CSS Bundle Size Budget**: Combined `< 100KB` gzipped for the initial `/research` console.
- **Call Overhead Target**: `< 2ms` typical, `< 10ms` p99.
- **FFT Streaming**: Subscriptions streaming high-rate data like spectrums use standard throttled loops (maximum 60Hz) or SharedArrayBuffers in high-performance contexts.

### Verification Commands

- **Bundle Size**: We will build using `npm run build:research` and inspect the `dist-research/` outputs.
- **Latency profiling**: We will implement a benchmark script under `src/research/bridge/__tests__/bridge_perf.test.ts` that measures call trip duration over 1000 iterations.

## 6. Safety Model

The Research Bridge acts purely as a local control interface. It **cannot** perform the following destructive or unauthorized operations:

1. **Save Patches**: Cannot trigger backend writes to database tables (such as `patches` or `pieces` tables). Saving must be done manually by the authenticated user via the main app UI.
2. **Social Interaction**: Cannot follow users, like patches, or post comments.
3. **Gallery Publishing**: Bridge cannot trigger publishing of pieces or patches to the public gallery.
4. **Data Exfiltration**: The bridge server lives entirely within the sandboxed web client, with no permissions to access remote endpoints beyond standard same-origin WebSocket tunnels.

## 7. Telemetry & Debug Panel Design

The research shell features a dedicated **RPC Telemetry Console**:

- **Chronological Logs**: Displays all incoming requests, outgoing responses, and subscription chimes in real-time.
- **Latency Timers**: Tags each method execution with round-trip delta in milliseconds.
- **Error Highlights**: Flags parsing or validation failures in distinct high-visibility red labels.
- **Strict Privacy**: Telemetry lives purely in-memory in `/research`. No logs are collected or written to any remote telemetry database. Users can clear logs at any time or trigger a manual JSON export.

## 9. Risks & Mitigations

- **Risk: Version Drift & Breakages**
  - _Mitigation_: The `src/research/bridge/schema.ts` file acts as the single source of truth. Both `BridgeServer` and `BridgeClient` use shared TypeScript interfaces derived directly from this file.
- **Risk: Hidden Circular Dependencies**
  - _Mitigation_: The bridge crosses the bundle boundaries only using the serialized JSON-RPC 2.0 format over the `BroadcastChannel`. Neither the main app imports anything from the `research/` UI, nor the research bundle imports the active main app. They are decoupled completely.
- **Risk: Performance Lag in Main App**
  - _Mitigation_: The `BridgeServer` executes all JSON-RPC dispatching inside standard microtasks. High-frequency operations like spectrum subscriptions are gated using active throttlers.
