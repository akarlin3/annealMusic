# Anneal Ambiance v2.0 · Social on Gallery · Implementation Plan

This document details the architectural design and implementation plan for adding the social surface (Likes, Follows, Activity Feed, Profile pages, Blocks/Mutes, and Curated Featured Picks) on top of Anneal Ambiance v1.9.

---

## 1. Data Model Additions

We extend the database schema with table definitions and relationships compatible with both PostgreSQL (production) and SQLite (testing/local development).

```sql
-- Likes table: represents user interactions with patches or recordings.
-- Both logged-in users (via their account's claimed user_id or active user_id) and anonymous users can like.
CREATE TABLE likes (
  user_id        UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  target_kind    TEXT NOT NULL,                -- 'patch' | 'recording'
  target_id      UUID NOT NULL,
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, target_kind, target_id)
);
CREATE INDEX idx_likes_target ON likes(target_kind, target_id);
CREATE INDEX idx_likes_user   ON likes(user_id, created_at DESC);

-- Follows table: one-directional relationships between accounts.
CREATE TABLE follows (
  follower_account_id  UUID NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  followed_account_id  UUID NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  created_at           TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (follower_account_id, followed_account_id),
  CONSTRAINT chk_no_self_follow CHECK (follower_account_id <> followed_account_id)
);
CREATE INDEX idx_follows_followed ON follows(followed_account_id);
CREATE INDEX idx_follows_follower ON follows(follower_account_id);

-- Blocks table: bidirectional content hiding and one-way prevention of social interactions.
CREATE TABLE blocks (
  blocker_account_id  UUID NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  blocked_account_id  UUID NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (blocker_account_id, blocked_account_id),
  CONSTRAINT chk_no_self_block CHECK (blocker_account_id <> blocked_account_id)
);
CREATE INDEX idx_blocks_blocked ON blocks(blocked_account_id);

-- Mutes table: removes a muted user's content from the blocker/muter's activity feed only.
CREATE TABLE mutes (
  muter_account_id  UUID NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  muted_account_id  UUID NOT NULL REFERENCES accounts(id) ON DELETE CASCADE,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (muter_account_id, muted_account_id),
  CONSTRAINT chk_no_self_mute CHECK (muter_account_id <> muted_account_id)
);

-- Featured Picks table: curated weekly picks selected by admins.
CREATE TABLE featured_picks (
  id              UUID PRIMARY KEY,
  week_starting   DATE NOT NULL,             -- Monday of the featured week
  patch_id        UUID NOT NULL REFERENCES patches(id) ON DELETE CASCADE,
  position        INTEGER NOT NULL,          -- 1..8 ordering
  curator_note    TEXT,                      -- optional admin one-liner
  added_at        TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX idx_featured_week_position ON featured_picks(week_starting, position);
CREATE INDEX idx_featured_week ON featured_picks(week_starting);

-- Account additions for bio and privacy preferences.
ALTER TABLE accounts
  ADD COLUMN bio VARCHAR(280) DEFAULT NULL,
  ADD COLUMN likes_public BOOLEAN NOT NULL DEFAULT FALSE,
  ADD COLUMN follows_public BOOLEAN NOT NULL DEFAULT FALSE,
  ADD COLUMN suspended BOOLEAN NOT NULL DEFAULT FALSE;

-- Denormalized counts for instant performance.
ALTER TABLE patches ADD COLUMN like_count INTEGER NOT NULL DEFAULT 0;
ALTER TABLE recordings ADD COLUMN like_count INTEGER NOT NULL DEFAULT 0;
ALTER TABLE accounts ADD COLUMN follower_count INTEGER NOT NULL DEFAULT 0;
ALTER TABLE accounts ADD COLUMN following_count INTEGER NOT NULL DEFAULT 0;
```

## 2. API Surface Additions

### Likes

- `POST /api/v1/likes` (Body: `{ "target_kind": "patch" | "recording", "target_id": "UUID" }`): Creates a like. Rate limited.
- `DELETE /api/v1/likes/:target_kind/:target_id`: Removes a like.
- `GET /api/v1/likes/status?target_kind=...&target_id=...`: Returns `{ "liked": bool }` for the current user.

### Follows

- `POST /api/v1/follows/:account_id`: Follows an account.
- `DELETE /api/v1/follows/:account_id`: Unfollows an account.
- `GET /api/v1/accounts/:account_id/followers`: Public if `follows_public` is true, or if requester is the owner.
- `GET /api/v1/accounts/:account_id/following`: Public if `follows_public` is true, or if requester is the owner.

### Blocks + Mutes

- `POST /api/v1/blocks/:account_id`: Bidirectional block. Auto-unfollows in both directions.
- `DELETE /api/v1/blocks/:account_id`: Unblocks.
- `GET /api/v1/blocks/me`: Returns list of blocked accounts.
- `POST /api/v1/mutes/:account_id`: Mutes (feed-only filter).
- `DELETE /api/v1/mutes/:account_id`: Unmutes.
- `GET /api/v1/mutes/me`: Returns list of muted accounts.

### Activity Feed

- `GET /api/v1/feed` (Query: `cursor=TIMESTAMPTZ_ISO_STR,UUID`, `limit=24`): Returns chronological feed of followed users' public patches/recordings.

### Featured

- `GET /api/v1/featured`: Current week's picks.
- `GET /api/v1/featured/history` (Query: `limit=12`, `offset=0`): Paginated past weeks.
- `POST /api/v1/admin/featured` (Gated by admin key, Body: `{ "picks": [{ "patch_id": UUID, "position": int, "curator_note": str }] }`): Configures current week's picks.
- `DELETE /api/v1/admin/featured/:id` (Gated by admin key): Deletes a featured pick.

### Profile

- `GET /api/v1/profiles/:account_id`: Public profile data, public counts, follow status.
- `PATCH /api/v1/account/me` (Body: `{ "bio": str, "likes_public": bool, "follows_public": bool }`): Updates profile/privacy preferences.
- `GET /api/v1/profiles/:account_id/liked`: Returns public liked patches/recordings if `likes_public` is true (or if requester is the owner).

### Admin

- `POST /api/v1/admin/accounts/:account_id/suspend`: Suspends account (hide public content, block login).
- `DELETE /api/v1/admin/accounts/:account_id/suspend`: Removes suspension.

## 3. Counts: Trigger vs Job

We **strongly recommend database triggers** to maintain the denormalized count columns (`like_count`, `follower_count`, `following_count`):

- **Real-time Accuracy:** User trust relies on immediate feedback (e.g., hitting a heart incrementing the number instantly). Batch jobs introduce latency, causing jarring UI behavior where a clicked heart goes back to the old number until the next refresh.
- **Data Integrity:** PostgreSQL and SQLite triggers guarantee atomicity. Row modifications automatically increment/decrement the denormalized counts on the parent records, preventing desynchronization.
- **Production vs Test Portability:** Simple SQL triggers work flawlessly on both Postgres (prod) and SQLite (test suite).
- **Trigger Logic:**
  - `AFTER INSERT` on `likes` → `UPDATE patches/recordings SET like_count = like_count + 1 WHERE id = NEW.target_id`
  - `AFTER DELETE` on `likes` → `UPDATE patches/recordings SET like_count = like_count - 1 WHERE id = OLD.target_id`
  - `AFTER INSERT` on `follows` → Increment `follower_count` for `followed_account_id`, `following_count` for `follower_account_id`.
  - `AFTER DELETE` on `follows` → Decrement counts accordingly.

## 4. Authorization Model Extension (Block-Aware Filtering)

To satisfy the **bidirectional content invisibility** requirement, we implement a centralized, reusable SQLAlchemy filter helper. This helper is applied dynamically to all list and detail queries.

```python
def apply_block_filter(query, viewer_account_id: UUID | None):
    """
    Extends the SQLAlchemy query to exclude content belonging to or created by
    accounts that have blocked the viewer, or that the viewer has blocked.
    """
    if not viewer_account_id:
        return query

    # Exclude content where the creator is in the set of accounts the viewer blocked,
    # or the creator is in the set of accounts that have blocked the viewer.
    block_subq = (
        select(blocks.c.blocked_account_id)
        .where(blocks.c.blocker_account_id == viewer_account_id)
    )
    blocker_subq = (
        select(blocks.c.blocker_account_id)
        .where(blocks.c.blocked_account_id == viewer_account_id)
    )

    # Apply to accounts, patches, recordings, or any query joining them
    # (implemented in Python by checking model entity types and applying .where)
```

** viewer-aware endpoints**:

- **Gallery**: Excludes patches from blocked/blocker users.
- **Feed**: Excludes patches/recordings from blocked/blocker users.
- **Profiles**: Fetching a blocked/blocker profile returns a strict `404 Not Found` (to avoid leaking presence).
- **Similar-Patches**: Excludes blocked content.

## 5. Feed Pagination

Pagination is **strictly cursor-based** to handle dynamic feeds without duplication or omission:

- **Cursor format**: `(created_at, id)` represented as a base64 or comma-separated string (e.g. `2026-05-29T19:04:00Z,3fa85f64-5717-4562-b3fc-2c963f66afa6`).
- **Ordering**: Strict `created_at DESC, id DESC`.
- **Query Structure**:
  ```sql
  WHERE (created_at < cursor_time)
     OR (created_at = cursor_time AND id < cursor_id)
  ```
- **Page size**: 24 items.

## 6. Feed Performance

- **Low/Mid Scale (≤10k Follows per user)**: A direct index-backed SQL query is highly performant. A join between `patches`/`recordings` and `follows` filtered by the viewer's `follower_account_id` takes `<10ms` on indexed tables.
- **Scaling Threshold**: When active users follow >10,000 accounts, or the query volume exceeds 10,000 requests/sec, the index scan on the join table degrades. At this scale, we would shift to a denormalized fan-out (e.g., Redis-backed inbox queues updated on patch publish).
- **v2.0 Strategy**: The relational index-backed approach is highly optimized and perfectly sufficient for now. Defer Redis fan-out.

## 7. Featured Curation

- **Curation window**: Admin sets current week's picks via `/admin` featured panel.
- **Archiving/Rotation**: The rotation occurs on Mondays at 00:00 UTC.
- **Query Logic**: The API resolves `week_starting = current_monday` (calculated dynamically as `now() - timedelta(days=now().weekday())`).
- **Archive History**: `/api/v1/featured/history` allows browsing previous weeks by sending chronological Monday dates.

## 8. Block + Mute Semantics

- **Block (Bidirectional Hiding, One-Way Interaction Prevention)**:
  - If Account A blocks Account B:
    - Auto-unfollow in both directions (`DELETE FROM follows WHERE (follower_account_id = A AND followed_account_id = B) OR (follower_account_id = B AND followed_account_id = A)`).
    - Prevents B from following, liking, or reporting A.
    - Prevents B from viewing A's profile page (raises `404`).
    - Bidirectionally hides content in feeds, gallery, and similarity lists.
    - Existing likes by B on A's content are kept in DB (to avoid breaking like integrity or letting B know they were blocked), but are completely hidden from UI count and lists when A views them.
- **Mute (Feed-Only Filter)**:
  - If Account A mutes Account B:
    - B's public content is filtered out of A's chronological feed.
    - A can still view B's profile, follow B, and like B's patches. B can interact with A normally. Transparent to B.

## 9. Account Suspension

Suspended accounts are restricted from the platform:

- **State**: `suspended = True` on `accounts`.
- **API restriction**: Any request authenticated under a suspended account's sessions returns `403 Forbidden` with `"suspended"` reason and `"support@annealmusic.com"` contact info.
- **Visibility**: All patches, recordings, and profile details of a suspended account are treated as private/hidden from all search, gallery, feed, and direct API endpoints.
- **Admin**: An admin can suspend/unsuspend any account by ID using the admin key.

## 15. Rate Limits

We implement strict rate limits to prevent script-based follow, block, and like spam:

- **Likes**: `100 / hour` per account/IP.
- **Follows**: `60 / hour` per account/IP.
- **Blocks**: `30 / hour` per account/IP.
- **Justification**: Humans do not realistically click more than 100 likes or follow 60 accounts per hour during organic sound-making sessions. Lower thresholds prevent API abuse while ensuring authentic social metrics.

## 18. Smoke Matrix Preview

We will execute the following core scenarios to verify completeness:

1. **Anon Like**: Tap heart as anon → counts increment, tooltip nudges sign-in.
2. **Account Claiming**: Sign in after anon-like → like persists and links to the account.
3. **Follow Mechanics**: Follow account B → counts update, B shows in feed.
4. **Feed chronological order**: Publish patch on B → immediately displays at top of B's followers' feeds.
5. **Bidirectional Block**: Block account B → A and B can no longer see each other's patches, profiles (404), or search results.
6. **Mute Filter**: Mute account B → B's patches disappear from feed but remain visible in gallery/profiles.
7. **Curator featured week rollover**: Set system clock or mock date → rollover shifts pick views automatically.

## 19. Risks & Tradeoffs

- **Discipline on Engagement**: Social loops can easily degenerate into dopamine hooks. To preserve "Calm-by-Design", we have no push notifications, no email digests, no badges, and no sorting biases toward likes by default.
- **Block Leaks via Embedding**: Ensure blocked content does not sneak into similarity embeddings or recommendation carousels. Verified by block filtering inside the similar-patches endpoint.
