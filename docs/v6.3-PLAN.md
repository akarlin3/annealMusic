# Implementation Plan — AnnealMusic v6.3 · Progress Tracking + Next-Lesson Picker

This document is the **Checkpoint 0** deliverable for v6.3. It finalizes the design before any
code is written. v6.3 adds two things on top of the v6.0–v6.2 Learn surface:

1. **Progress tracking** — per-user, cross-device lesson state (`not_started` / `in_progress` /
   `completed` / `abandoned`) with per-step state so a user can pause mid-lesson and resume,
   including on another device via their v1.3 account.
2. **LLM-driven next-lesson selection** — a two-stage picker (deterministic candidate filter →
   Haiku ranking) that surfaces 1–3 recommended next lessons after a completion or on a `/learn`
   return-after-gap, plus an onboarding picker for brand-new users.

> Branch: `claude/v6.3-progress-recommendations-8OXyY`
> Prereqs shipped: v6.0 (Learn surface, curriculum browser, iframe lesson player, step types),
> v6.1 (LLM lesson-generation pipeline + per-step caching), v6.2 (audio-clip library + step type),
> v1.3 (accounts + cross-device anon-claim), v4.5 (private per-user Session History — the closest
> existing precedent for a private, account-gated, anon→authed, calm-by-design user-data surface).

Five facts that drive the design below:

1. **There is already a battle-tested private-user-data surface** (`me_sessions.py`). v6.3 is "v4.5
   again, for lessons" plus an LLM ranker. We reuse its account-gating, anon-merge, and
   single-source-of-truth-stats discipline verbatim rather than inventing a new shape.
2. **Anon identity + claim already works cross-device.** We don't build new auth; anon progress is
   client-side localStorage, and on sign-in we POST the buffered progress to the server keyed to the
   now-claimed account (§4). No server-side anon `lesson_progress` rows for un-claimed anons.
3. **Difficulty is a 3-value enum, not a number.** "Within ±1 level" (§5 stage 1) maps to an ordinal
   `intro=0, intermediate=1, advanced=2`; "recently completed" anchors the band.
4. **Recommendation state is volatile**, so the v6.1 immutable content-cache model doesn't fit. A
   small per-process TTL cache keyed on a hash of the compact progress summary (§7) is correct and
   cheap.
5. **`abandoned` is implicit, never written by a mutation.** It is *computed* for the picker
   (>30 days since `last_active_at` while `in_progress`). The stored `state` stays `in_progress` so
   the user can always resume. This keeps the state machine honest and avoids a background job.

## 1. Data model

New table `lesson_progress`, one row per `(user_id, lesson_id)`. Refined from the brief: composite
PK, `step_actions` is a **bounded** JSON log, all timestamps `TIMESTAMPTZ`.

```sql
CREATE TABLE lesson_progress (
  user_id        UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  lesson_id      UUID NOT NULL REFERENCES lessons(id) ON DELETE CASCADE,
  state          TEXT NOT NULL DEFAULT 'not_started',  -- not_started|in_progress|completed|abandoned*
  current_step_position INTEGER NOT NULL DEFAULT 0,
  scroll_ratio   REAL NOT NULL DEFAULT 0,              -- 0..1 within the current step body (resume)
  started_at     TIMESTAMPTZ,
  last_active_at TIMESTAMPTZ,
  completed_at   TIMESTAMPTZ,
  step_actions   JSONB NOT NULL DEFAULT '[]'::jsonb,   -- bounded log, see below
  reflection_text TEXT,                                 -- private; never sent to LLM, never published
  created_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, lesson_id),
  CONSTRAINT ck_lesson_progress_state
    CHECK (state IN ('not_started','in_progress','completed')),  -- 'abandoned' is computed, never stored
  CONSTRAINT ck_lesson_progress_scroll CHECK (scroll_ratio >= 0 AND scroll_ratio <= 1)
);
CREATE INDEX idx_lesson_progress_user  ON lesson_progress(user_id, last_active_at DESC);
CREATE INDEX idx_lesson_progress_state ON lesson_progress(user_id, state, last_active_at DESC);
```

Notes / refinements vs. the brief:

- **`scroll_ratio` added** (brief §"Pausing+resuming" requires scroll position; storing a 0..1 ratio
  is device-width-independent and survives layout changes better than a pixel offset).
- **CHECK omits `abandoned`** — abandonment is implicit (fact 5). The `state` enum in the DB is the
  three writable states; the picker derives `abandoned` in code via `derive_effective_state()`.
- **`step_actions` is bounded**: each entry `{step_position:int, action:'started'|'completed'|'skipped', ms:int, at:iso8601}`.
  Hard cap **200 entries** (server truncates oldest on write) — enough signal for the ranker, no
  unbounded growth. No free-text; no PII. `reflection_text` is *separate* and never enters
  `step_actions`.
- **SQLAlchemy model** `LessonProgress(Base)` in `api/app/models.py` using `GUID()`, `JSONType()`,
  `DateTime(timezone=True)`, composite `PrimaryKeyConstraint`, mirroring `SessionPlay`.
- **`track_progress` is a view, not a table** — computed on read (count of `completed` lessons /
  count of visible lessons per track) by the per-track summary endpoint. Single computation site
  (`compute_track_progress`, the calm-by-design "stats computed once" analog of `compute_stats`).

State machine (writable transitions only):

```
not_started ──(open lesson / first step started)──▶ in_progress
in_progress ──(reach last step + acknowledge)─────▶ completed
in_progress ──(resume)────────────────────────────▶ in_progress (idempotent, bumps last_active_at)
completed   ──(re-open)────────────────────────────▶ completed (re-reading never un-completes)
```
`abandoned` overlays `in_progress` at read time when `now - last_active_at > 30d`.

## 2. API surface

All routes account-or-anon aware via `Identity`; **cross-device** features require `account_id`.
Refined from the brief (added explicit single-lesson GET; clarified verbs):

```
POST   /api/v1/lesson-progress                 -- upsert progress for one lesson (heartbeat/pause/complete)
GET    /api/v1/lesson-progress                 -- list caller's progress rows (for browser checkmarks)
GET    /api/v1/lesson-progress/{lesson_id}     -- single lesson's progress (resume fetch on lesson open)
GET    /api/v1/lesson-progress/me/track/{slug} -- per-track summary (completed/total + in-progress)
POST   /api/v1/lesson-progress/import          -- anon→authed bulk import on first sign-in (§4)
POST   /api/v1/recommendations/next            -- next-lesson picks (stage1+2+3); body: {context: 'completion'|'arrival', just_completed_lesson_id?}
```

- **Router files**: `api/app/routers/lesson_progress.py` (prefix `/api/v1/lesson-progress`) and
  `api/app/routers/recommendations.py` (prefix `/api/v1/recommendations`), registered in
  `api/app/main.py`. (The brief's directory names `api/v1/lesson_progress/` map to single router
  modules, consistent with the existing one-file-per-router layout.)
- **Schemas** in `api/app/schemas.py`: `LessonProgressUpsert`, `LessonProgressOut`,
  `LessonProgressListOut`, `TrackProgressOut`, `ProgressImportIn`, `RecommendationRequest`,
  `RecommendationItem`, `RecommendationsOut`. `LessonProgressOut.state` includes the **derived**
  `abandoned`; `RecommendationItem = {lesson_id, slug, title, difficulty, rationale}`.
- **Auth posture**:
  - `POST /lesson-progress` and the GETs work for **both** anon (server stores under the anon
    `user_id` minted by `current_writer`) and account users. *However*, the client keeps **anon
    progress in localStorage only** (privacy: "client-side anon stays client-side") and only writes
    to the server once there's an account. So in practice these endpoints are exercised by
    authenticated callers; the anon code path exists but the client doesn't drive it. (Decision
    recorded in §8.)
  - `/import` and `/me/track` and `/recommendations/next` for completion/arrival all function
    anonymously too, but cross-device persistence is account-only — matching `me_sessions`'
    `_require_account` nudge pattern for the sync prompt.
- **Rate limiting**: reuse `rate_limit` dependency. Progress upsert is high-frequency (heartbeat) →
  generous bucket; `/recommendations/next` is low-frequency → tight bucket (and TTL-cached, §7).

## 3. Pause / resume mechanics

Client-owned, server-persisted. Modules `src/learn/progress/ProgressClient.ts` +
`src/learn/progress/ResumeHandler.ts`, wired into `LessonPlayer.tsx`.

**Capture (pause):**
- `LessonPlayer` already holds `currentStepIndex` (`src/learn/LessonPlayer.tsx:14`). On step change,
  on `visibilitychange → hidden`, and on `beforeunload`, `ProgressClient.saveProgress()` records
  `{current_step_position, scroll_ratio, state:'in_progress', step_actions delta}`.
- `scroll_ratio` = `scrollTop / (scrollHeight - clientHeight)` of `.player-step-body`, clamped 0..1.
- Writes are **debounced (~1.5s)** and **coalesced**; the `beforeunload` write uses
  `navigator.sendBeacon` so a closing tab still flushes. Heartbeat updates `last_active_at`.

**Persistence tiers:**
- **Account user** → POST to `/api/v1/lesson-progress` (source of truth; cross-device).
- **Anon user** → `localStorage` key `am_learn_progress` (a `{lessonId → progressRecord}` map),
  device-scoped only. *No* server write (privacy). `ProgressClient` abstracts the tier so callers
  don't branch.

**Restore (resume):**
- On lesson open, `ResumeHandler.resume(lessonId)`:
  1. account → `GET /api/v1/lesson-progress/{lesson_id}`; anon → read localStorage.
  2. If `state==='in_progress'` and `current_step_position>0`, set `LessonPlayer` initial
     `currentStepIndex = current_step_position`, then after the step body renders, restore
     `scrollTop` from `scroll_ratio`.
  3. If `completed`, open at step 0 (re-reading is fine), show a quiet "you've completed this" line —
     **no** celebratory styling.
- Resume is silent: no modal, no "Welcome back!" nag. At most a low-contrast "Resuming where you
  left off" inline note that fades.

## 4. Anon → authed migration of localStorage progress on sign-in

Mirrors the v4.5 anon-claim flow (`account.py:/me/claim` + `owned_anon_ids`), but progress lives in
localStorage (not anon DB rows), so the migration is a one-shot **client-driven import**:

1. User signs in (existing v1.3 / AuthProvider flow). `AuthProvider` already performs `/me/claim` for
   the device's `anon_id`.
2. After claim succeeds, the Learn client checks `localStorage.am_learn_progress`. If non-empty and
   not yet imported (guard flag `am_learn_progress_imported`), it calls
   `POST /api/v1/lesson-progress/import` with the buffered records.
3. **Server merge policy** (`import` handler): for each `(lesson_id)` in the payload, upsert into
   `lesson_progress` for the account's `user_id` using a **max-merge**:
   - `state`: keep the more-advanced of {existing server state, imported state}
     (`completed > in_progress > not_started`); never downgrade `completed`.
   - `current_step_position`: `max(existing, imported)`.
   - `last_active_at`: `max(...)`; `completed_at`: prefer existing non-null else imported.
   - `step_actions`: concatenate then truncate to the 200-entry cap, sorted by `at`.
   - `reflection_text`: keep existing if present, else take imported (never concatenate private text).
4. On 2xx, client sets `am_learn_progress_imported=1` and **clears** `am_learn_progress` (the server
   is now source of truth). Idempotent: re-running import is a no-op due to max-merge + guard.
5. **Gentle sync nudge** (calm-by-design): while anon, a single dismissible "Sign in to keep your
   progress across devices" line appears **at most once per session** (session-storage flag
   `am_learn_sync_nudge_shown`), never recurring, never guilt-framed — identical discipline to the
   v4.5 history nudge.

Switching devices while anon loses progress (expected, documented). The nudge is the only mitigation;
we never silently upload anon data.

## 5. Next-lesson picker algorithm

Endpoint `POST /api/v1/recommendations/next` → service `api/app/services/next_lesson_ranker.py`.
Three stages.

### Stage 1 — candidate filtering (deterministic, pure Python, no LLM)
Input: caller progress rows + full visible curriculum (reuse `learn.py` visibility helpers).
Filters, in order:
1. **Visible & not archived & not already `completed`** by the user.
2. **Prerequisites satisfied** — every UUID in `Lesson.prerequisites` is in the user's completed set.
3. **Track scope** — same track as the just-completed/last-active lesson, OR a "related track"
   (related = adjacency table; v6.3 uses a static `RELATED_TRACKS` map seeded from the 5-track
   curriculum, e.g. *synthesis-fundamentals ↔ composition-technique*). Cross-track only if same-track
   candidates < 3.
4. **Difficulty band** — `ordinal(difficulty)` within **±1** of the anchor (the difficulty of the
   most-recently completed lesson; for onboarding/no-completions the anchor is `intro`).
5. Sort by `(track match desc, position asc)`; **cap at 8 candidates** sent to the LLM (cost + focus).

If Stage 1 yields **0** candidates (e.g. user completed everything reachable) → return a calm empty
state ("You've explored everything here for now") with **no** LLM call. If it yields **1–3**, we may
still call the LLM for rationales but can short-circuit to deterministic order if the LLM is
unavailable (graceful degradation — never block the UI on the LLM).

### Stage 2 — LLM ranking (Haiku 4.5)
The ranker builds a **compact** progress summary (metadata only — no reflection text, no PII; §"Privacy"):
- completed lesson IDs (+ titles), current track slug + aggregate completion (`compute_track_progress`),
- recent `step_actions` digest (e.g. "paused on step 3 of X; spent 4m on a prompt step; skipped a demo"),
- time since last session (bucketed: "same day" / "a few days" / "weeks"),
- the Stage-1 candidate list (`lesson_id`, `title`, `difficulty`).

Full prompt draft in §6. LLM returns strict JSON: ordered `lesson_id`s (subset of candidates) +
a 1-sentence rationale each. **Validation**: every returned `lesson_id` must be in the candidate set
(drop hallucinations); truncate to ≤3; if JSON invalid after one retry, fall back to deterministic
Stage-1 order with neutral rationales. Reuse the validate-then-retry≤1 discipline from
`lesson_generation.py`.

### Stage 3 — presentation
`POST` returns `RecommendationsOut{items:[{lesson_id,slug,title,difficulty,rationale}], source:'llm'|'deterministic'|'onboarding'}`.
Client `src/learn/recommend/NextLessonPicker.tsx` renders **1–3 calm cards**: title + the 1-sentence
"why this next." No counts, no "next up #1", no pressure. Always accompanied by a "browse all
lessons" escape hatch so the user is never funnelled. Shown (a) on the lesson summary screen after
completion, and (b) on `/learn` arrival after a gap (see §9 for onboarding variant).

## 6. LLM prompt for ranking (full draft)

**Model**: `claude-haiku-4-5`, `temperature=0.2`, `max_tokens≈600`. Built in `next_lesson_ranker.py`.

**System prompt:**
```
You are the lesson-sequencing assistant for AnnealMusic, a calm, non-gamified tool for learning
ambient-music synthesis and listening. Your only job is to choose, from a provided candidate list,
the 1–3 lessons that best maintain pedagogical flow for this specific learner, and to write one
short, warm, non-coercive sentence per pick explaining why it follows naturally.

Hard rules:
- Recommend ONLY lessons whose lesson_id appears in the candidate list. Never invent IDs.
- Recommend at most 3. Fewer is fine if fewer fit well.
- Prefer lessons that build directly on what the learner just completed or has been working through.
- If the learner's recent activity suggests they struggled with or shallowly completed a prerequisite
  topic (e.g. they skipped steps, or spent very little time, on a foundation the candidate depends on),
  AVOID jumping ahead; prefer a consolidating or gentler-slope lesson instead.
- Respect difficulty: do not leap to 'advanced' right after a struggled 'intro'.
- Tone: calm and descriptive. NO motivational language, NO streaks/goals/urgency, NO "keep it up",
  NO "don't lose momentum". One plain sentence describing why the lesson fits — that is all.

Output ONLY valid JSON, no prose, in exactly this shape:
{"recommendations":[{"lesson_id":"<uuid from candidates>","rationale":"<one sentence>"}]}
```

**User prompt (templated):**
```
LEARNER PROGRESS (metadata only):
- current_track: {track_slug} ({completed_in_track}/{total_in_track} lessons completed)
- completed_lessons: [{ "id": "...", "title": "..." }, ...]   # may be empty
- just_completed: {title or "none — arriving at /learn"}
- time_since_last_session: {"same day" | "a few days" | "a few weeks" | "first visit"}
- recent_step_signals: {e.g. "spent ~4 min on a prompt step in 'FM Ratios'; skipped one demo step;
                        paused mid-lesson on step 3 of 6"}   # derived from step_actions, no text content

CANDIDATE LESSONS (choose only from these):
[ { "lesson_id": "...", "title": "...", "difficulty": "intro|intermediate|advanced" }, ... ]

Recommend 1–3 lessons that maintain pedagogical flow and respect the learner's recent struggles.
Avoid recommending lessons whose prerequisites the learner only completed shallowly. Prefer lessons
that build on what was just learned. Return JSON only.
```

Privacy guarantee baked into the builder: `recent_step_signals` is generated from `step_actions`
counts/durations only; `reflection_text` is **never** read by the ranker. A unit test asserts the
prompt string contains no reflection content.

## 7. Caching strategy

- **Per-process TTL cache** (`functools`-style dict with timestamps, or a tiny `TTLCache` helper in
  `next_lesson_ranker.py`), **5-minute TTL**, keyed on
  `sha256(user_id + state_fingerprint)` where `state_fingerprint` = stable hash of
  `(sorted completed lesson_ids, current_track, last_active bucket, candidate id set)`.
- Identical state within 5 min → return cached `RecommendationsOut` **without** an LLM call (brief
  requirement). A new completion or a new `last_active_at` bucket changes the fingerprint → cache
  miss → fresh ranking.
- **Why not the v6.1 `AIGeneration` cache?** That cache is for *immutable content* keyed by lesson
  spec. Recommendations depend on volatile per-user state and should expire quickly; a DB row per
  (user, state) would accumulate noise. In-process TTL is correct and matches the "<10 calls/day"
  cost envelope (Haiku 4.5 ≈ cents/month/learner). Cache is best-effort (lost on restart — fine).
- The deterministic Stage-1 result is cheap and **not** cached (recomputed each call); only the LLM
  ranking is cached.

## 8. Calm-by-design review (v6.3) — the highest engagement-loop-risk slice

This slice is where AnnealMusic could most easily drift into a habit-forming product. Explicit guardrails:

- **Progress is descriptive, never motivational.** The curriculum browser shows a quiet completed
  checkmark per lesson and "N of M lessons" per track — and nothing else. **No** percentage-as-pressure
  framing ("you're 47% through"), **no** progress bars that imply an obligation to fill them,
  **no** "you're almost there!".
- **No streaks / no counts that gamify.** No "days in a row", no session streak, no XP, no levels,
  no badges, no leaderboards, no completion celebrations (no confetti/sound) — consistent with the
  v6.0 checklist. `compute_track_progress` is the **single** place track aggregates are derived
  (the `compute_stats` discipline) so framing can't drift between surfaces.
- **The picker is an offer, not a funnel.** 1–3 cards + an always-present "browse freely" escape.
  No "recommended for you" infinite feed, no autoplay-next-lesson, no "continue your streak", no
  ranking-as-competition. The picker runs at most once per session-ish (completion or arrival), never
  as a persistent nag.
- **No outbound nudges.** No emails, push, or reminders about lessons — ever (these are in the
  permanent "never" list). The only nudge is the **single, dismissible, once-per-session** "sign in
  to keep your progress" line (§4), matching the v4.5 history nudge exactly.
- **Abandonment is invisible to the user.** The 30-day `abandoned` derivation is *picker-internal*.
  Users are never told a lesson "expired" or shown a guilt-framed "you abandoned this".
- **Reflection text is private and never leaves the user's column** — not published, not shown to
  others, not sent to the LLM.
- **CI gate extended.** Add `src/learn` to `ROOTS` in `src/test/calm-by-design.test.ts` (currently
  only `src/history`/`src/library`) so the new progress + recommend UI is lexically scanned for
  banned tokens. This is a CP2/CP3 hard deliverable.
- **Decision recorded:** anon progress stays client-side (localStorage), never silently uploaded —
  the server only ever receives progress for an authenticated, consenting account (or the explicit
  one-shot import the user triggers by signing in).

The v6.3 checklist itself is appended to `docs/CALM_BY_DESIGN.md` §4 at CP3.

## 9. Onboarding picker design

For a brand-new user (no `completed` lessons):
- Stage 1's anchor is `intro`; candidates are **intro lessons across all five tracks** (one or two
  lead intro lessons per track, by `position`).
- The picker presents two calm choices side by side:
  1. **Pick a track** — a small row of the five tracks, each with its one-line description; choosing
     one opens that track's first intro lesson.
  2. **Start "Intro to AnnealMusic"** — a single guided lesson that touches all five tracks lightly
     (this lesson is authored/seeded as part of CP3; slug `intro-to-annealmusic`, track =
     a neutral/`getting-started` track or the synthesis-fundamentals intro slot — finalized in CP3
     seed).
- The LLM is **not** strictly required for onboarding (there's no history to reason over); onboarding
  returns `source:'onboarding'` deterministically. Optionally the LLM can pick which 3 of the 5 intro
  lessons to surface first, but the default is deterministic + calm.
- Same calm rules: no "complete onboarding to unlock", no progress meter, no pressure to finish.

## 11. Risks & mitigations

| Risk | Likelihood | Mitigation |
| --- | --- | --- |
| **Engagement-loop creep** — picker/progress quietly becomes a streak/leaderboard/funnel. | Med | §8 guardrails; CI lexical gate extended to `src/learn`; "stats computed once"; picker is an offer with a permanent browse-escape; v6.3 calm checklist in CI/review at CP3. **Primary risk.** |
| **LLM produces strange/irrelevant suggestions** or hallucinated IDs. | Med | Stage-1 deterministic filter constrains the option set; Stage-2 output validated against candidate IDs (drop unknowns); retry≤1; graceful fallback to deterministic order; ≤3 cap. |
| **Resume restores wrong step / lost scroll** after lesson content edits. | Low | `current_step_position` clamped to current `stepsCount`; `scroll_ratio` (0..1) is layout-resilient; if position out of range, open at step 0. |
| **Anon→authed merge data loss or duplication.** | Low | Idempotent max-merge keyed on `(user_id,lesson_id)`; import guard flag; never downgrades `completed`; covered by `test_progress_import_merge.py`. |
| **Privacy leak** — reflection text or PII reaching the LLM/logs. | Low/High-impact | Ranker reads only `step_actions` metadata; unit test asserts no reflection text in prompt; documented in `PRIVACY.md`; LLM payload review at CP3. |
| **Heuristic drift** — progress/abandoned/track-% computed in multiple places. | Med | `progress_state.py` (`derive_effective_state`, `compute_track_progress`) is the single source; client renders server values verbatim. |
| **Cost blow-up** from frequent ranking calls. | Low | 5-min TTL cache; called only on completion/arrival; <10/day/learner; Haiku 4.5 ≈ cents/month. |
