# Clinical Study Export & Reproducibility Plan (v7.5)

This plan defines the architecture, data models, APIs, and CLI integrations for making clinical and sonification studies fully reproducible via self-contained export bundles.

---

## 1. Final Bundle Structure

A study export bundle is a `.zip` archive containing the following layout:

```
study-export-bundle/
├── manifest.json            # Top-level study metadata, locks, and file registry
├── README.md                # Auto-generated researcher/reviewer instructions
├── CITATION.bib             # BibTeX reference with DOIs and AnnealMusic versions
├── stimuli/                 # Stimulus configurations and states
│   ├── patch_[id].json
│   ├── piece_[id].json
│   ├── listening_session_[id].json
│   └── sonification_[id].json
├── protocols/               # Research/experiment configurations
│   ├── clinical_protocol_[id].json
│   └── experiment_[id].json
├── data/                    # [OPTIONAL] Anonymized session records & signals
│   ├── clinical_session_records.json
│   └── biosignal_streams/
│       └── stream_[channel_id].csv
├── scripts/                 # Analysis routines and execution environments
│   ├── user_script_[id].py
│   └── requirements.txt     # Pinned Python package dependencies
└── audio/                   # Pinned audio assets and references
    ├── audio_clip_[slug].opus
    └── calibrated_stimulus_[hash].wav
```

## 2. Manifest Schema (`manifest.json`)

The `manifest.json` is the single source of truth for verification. It has the following schema:

```json

  "study_id": "uuid",
  "version_id": "uuid",
  "version_label": "v7.5.0",
  "reproducibility_level": "bytes-identical | perceptually-identical | statistically-equivalent",
  "includes_subject_data": true,
  "annealmusic_version": "7.5.0",
  "created_at": "2026-05-31T03:10:00Z",
  "study_metadata": {
    "title": "Study Title",
    "slug": "study-slug",
    "abstract": "Abstract text...",
    "concept_doi": "10.5281/zenodo.xxxxxx",
    "funding_sources": []
  },
  "version_locks": {
    "engine": {
      "name": "physical engine",
      "version": "1.2.3",
      "pyodide_version": "0.26.0"
    },
    "python_environment": {
      "cpython_version": "3.11",
      "requirements": {
        "numpy": "1.26.4",
        "scipy": "1.12.0"

    "assets": {
      "bell_library_sha256": "abcdef...",

    }
  },

      "path": "stimuli/patch_abc.json",
```

## 3. Version Lock Policy

To protect studies from **heuristic-drift** and changes in external packages, the export process pins:

1. **AnnealMusic Core Version**: Stored in `manifest.json` as `annealmusic_version` (e.g., `7.5.0`).
2. **Schema & Engine Versions**: Schema versions are verified against our schema manifests (e.g., `manifest.v20.json`).
3. **Python Runtime**: Pinned to CPython `3.11` and Pyodide `0.26.0`.
4. **Libraries**: Pinned package versions (e.g., `numpy==1.26.4`, `scipy==1.12.0`) in `scripts/requirements.txt`.
5. **Asset Hashes**: Calibrated audio assets are indexed with SHA-256 hashes to guarantee they are identical.

---

## 4. Reproducibility Level Definitions

- **`bytes-identical`**: Strict execution parity. The input JSON states + same random seed + same engine version yields sample-accurate, bit-identical rendered WAV files. Verified via SHA-256 match of the outputs.
- **`perceptually-identical`**: Small floating-point drift across CPU architectures or runtimes (e.g., Node vs Browser Web Audio) is allowed, but the audio is perceptually equivalent (checked via peak amplitude and zero-crossing rate variance limits).
- **`statistically-equivalent`**: For stochastic stimuli (e.g., Kuramoto oscillators with null seeds, random walks), sample-level parity is impossible. Parity is defined by matching statistical descriptors (e.g., average spectral centroid, power distribution).

---

## 5. Anonymization Techniques & IRB Compliance

When exporting subject data, privacy must be strictly protected:

1. **Subject IDs**: Replaced with clean UUIDs (e.g., `subj_001` becomes a random `uuid4`).
2. **Relative Timestamps**: Absolute timestamps (`started_at`, `completed_at`) are removed. All events in datalogs and biofeedback/biosignal channels are shifted to relative offsets in seconds from the session's start (`0.000s`).
3. **Differential Privacy**: PIs can toggle differential privacy on subject response data, adding SciPy-based Laplace noise scaled to clinical threshold sensitivity.
4. **IRB Attestation Flow**: PIs must actively sign a compliance check confirming they have reviewed the data preview and have IRB approval to distribute anonymized records.

---

## 6. Reproducer Mode Design

A dedicated `/reproduce` route acts as the auditor:

1. **Validate**: Verifies `manifest.json` schemas, referenced files, and file hashes.
2. **Load & Re-render**: Loads stimulus parameters into the active audio engine, triggers a silent batch render, and compares the resulting audio hash/characteristics with the bundled audio.
3. **Run Analysis**: Spawns a lightweight python executor (using CPython locally or Pyodide in-browser) to run the scripts in `scripts/` against the anonymized database records in `data/`.
4. **Discrepancy Reporting**: Generates a visually beautiful validation checklist flagging any floating-point drifts, script exceptions, or missing files.

---

## 7. CLI Integration

Extends the current `annealmusic` commander CLI tool:

- `annealmusic export <study_id> -o <output_zip> [--include-data]`: Hits the export API and streams the zipped bundle.
- `annealmusic validate <bundle.zip>`: Unpacks and runs a schema and hash verification script.
- `annealmusic reproduce <bundle.zip>`: Unpacks, runs re-rendering, executes analysis scripts, and reports correctness.

## 8. Module Plan

### Frontend (`src/studies/export/`)

- `ExportDialog.tsx`: Core export configuration (level, include data toggle, DP toggle, IRB attestation).
- `AnonymizationPreview.tsx`: Collapsible grids displaying the table of subject records _after_ anonymization transformation but _before_ packaging.
- `ReproducerPage.tsx`: Upload portal accepting zip bundles, running validations, and rendering the replication checklist.

### CLI (`tools/cli/src/commands/`)

- `export.ts`: CLI command calling backend export.
- `reproduce.ts`: CLI command running local reproduction pipeline.
- `validate.ts`: Updates `validate` to support `.zip` files.

### Backend (`api/services/`)

- `study_export.py`: Handles ZIP generation, resource gathering, metadata hashing, and file creation.
- `study_validation.py`: Verification logic for ZIP manifests, schema parsing, and execution.

## 9. Risks & Mitigations

- **Risk: Large Bundle Sizes**: Audio stimuli could make bundles hundreds of megabytes.
  _Mitigation_: Pre-render audio strictly in Ogg/Opus mono at calibrated bitrates (e.g., 96kbps), and only render short segments if acceptable.
- **Risk: Sandbox Escapes**: Running arbitrary user python scripts in `reproduce/run` is a security risk.
  _Mitigation_: Execution is isolated via restricted sandboxed execution (or marked as local-only/warn-before-running).
