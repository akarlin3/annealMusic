# AnnealMusic v3.5 · Movements · Implementation Plan

This design document outlines the plan for introducing the **Movement** data model and editor UI, player navigation APIs, per-movement stem export, and CRDT synchronization.

---

## 1. Data Model & Schema updates

### 1.1 Model Definition

A **Movement** represents a named contiguous group of segments with optional metadata:

```typescript
export interface Movement {
  name: string;
  description?: string;
  transition_in_ms?: number;
  transition_out_ms?: number;
  startSegmentIndex: number; // inclusive
  endSegmentIndex: number; // inclusive
}
```

We will add the optional `movements` array to the `Piece` interface:

```typescript
export interface Piece {
  // ...
  movements?: Movement[];
}
```

### 1.2 URL Schema Bump to v13

To support sharing compositions with movements offline, we will bump the schema version to `13` in `src/share/schema.ts` and add `13` to `SUPPORTED_SCHEMA_VERSIONS`.

We will serialize movements sequentially using the `mov` prefix. For each movement `i`:

- `mov{i}.name={name}`
- `mov{i}.desc={description}`
- `mov{i}.in={transition_in_ms}`
- `mov{i}.out={transition_out_ms}`
- `mov{i}.start={startSegmentIndex}`
- `mov{i}.end={endSegmentIndex}`

Example: `&mov0.name=Prelude&mov0.start=0&mov0.end=2&mov1.name=Chaconne&mov1.start=3&mov1.end=4&mov1.in=1000`

### 1.3 Backend Persistence

To persist movements seamlessly on the backend without requiring complex schema migrations on local SQLite vs production PostgreSQL, we will:

1. Extend `PieceCreate`, `PieceUpdate`, and `PieceOut` schemas in `api/app/schemas.py` to include `movements: list[dict] | None = None`, `tempo_bpm: int | None = None`, `notation: list[dict] | None = None`, `variation_seed: int | None = None`, and `variations: list[dict] | None = None`.
2. Save these values inside the existing JSONB `defaults_state` column under hidden keys (e.g. `_movements`, `_tempo_bpm`, `_notation`, `_variation_seed`, `_variations`).
3. Dynamically unpack them back into the root of `PieceOut` in `to_out` in `api/app/routers/pieces.py`.
4. Add support for segment-level `variations` in `PieceSegment` by storing them inside `config["_variations"]` and parsing them in `to_out`.
5. Update the SQL check constraint in Python `CheckConstraint` on `PieceSegment` model to include `'meta-arc'`.

## 2. Player Navigation API

We will add two navigation methods to `PiecePlayer`:

1. `skipToMovement(movementIdx: number): void` - Jumps to the start of the first segment of movement `movementIdx` (`activeSegmentIdx = movement.startSegmentIndex; playheadMs = 0;`).
2. `replayCurrentMovement(): void` - Finds the movement containing the currently active segment index and replays it from the start. If the active segment does not belong to any movement, it jumps to the start of that segment.

---

## 3. Editor UI

We will add a beautiful movements editor row above the segments timeline in `TimelineEditor.tsx`:

1. **Movement Brackets Row**: A row of visual horizontal brackets that span the width of their grouped segments.
2. **Dragging & Edge Resizing**: Dragging the left/right boundaries of a bracket will extend or contract the movement by changing its `startSegmentIndex` / `endSegmentIndex` (without overlapping other movements).
3. **Adding Movements**: A dedicated "+ Add Movement" button above the timeline. Clicking it adds a new movement grouping a default range of segments.
4. **Renaming / Editing**: Clicking a movement bracket will open a beautiful popover or form to edit its name, description, `transition_in_ms`, and `transition_out_ms`, or delete the movement.

---

## 4. Collaborative Syncing (CRDT)

To ensure movements and segments are synchronized among collaborative jam participants:

1. We will add sync capabilities for `piece_movements` and `piece_segments` to `sessionConfigMap` in `src/jam/crdt.ts`.
2. When the leader updates movements/segments locally or plays a piece, the new structure will be synced to other players in the session, allowing all clients to visualize and follow along with the movement structure.

---

## 5. Stem Export integration

We will add a **"per-movement"** stem export mode to `ExportDialog.tsx`:

1. **Per-movement mode toggle**: The user can choose to render either the "Full Piece" or "Per Movement".
2. **Zip archive output**:
   - In "Full Piece" mode, the ZIP contains a single folder with the stems of the full piece.
   - In "Per-Movement" mode, the ZIP will package stems separately for each movement, producing named folders or ZIPs inside the parent ZIP (e.g. `Movement 1 - Prelude/`, `Movement 2 - Chaconne/`).
3. **`session.json` manifest**: The manifest will contain per-movement manifest entries tracking the names, durations, and structural offsets of each movement.

---

## 6. Risks & Mitigation

- **Segment drift / indices changes**: When segments are added, removed, or reordered, movement start/end segment indices could drift.
  - _Mitigation_: We will write defensive hooks to auto-adjust movement boundaries when segments are added or deleted, keeping bounds legal and aligned.
- **Overlap prevention**: Dragging boundaries must be strictly constrained to prevent movement overlaps.
  - _Mitigation_: The drag handler will prevent `endSegmentIndex` of movement `N` from exceeding `startSegmentIndex` of movement `N+1` or falling below its own `startSegmentIndex`.
