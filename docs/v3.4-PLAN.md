# AnnealMusic v3.4 · Variation Rules · Technical Specification & Plan

This document details the architectural design, mathematical models, collaborative networking semantics, offline rendering guidelines, and user interface features to introduce procedural **Variation Rules** to AnnealMusic.

---

## 1. Variation Point Data Model & URL Schema Version 12 Spec

### 1.1 Data Model

In `src/piece/types.ts`, the `Piece` and `PieceSegment` interfaces will be extended to support variation points and seeds:

```typescript
export interface VariationPoint {
  id: string; // Unique identifier for the variation point
  paramKey: string; // Sculpt parameter (e.g. 'rootFreq', 'brightness') or path (e.g. 'randomWalk.driftStrength')
  constraint: {
    type: 'range' | 'enum' | 'relative' | 'correlated';
    min?: number; // for 'range'
    max?: number; // for 'range'
    choices?: number[]; // for 'enum'
    percent?: number; // for 'relative' (e.g., 10 for ±10% deviation)
    targetParam?: string; // for 'correlated' (e.g., tie brightness variation to rootFreq)
    coefficient?: number; // for 'correlated' (e.g., -1.0 for inverse, 1.0 for parallel)
  };
  rule: 'per-play' | 'per-segment' | 'per-render';
}

export interface Piece {
  // ...
  variationSeed: number | null; // Top-level seed. null represents dynamic evaluation per play.
  variations?: VariationPoint[]; // Piece-level defaults_state variation points
  // ...
}

export interface PieceSegment {
  // ...
  variations?: VariationPoint[]; // Segment-specific variation points
}
```

### 1.2 URL Schema Bumps to Version 12

We bump `SCHEMA_VERSION` in `src/share/schema.ts` to `12`.

#### 1.2.1 Serialization Strategy

To avoid URL length bloat, variation points are compressed using flat, query-parameter arrays and indexes:

- Piece-level seed: `var_seed=<seed>` (or `null`)
- Piece-level variations: `v.p=<id>:<paramKey>:<type>:<config>:<rule>;...`
- Segment-level variations: `seg<idx>.v=<id>:<paramKey>:<type>:<config>:<rule>;...`

Where `<config>` is serialized compactly based on constraint type:

- `range`: `<min>,<max>`
- `enum`: `<choices separated by pipe |>`
- `relative`: `<percent>`
- `correlated`: `<targetParam>,<coefficient>`

Example:
`v.p=vp1:brightness:relative:15:per-play;vp2:space:range:0.2,0.8:per-render`

## 2. Constraint Types & Precise Semantics

Let $B$ represent the baseline value of the parameter being varied.

### 2.1 Range Constraint (`range`)

- **Semantics**: The parameter's resolved value $V$ is picked uniformly at random within the interval $[V_{min}, V_{max}]$:
  $$V = V_{min} + R \times (V_{max} - V_{min})$$
  where $R \sim \text{Uniform}(0, 1)$ via the seeded PRNG.
- **Clamp**: Clamped to the absolute parameter limits defined in `state/params.ts`.

### 2.2 Enum Constraint (`enum`)

- **Semantics**: The parameter's resolved value $V$ is chosen uniformly at random from a discrete list of choices $C = [c_1, c_2, \dots, c_k]$:
  $$\text{index} = \lfloor R \times k \rfloor$$
  $$V = C[\text{index}]$$
  where $R \sim \text{Uniform}(0, 1)$.

### 2.3 Relative Constraint (`relative`)

- **Semantics**: The parameter deviates by up to $\pm P\%$ from its baseline value $B$.
  $$V = B \times (1.0 + R_{offset})$$
  where $R_{offset} \sim \text{Uniform}(-P/100, P/100)$.
- **Clamp**: Clamped to the absolute parameter limits defined in `state/params.ts`.

### 2.4 Correlated Constraint (`correlated`)

- **Semantics**: The parameter $Y$ varies linearly in response to the resolved variation of another parameter $X$:
  $$Y_{resolved} = Y_{base} + \text{coefficient} \times (X_{resolved} - X_{base})$$
  where $X_{resolved}$ is the resolved value of the target parameter, and $X_{base}$ is its baseline.
- **Sorting Requirement**: To ensure correct resolution, independent variation points are resolved first, followed by correlated variation points. If a circular dependency is detected, the correlation falls back to baseline values.

## 3. Regeneration Rules & Composition Hierarchies

### 3.1 Regeneration Rules

- **`per-play`**: Resolved at the start of a play session using a play-specific seed.
- **`per-segment`**: Re-rolled on each segment boundary (remains stable within the segment).
- **`per-render`**: Stable within a render but re-rolled across renders.
  _Note_: In live playback, a "render" corresponds to a playback session. In offline stem export, the entire file is one single "render", meaning `per-render` is completely locked to the export seed.

### 3.2 Composition of Seeds (Heuristic-Drift Hierarchy)

To guarantee absolute reproducibility, all random seeds compose deterministically down from the top-level master seed:

```
                            [Master Seed]
                     (piece.variationSeed or random)
                                  │
         ┌────────────────────────┼────────────────────────┐
         ▼                        ▼                        ▼
  [Piece-Level Seed]      [Segment-Level Seed]      [Meta-Arc Seed]
   hash(Master, 'piece')   hash(Master, 'seg', i)   hash(Master, 'meta', i)
         │                        │                        │
         ▼                        ▼                        ▼
  [Param Roll Seed]       [Param Roll Seed]        [Generator Roll Seed]
```

1. **`activeVariationSeed`**: If `piece.variationSeed` is configured, it is utilized. Otherwise, a random seed is rolled at playback start. Let's call this the `activeVariationSeed`.
2. **Piece-level variations**: Resolved once at playback start. For parameter `p`, the seed is `hash(activeVariationSeed, 'piece', p)`.
3. **Segment-level overrides / `per-segment` variations**: Re-resolved at the start of segment `i`. The seed is `hash(activeVariationSeed, 'segment-' + i, p)`.
4. **Meta-Arcs**: If the meta-arc's `seed` in segment `i` is null/dynamic, it is derived as `hash(activeVariationSeed, 'meta-arc-' + i)`.
5. **Heuristic Drift**: Dynamic parameter drift is seeded with `hash(activeVariationSeed, 'drift')`.

This single-hierarchy composition ensures that locking the top-level `variationSeed` locks every single aspect of procedural generation, segment transitions, and variations down to the last bit.

## 4. The Pure Variation Resolver

The variation resolver is a pure function with no side effects or global state references. It resolves a set of baseline parameters using variation points and a seed.

```typescript
export function resolveVariations(
  baseParams: Record<string, number>,
  variations: VariationPoint[],
  seed: number,
): Record<string, number> {
  const resolved = { ...baseParams };
  const rngCache = new Map<string, () => number>();

  // 1. Separate independent and correlated variation points
  const independent = variations.filter(
    (v) => v.constraint.type !== 'correlated',
  );
  const correlated = variations.filter(
    (v) => v.constraint.type === 'correlated',
  );

  // Helper to get or create seeded RNG per param to keep rolls isolated
  const getRng = (paramKey: string, salt: string) => {
    const key = `${paramKey}-${salt}`;
    if (!rngCache.has(key)) {
      rngCache.set(key, createSeededRng(hashStringToInt(salt) + seed));
    }
    return rngCache.get(key)!;
  };

  // 2. Resolve independent variation points
  for (const vp of independent) {
    const rng = getRng(vp.paramKey, vp.id);
    const r = rng();
    const baseVal = baseParams[vp.paramKey] ?? 0;

    let newVal = baseVal;
    if (vp.constraint.type === 'range') {
      const min = vp.constraint.min ?? 0;
      const max = vp.constraint.max ?? 1;
      newVal = min + r * (max - min);
    } else if (vp.constraint.type === 'enum') {
      const choices = vp.constraint.choices ?? [baseVal];
      const idx = Math.floor(r * choices.length);
      newVal = choices[idx] ?? baseVal;
    } else if (vp.constraint.type === 'relative') {
      const pct = vp.constraint.percent ?? 10;
      const offset = (r * 2 - 1) * (pct / 100);
      newVal = baseVal * (1.0 + offset);
    }

    resolved[vp.paramKey] = clampParam(vp.paramKey as any, newVal);
  }

  // 3. Resolve correlated variation points (multi-pass or simple lookup)
  for (const vp of correlated) {
    const target = vp.constraint.targetParam;
    if (!target) continue;

    const coefficient = vp.constraint.coefficient ?? 1.0;
    const targetBase = baseParams[target] ?? 0;
    const targetResolved = resolved[target] ?? targetBase;

    const delta = targetResolved - targetBase;
    const baseVal = baseParams[vp.paramKey] ?? 0;
    const newVal = baseVal + coefficient * delta;

    resolved[vp.paramKey] = clampParam(vp.paramKey as any, newVal);
  }

  return resolved;
}
```

## 5. UI Implementation Plan

### 5.1 Hover / Right-Click Affordance

On any slider in the parameter list (both `ControlPanel` and `SegmentProperties` overrides), a small, premium amber "Vary" dot button will appear upon hover. Right-clicking or clicking this dot will trigger the **Variation Dialog**.

### 5.2 The Variation Dialog

A sleek modal with glassmorphic styling (dark background, amber borders, subtle blur) that lets the user:

- Pick constraint type (`Range`, `Enum`, `Relative`, `Correlated`).
- Configure boundaries (numerical ranges, percent sliders, discrete values, or target correlation param).
- Pick regeneration rule (`per-play`, `per-segment`, `per-render`).
- Add / Update the variation point in the active piece or segment config.

### 5.3 Slider Visual Indicator

When a parameter is varied:

- A prominent yet tasteful **amber dot** is rendered on the slider thumb.
- A **subtle glowing band** is drawn across the track background representing the active constraint range/boundaries.

### 5.4 "Re-Roll" and Variation Editor Panel

- **Re-Roll Button**: Rendered at the top of the timeline transport controls (a Dice icon 🎲 in amber). When clicked during active playback, it rolls a new master seed, forces parameter recalculation, and triggers dynamic audio re-evaluation.
- **Variation Editor Panel**: An expandable drawer/collapsible card in the editor displaying a tabular list of all active variation points in the piece. User can quickly delete points, tweak limits, or adjust regeneration rules.

## 6. Collaborative Synchronization Semantics

To prevent collaborative audio drift when pieces use `null` (dynamic) seeds:

1. **Host Broadcaster**: The room leader/host generates the `activeVariationSeed` at the start of playback and broadcasts it to all session followers via the shared CRDT Yjs map (`sessionConfigMap` key: `variation_seed_active`).
2. **Follower Hydrator**: Guest clients monitor the Yjs event bus and immediately apply the shared seed upon arrival.
3. **Safety Fallback**: If a leader disconnects or is delayed by more than $200\text{ms}$, guest clients fall back to a locally generated seed to avoid clicking/popping or timeline stoppage.

---

## 7. Determinism Guarantees

Stem exports and recordings must stay bit-identical across runs.

- **Export Mode**: All `per-play` and `per-render` variation points are completely locked to the master `exportSeed`.
- **Determinism**: Since the PRNG and resolver are purely deterministic functions, executing multiple stem exports with the same seed will produce exactly identical audio results, honoring the v1.5 DAW export specification.

---

## 8. Guardrails: Max-Variability Heuristic

Too much variation can dilute a piece's core identity (e.g. setting every parameter to vary 100%).

- **Heuristic Guardrail**: We compute a "Piece Coherence Index" based on the accumulated percentage delta allowed by all variation points:
  $$\text{Coherence Index} = 1.0 - \sum \text{variance}_i$$
- **Alert UX**: If the Coherence Index drops below `0.6`, a subtle amber warning badge appears: `"High variability – piece identity may drift"`, helping composers keep their compositions recognizable.
