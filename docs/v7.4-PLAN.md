# AnnealMusic v7.4 Plan — Biofeedback Ingest & Sonification

We are introducing a real-time, stimulus-locked biosignal ingest framework (HRV, breath rate, EEG, GSR) over **WebSerial**, **Web Bluetooth**, and **WebHID** APIs. This framework feeds both clinical research protocols and live audio sonifications.

---

## 1. Final Adapter List & Device Specifications

### Polar H10 / Verity Sense / OH1

- **Transport**: Web Bluetooth API.
- **Service UUID**: Standard Heart Rate Service (`0x180D`).
- **Measurement Characteristic**: Heart Rate Measurement (`0x2A37`).
- **Protocol Details**:
  - Subscribes to notifications on characteristic `0x2A37`.
  - Parses binary GATT bytes:
    - Byte 0: Flags (checks if R-R intervals are present - bit 4).
    - Byte 1: Heart Rate Value (BPM) (either 8-bit or 16-bit depending on bit 0 of flags).
    - Bytes 2+: R-R Intervals (uint16, in units of 1/1024 seconds). We convert these to milliseconds (`value * 1000 / 1024`).
- **Capabilities**: `['hrv', 'heart_rate']`

### OpenBCI Cyton + Ganglion

- **OpenBCI Cyton**:
  - **Transport**: WebSerial API.
  - **Connection Parameters**: Baudrate 115200, 8 data bits, no parity, 1 stop bit.
  - **Protocol Details**:
    - Listens for raw serial stream from FTDI chip.
    - Parses OpenBCI Standard Binary Protocol:
      - Byte 0: Header (`0xA0`).
      - Byte 1: Sample Counter.
      - Bytes 2-25: 8 ADS1299 EEG channels (24-bit signed big-endian integers). Convert to microvolts using scale factor `V_ref / (2^23 - 1) / Gain`.
      - Bytes 26-31: Accelerometer values or auxiliary data.
      - Byte 32: Footer (`0xC0` or standard sequence).
  - **Capabilities**: `['eeg', 'accelerometer']`
- **OpenBCI Ganglion**:
  - **Transport**: Web Bluetooth API.
  - **Capabilities**: `['eeg', 'accelerometer']`

### Muse 2 / S

- **Transport**: Web Bluetooth API.
- **Service UUID**: Custom Muse Service (`fe8d`).
- **Characteristics**: EEG Channels TP9 (`27330003-...`), AF7 (`27330004-...`), AF8 (`27330005-...`), TP10 (`27330006-...`).
- **Protocol Details**:
  - Enables notifications on the corresponding characteristics.
  - Parses compressed 12-bit delta-encoded EEG samples, restoring raw microvolts at 256Hz.
- **Capabilities**: `['eeg']`

### Empatica E4 / EmbracePlus

- **Transport**: Offline export file import path.
- **Capabilities**: `['hrv', 'gsr', 'accelerometer']`

## 2. Frame Format & Channel Naming Convention

All adapters emit structured biosignal frames at their native capture frequency.

```ts
type BiosignalFrame = {
  timestamp: number; // ms since session start
  device_clock?: number; // device's own internal clock in ms (for sync verification)
  channels: {
    [channel_name: string]: {
      value: number; // parsed numerical reading
      unit: string; // 'bpm' | 'rr_ms' | 'breaths_per_min' | 'volts' | 'siemens' | 'microvolts'
      confidence?: number; // 0-1, if device provides signal quality
    };
  };
};
```

### Naming Conventions:

- **Heart Rate / HRV**:
  - `heart_rate` (unit: `bpm`)
  - `hrv` (unit: `rr_ms`)
- **Breath**:
  - `breath_rate` (unit: `breaths_per_min`)
- **EEG**:
  - `eeg_tp9`, `eeg_af7`, `eeg_af8`, `eeg_tp10` (unit: `microvolts`)
- **GSR**:
  - `gsr` (unit: `siemens` or `microsiemens`)

## 3. Privacy & Per-Channel Consent Flow

Biosignals are highly sensitive. v7.4 enforces:

1. **Double-Opt-In:** Subject must accept general clinical trial terms AND grant specific per-channel permissions.
2. **Opt-In Granular Checks:** A subject can consent to HRV recording but opt out of EEG recording.
3. **Data Flow Disclosures:** The consent step clearly highlights if the biometric streams are stored locally, uploaded to a designated PI server endpoint, or both.
4. **Shredding (Withdrawal Compliance):** When a user triggers withdrawal and requests data destruction (`partial_data_disposition: 'discarded'`), the backend API will:
   - Physically delete the uploaded Parquet files from disk/object-store.
   - Delete corresponding rows from `biosignal_streams`.
   - Clear any local browser session database caches.

## 4. Calibration Procedures Per Signal Type

Before starting a study, biosignal channels are calibrated during the `SubjectRunner` prep stage.

### HRV Calibration

- **Duration**: 60 seconds resting.
- **Process**: Subject sits quietly while R-R intervals are captured.
- **Outputs**:
  - `baseline_mean_rr`: Mean R-R interval (ms).
  - `baseline_sdnn`: Standard deviation of normal-to-normal intervals (representing parasympathetic tone).
- **Audio Influence**: SDNN is used to scale dynamic range limits of sonification maps.

### EEG Calibration

- **Duration**: 10 seconds.
- **Process**: Subject blinks when prompted, then closes eyes.
- **Outputs**:
  - `impedance_ok`: boolean verifying electrode contact quality.
  - `blink_threshold`: Max voltage during eyeblinks (used to filter artifact spikes during the study).

### GSR Calibration

- **Duration**: 30 seconds.
- **Process**: Establishes baseline tonic skin conductance level (SCL).
- **Outputs**:
  - `tonic_scl`: average micro-Siemens reading.
  - `active_range`: variance bounds used to scale filters.

## 5. Sonification Integration Design

Live biosignals can drive real-time synthesis:

1. **Source Type `'live-biosignal'`**: We introduce a new source kind in the `MappingSpec` schema.
2. **Player Hydration**: When the player initializes a `'live-biosignal'` source, it establishes connection to the matching connected adapter.
3. **Run-Loop Binding**:
   - The adapter's `stream(connection)` feeds the player.
   - To keep Web Audio thread free of lock-up, the player stores the latest channel values inside a throttled map.
   - On every audio request frame (requestAnimationFrame), the mapping rule transforms the current channel reading (e.g. mapping `hrv` interval to oscillator `sine.detune`) and pushes it to the synthesis engine.

## 6. Clinical Protocol Integration

1. **Declared Channels**: `clinical_protocols` table contains `biosignal_channels` JSONB array:
   ```json
   [
     { "channel_name": "hrv", "required": true, "baseline_seconds": 60 },
     { "channel_name": "eeg_tp9", "required": false, "baseline_seconds": 10 }
   ]
   ```
2. **Telemetries Capture**:
   - Client records streams under a `StreamCapture` buffer class.
   - When the session finishes, the buffer compiles into a compressed payload.
   - Client uploads this payload to `/api/v1/clinical-session-records/:id/biosignal-stream`.
3. **Database Links**: The backend inserts records into the `biosignal_streams` table pointing to the Parquet binary storage locations on disk or object-store.

## 7. Browser Support Matrix & Mobile Fallback Plan

| Transport         | Chromium (Chrome, Edge, Opera) | Firefox | Desktop Safari | iOS Safari (Mobile) | Android WebView |
| :---------------- | :----------------------------- | :------ | :------------- | :------------------ | :-------------- |
| **Web Bluetooth** | Yes                            | No      | No             | No                  | No              |
| **WebSerial**     | Yes                            | No      | No             | No                  | No              |
| **WebHID**        | Yes                            | No      | No             | No                  | No              |

### Mobile Fallback Path:

For native mobile applications (iOS/Android) built using **Capacitor**, we implement a native polyfill wrapper. The web client checks `Capacitor.isNativePlatform()`.

- If native: it requests connection via the Capacitor plugins (`BiofeedbackBridge`), which communicate with the native BLE manager.
- If browser: it uses standard Web Bluetooth/WebSerial API calls.

## 9. Major Risks & Mitigation Strategies

1. **Chrome vs Safari BLE discrepancies**:
   - _Risk_: A researcher tries to run the clinical trial on an iPad using Safari, which does not support Web Bluetooth.
   - _Mitigation_: The Subject Runner checks compatibility on start. If Web Bluetooth/WebSerial are missing, it shows a highly visible guidance alert asking the operator to use a Chromium browser or download our native iOS app.
2. **Serial Buffer Latency & Drift**:
   - _Risk_: At high sample rates (250Hz OpenBCI), serial parsing in Javascript can lag, creating a backlog buffer and increasing latency up to several seconds.
   - _Mitigation_: The `OpenBCICytonAdapter` runs inside a lightweight Web Worker or Web Assembly module if available, or utilizes highly optimized fast buffer slices, dropping older frame renders if processing lags behind real-time.
3. **Subject Consent Revocations (GDPR Shredding)**:
   - _Risk_: A subject withdraws and requests data deletion, but database cascades fail to clear the actual raw binary file stored in regional cloud storage.
   - _Mitigation_: The DELETE endpoint implements strict two-phase deletion: it first deletes the file from physical storage, and only deletes the database record after confirming successful file system removal.
